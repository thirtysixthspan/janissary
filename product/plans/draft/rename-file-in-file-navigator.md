# Rename a file or directory in the file navigator

**Complexity: 4/10** — a small feature with a precedented shape (mirror the `deleteFileTreeItem` RPC chain end-to-end), but correctness spans web + server across five touched files plus a new protocol method, and the client side must stay under the 200-line limit by extracting a hook rather than growing `FileTreeTab.tsx`.

Give the file tree tab an in-place rename: with a row selected, pressing the rename chord turns that row's name into an editable text field. Editing the name and pressing Enter renames the file or directory on disk, but only if the name actually changed; the tree then refreshes to show the new name. This mirrors how a conventional editor's Explorer renames an entry — F2/inline edit — rather than a separate dialog. The tree has no existing inline-edit affordance to copy (its "new file" flow opens a separate editor tab and renames via that tab's label — it does not edit within the tree), so the editable field itself is genuinely new UI; everything behind it is reuse: the tree's existing on-disk rename primitive (the same `renameSync` the drag-move flow already uses at `src/file-tree/manager.ts:125`), the existing Overwrite/Cancel conflict dialog (`web/src/MoveConflictDialog/MoveConflictDialog.tsx`), and the existing rebuild-and-watch refresh path (`manager.rebuild`). It adds one new RPC (`renameFileTreeItem`) modeled exactly on `deleteFileTreeItem`, not any new subsystem.

## Design decisions

1. **Scope: files and directories.** Any selected row except `..` can be renamed, matching how the tree's delete and move operations already treat both file and directory rows. The on-disk operation (`renameSync`) is identical for either, so there is no reason to restrict it to files.
2. **Chord: Cmd+R on macOS, Ctrl+R elsewhere.** The tree captures this chord for itself (preventing it from reaching window-level bindings), consistent with the tree's existing per-platform chords — undo `Cmd+Z`/`Ctrl+Z` and new-file `Cmd+N`/`Ctrl+N`, which the tree already captures via the `e.ctrlKey || e.metaKey` test. It fires only when a renameable row (not `..`, and a row exists) is selected; otherwise it is a no-op.
3. **Inline editable field, not a dialog.** Pressing the chord replaces the selected row's static name span with a text input pre-filled with the current name. The rest of the tree stays visible and in place. This is the behavior the backlog item calls for ("turns filename into an editable field").
4. **Commit only on a real change.** Enter commits the rename, but only if the edited name differs from the original — an unchanged name (Enter with no edit) is a silent no-op that just closes the field, per the backlog item ("renames the file if it has changed"). An empty or whitespace-only name is treated as a cancel: the field closes and nothing is renamed.
5. **Escape and blur both cancel.** Pressing Escape, or the field losing focus (clicking elsewhere), abandons the edit and restores the original name with no on-disk change. This avoids accidental commits from a stray click.
6. **Name collision reuses the Overwrite/Cancel dialog.** If the committed name matches a sibling already in the same directory, the existing `MoveConflictDialog` (the one the drag-move flow shows) appears. Overwrite replaces the existing entry and completes the rename; Cancel returns the user to the still-open edit field so they can choose a different name.
7. **Rename is not added to the move undo/redo stack in this version.** A rename does not push onto the tab's move undo stack (`state.undoStack`/`state.redoStack` in `manager.move` at `src/file-tree/manager.ts:126-127`) and is not reversible with `Cmd+Z`. Keeping the move stack move-only avoids conflating two operations in one timeline; revisiting this is listed under Out of scope.
8. **Overwrite uses `renameSync` and inherits its limits.** Overwriting a plain file works; `renameSync` will *not* replace a non-empty directory or replace across a file-vs-directory type mismatch — it throws `ENOTEMPTY`/`EISDIR`, which the manager's try/catch swallows into a silent no-op, exactly as the drag-move flow already behaves (`manager.move` at `src/file-tree/manager.ts:125` catches and returns). This is an accepted limitation, not something to special-case.

Implement in the order below (protocol type first, then the server chain, then the client) so `npm run typecheck:diff` stays green at each checkpoint — the client's `client.send({ method: 'renameFileTreeItem', ... })` will not type-check until the protocol method exists.

**Protocol — `src/protocol.ts`.** Add, next to `deleteFileTreeItem` at line 190, a new `ClientMessage` union member `| { method: 'renameFileTreeItem'; params: { index: number; relPath: string; newName: string } }` with a short doc comment matching the style of the surrounding members. `relPath` is the row's tree-relative path; `newName` is the bare new filename (no path separators).

**Server — `src/message-handler.ts`.** Add `case 'renameFileTreeItem':` to the shared `case` group at lines 66-74 that dispatches to `handleFileTreeMessage`. This is the routing site most likely to be missed — without it the message falls through to the default handler.

**Server — `src/message-handler-file-tree.ts`.** Add `'renameFileTreeItem'` to the `FileTreeMessage` `Extract<…>` union (lines 5-8) and a `case 'renameFileTreeItem':` that calls `controller.renameFileTreeItem(message.params.index, message.params.relPath, message.params.newName)` then breaks — reaching the shared `reply({ …, result: 'ok' })` at line 48, exactly like the `deleteFileTreeItem` case at line 22.

**Server — `src/controller.ts`.** Add a `renameFileTreeItem(index, relPath, newName)` pass-through next to `deleteFileTreeItem` (lines 203-205) that calls `fileTreeRpc.renameFileTreeItem(this.managers, index, relPath, newName)`.

**Server — `src/controller/file-tree.ts`.** Add `renameFileTreeItem(managers, index, relPath, newName)` resolving the tab index to its label (`managers.tab.tabs[index]?.label`) and delegating to `managers.fileTree.rename`, mirroring `deleteFileTreeItem` at lines 26-29.

**Server — `src/file-tree/manager.ts`.** Add a `rename(label, relPath, newName)` method next to `move`/`delete` (after line 162). It resolves the tab state (`this.tabs.get(label)`; return if absent), rejects a `newName` containing a path separator (`/` or the platform separator) as an invalid no-op so the rename can never move the item to another directory, computes `oldAbs = path.join(state.root, relPath)` and `newAbs = path.join(path.dirname(oldAbs), newName)`, performs `renameSync(oldAbs, newAbs)` inside a try/catch that silently returns on failure (matching `move` at line 125), then calls `this.rebuild(label)`. It does not touch `state.undoStack`/`state.redoStack` (Decision 7). Keep the method small; `manager.ts` is at 222 lines with heavy block comments, comfortably under the 200 non-comment-line limit, so this addition does not require an extraction.

**Client — new hook `web/src/useFileTreeRename.ts`.** Because `web/src/FileTreeTab.tsx` is already 209 lines, rename orchestration must live in a hook of the same shape as `useFileTreeSearch`/`useFileTreeDrag`, not inline in the component. The hook owns: the `editing` path state (or `null`); an `beginRename(path)` starter (called from the chord branch when a renameable row is selected); a `commit(rawName)` that uses the pure helpers below to decide no-op vs collision vs clean rename — sending `client.send({ method: 'renameFileTreeItem', params: { index, relPath, newName } })` on a clean rename, then closing the field and setting selection to the new path via the `setSelected` callback passed in; a `cancel()`; and the pending-collision state that drives the `MoveConflictDialog`, whose Overwrite handler re-sends the same RPC and whose Cancel handler reopens the edit field on the same row. `FileTreeTab.tsx` consumes it exactly like the other two hooks (one added line at lines 46-47), adds the `'r'` chord branch in `onKeyDown` before line 118, renders a text input in place of the static name span (`FileTreeTab.tsx:170`) when `rename.editing === row.path`, and renders `MoveConflictDialog` from the hook's pending state alongside the existing drag one.

**Client — new module `web/src/file-tree-rename.ts`.** A pure, DOM-free, unit-testable helper (mirroring `file-tree-keys.ts` and `file-tree-new-file.ts`). Export a function that, given the original relative path and the raw edited name, returns a tagged result: `noop` (trimmed name equals the original basename, or is empty/whitespace), or `rename` carrying the computed new relative path (the original path's parent joined with the trimmed name). Export a second helper that, given the new name and the set of sibling row names in the same directory, reports whether a same-directory collision exists. Export the default text-selection range for the field (basename without extension for a file, whole name for a directory) so the input can pre-select the editable portion without putting that convention in JSX.

**Spec + docs.** Update `product/specs/file-tree-tab.md`: add a `Cmd+R` / `Ctrl+R` row to the "Keyboard interactions" table (around lines 230-243, next to the `Cmd+N` row) and a short "Renaming a file or directory" prose section describing the inline field, commit-only-on-change, Escape/blur cancel, and the Overwrite/Cancel collision dialog. Add the chord to the list of Ctrl/Cmd chords the tree captures for itself rather than passing to the window (the paragraph at lines 244-248, which today names only undo/redo and `Cmd+N`). Mirror the keyboard-table addition in `documentation/user-documentation/` if that section documents the tree's keys (check before editing).

## Out of scope

- **Undo/redo of a rename.** A rename does not join the move undo/redo stack in this version and cannot be reversed with `Cmd+Z` (Decision 7).
- **Renaming into a different directory.** Typing path separators to move the item elsewhere is not supported; the field renames in place only. Cross-directory moves remain the job of drag-and-drop.
- **Renaming the `..` row or the tree root.** `..` is never renameable; the tree root is not a row and is not covered here.
- **Multi-select / bulk rename.** The tree has no multi-select; rename acts on the single selected row.
- **A `rename` command-line verb.** This is a direct-interaction affordance only; no new command is added to the command line.
