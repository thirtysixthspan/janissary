# File navigator: color git-modified files yellow

**Complexity: 3/10** — one new small server module plus wiring into two existing call sites (`open`/`rebuild` in `file-tree-manager.ts`) and a client render tweak; no new protocol message, no persistence, no concurrency — but correctness depends on catching both row-build call sites and on a file at the client's 200-line ceiling.

## Summary

In the file tree tab (`files [path]`), any row that git considers changed — a working-tree modification, a staged change, or an untracked (new) file — renders its name in yellow, the same way a conventional editor's Explorer/SCM decoration highlights dirty files. A directory row also renders yellow whenever any file beneath it (at any depth, expanded or not) has a git change, so the signal is visible without expanding every folder. Rows in a directory that is not part of a git repository (or where git status cannot be determined) render with no special coloring and no error is shown.

## Design decisions

1. **Which git states count.** Modified (tracked, unstaged), staged, and untracked files are all treated identically — all render in the same yellow, with no visual distinction between the three states. This matches the literal ask ("indicate modified files as indicated by git") without introducing a broader status-coloring language than requested.
2. **Directory propagation.** A directory row is colored yellow if any descendant file anywhere under it has a git change, regardless of whether that descendant's parent directories are currently expanded. This lets a user spot changes in collapsed subtrees without walking the whole tree, mirroring how IDEs decorate folders.

   Note that `buildRows` (`src/file-tree.ts:35`) only descends into directories that are already in the `expanded` set — an unexpanded directory's row is present, but its contents are never read from disk, so this cannot be implemented by "walking into" collapsed directories. It works anyway because `git status` (run once per rebuild against the whole tree root, not per visible directory) already returns every changed path in the tree, regardless of expansion state — propagation is a prefix check (`changedPath.startsWith(rowPath + '/')`, or `changedPath === rowPath` for a file) against that one flat set, applied to whichever rows `buildRows` currently returns. No extra filesystem reads are needed to color a collapsed directory.
3. **Refresh trigger.** Git status is not polled on its own timer. It is recomputed every time the tree's rows are already being recomputed — on initial `open()`, and on every trigger of the existing `rebuild()` (the watcher's debounced refresh, `toggle`, `collapseAll`, `move`, `delete`, `undo`/`redo`, and `reroot`) — so there is exactly one signal that decides "the tree's displayed state is stale" and git status rides along with it rather than needing its own staleness tracking.
4. **Non-git roots.** If the tree's root is not inside a git repository, or the git status command fails for any reason, the tree renders exactly as it does today — no yellow anywhere, no error, no header notice. This keeps the feature purely additive for repos and inert everywhere else.

5. **Git invocation.** Use `node:child_process`'s `execFileSync` with an argument array (`execFileSync('git', ['status', '--porcelain=v1', '--untracked-files=all', '-z'], { cwd: root })`), not a shell string — unlike the `execSync('git clone "${...}"', ...)` string-interpolation pattern already in `src/workspace.ts:74`, an argument array needs no shell-quoting and is safe against paths containing spaces or shell metacharacters. `--untracked-files=all` is required: without it, git collapses a wholly-untracked directory into a single `?? dir/` entry instead of listing its files individually, which would silently break decision 2's per-file propagation for new directories. `-z` NUL-terminates each entry as `XY path\0` and is required for filenames containing spaces or unusual characters to parse unambiguously; verified against a real repo, a rename/copy entry (status `R`/`C`) is two consecutive NUL-terminated fields — the new path first, then the old path — both of which count as one changed path each for this feature's purposes (no distinction between them is needed, since decision 1 already treats every git state identically).

## Proposed changes

1. **Server: git status lookup.** Add a small module `src/git-status.ts` exposing a function (e.g. `changedPaths(root: string): Set<string>`) that, given an absolute directory root, runs `git status --porcelain=v1 --untracked-files=all -z` scoped to that root (per decision 5) and returns the set of root-relative paths that are modified, staged, or untracked — or an empty set if the root isn't inside a git repository or the command throws. Wrap the `execFileSync` call in a `try/catch` that returns an empty set on any failure (non-git directory, git not installed, non-zero exit) — never surfaced to the caller as an error, per decision 4.
2. **Server: attaching the flag to rows.** Add a second small function in the same module (or in `src/file-tree.ts`, next to `buildRows`) — e.g. `markChanged(rows: FileTreeRow[], changed: Set<string>): FileTreeRow[]` — that takes the flat row list `buildRows` already produced and the changed-path set from step 1, and returns rows with a new field set: a file row is marked when its `path` is in `changed`; a directory row is marked when any path in `changed` starts with `${row.path}/` (per decision 2's prefix-check, not a recursive walk). Call this from **both** of `file-tree-manager.ts`'s two `buildRows` call sites — `open()` at `:82` and the private `rebuild()` at `:257` — so a freshly opened tree is colored immediately, not only after the first watcher-triggered rebuild. This produces a `changed?: boolean` field on `FileTreeRow` meaning "this row itself, or something beneath it, has a git change." Every call already re-reads the tree from disk (both `open()` and every trigger of `rebuild()` — the watcher's debounce, `toggle`, `collapseAll`, `move`, `delete`, `undo`/`redo`, and `reroot`), so recomputing git status on each call adds one subprocess invocation to an already-disk-bound operation rather than a new hot path.
3. **Shared type.** Extend `FileTreeRow` in `src/types.ts` (`:105`) with the new optional boolean field, documented the same way its existing fields are (a one-line comment on what "changed" means for a directory vs. a file row). No change needed in `src/protocol.ts` — it re-exports the type as-is (see the "what already exists" table).
4. **Client: rendering.** `web/src/FileTreeTab.tsx` is already 200 raw lines (`wc -l`), at the project's `max-lines` ceiling — adding the modifier-class logic inline risks tripping the lint rule. Extract the row's className computation (currently the inline template literal at `:164`, `` `files-row${row.path === selected ? ' selected' : ''}${drag.dropTarget?.path === row.path ? ' drop-target' : ''}` ``) into a small new helper module (e.g. `web/src/file-tree-row-class.ts`, exporting a function computing both the row wrapper's class and the `files-name` span's class from `row`, `selected`, and `drag.dropTarget`), and have `FileTreeTab.tsx` import it instead of inlining the logic — this keeps the component file's line count from growing while adding the `changed` modifier (e.g. `files-name--changed`) to the `files-name` span. Add the corresponding color rule to `web/src/theme.css`: reuse the existing `--running` custom property (already defined as a yellow/amber tone in every one of the file's theme blocks — `:root`/`dark` at `:15`, `light` at `:34`, `solarized-dark` at `:52`, `solarized-light` at `:70`, `nord` at `:88`, `dracula` at `:106` — currently used only for `.line.output.running` at `:273`) rather than inventing a new token that would need to be added to all six theme blocks.
5. **Spec update.** `product/specs/file-tree-tab.md`'s "Tree contents" section gains a short paragraph describing the git-modified coloring behavior (per the design decisions above), so the spec stays the source of truth for tree rendering rules.

## Tests

- `src/git-status.test.ts` (new, using the `mkdtempSync`/`execSync('git init', ...)` fixture pattern from `src/workspace.test.ts:13-19`): `changedPaths` returns the expected path set for a modified tracked file, a staged file, an untracked file, an untracked file inside an untracked directory (confirming `--untracked-files=all` surfaces it individually, not collapsed to the directory), and a renamed file; returns an empty set for a directory that is not a git repository; returns an empty set (not a thrown error) when the `git` binary invocation fails or is missing.
- `src/file-tree.test.ts` (or alongside `markChanged`'s actual location): marks a changed file's row and every ancestor directory row up to the root when a descendant file has a git change — including an ancestor directory that is currently collapsed (not expanded, so `buildRows` never read its contents) — confirming propagation works via the prefix check against the flat changed-path set, not by walking into the directory; unrelated rows are left unmarked.
- `src/file-tree-manager.test.ts`: opening a fresh tree (`open()`) on a root with a changed file shows the `changed` flag immediately, without waiting for a watcher-triggered `rebuild()`.
- `web/src/file-tree-row-class.test.ts` (new, for the extracted helper from Proposed changes item 4): a row with `changed: true` produces a class including `files-name--changed` (or equivalent); a row without it does not; existing `selected`/`drop-target` class behavior is preserved.
- `web/src/FileTreeTab.test.tsx`: a row with `changed: true` renders with the `files-name--changed` class end-to-end through the component.

## Out of scope

- Distinguishing modified/staged/untracked with different colors or icons — all three render identically per decision 1.
- Any polling mechanism independent of the existing watcher-driven refresh.
- Showing an error or notice when a root isn't a git repository.
- Any change to the delete/undo/redo/drag-and-drop behaviors already documented in `file-tree-tab.md`.
