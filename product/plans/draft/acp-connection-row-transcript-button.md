# Transcript button on each ACP connection row

**Complexity: 4/10** — a new client→server RPC plus a new `acpRef` discriminated field on `ConnectionView`, spanning server and web, with editor-persona sessions gaining lightweight exchange recording and three-way routing on click; every edit is small and precedented, and the recording reuse collapses to factoring out one existing helper rather than a new shared module.

## Summary

Per the backlog: "add a button on each acp connection row that shows the transcript of the acp, similar to how the button in monitors shows the transcript of the monitoring acp." The floating "connections" panel (`web/src/StatusPanels.tsx`) lists each tab's open connections; some of those rows are ACP sessions (`kind: 'acp'`) coming from three sources — the tab's own agent (`acp:opencode`), monitor sessions (`monitor:security …`), and editor-persona sessions (`reviewer (acp)`). This feature adds a clipboard-icon button to every ACP connection row that, when clicked, opens that session's transcript as a read-only editor tab — the same "write a capture file, open it in an editor tab" affordance the monitor reporting tab's context-snapshot button already uses (`src/monitor/context.ts`) and the agent tab's metadata-row "Open transcript" button already uses (`src/controller/transcript.ts`).

Today only monitor sessions record their exchange (`MonitorSub.contextText`); the bare editor-persona `AcpSession` records nothing. So covering "all acp rows" requires editor-persona sessions to record their exchange (reusing the monitor's existing snapshot format), and adding a single new client→server message that routes to the right session by a small identifier carried on each row.

## Design decisions

- **Every ACP connection row gets the button.** The button appears on the tab's own agent row (`acp:…`), on monitor rows (`monitor:…`), and on editor-persona rows (`… (acp)`) — every `kind: 'acp'` row. (Chosen over "main + monitor only" or "monitor only.")
- **Clicking opens a read-only snapshot in an editor tab.** It writes the session's accumulated exchange to a capture file and opens it as a scrollable editor tab, mirroring the monitor context-snapshot and the existing metadata-row transcript button. It is a point-in-time snapshot, not a live view — clicking again opens a fresh one, matching the monitor snapshot's documented behavior. (Chosen over copy-to-clipboard.)
- **An empty/not-yet-started session opens a tab with a placeholder.** When the session has no recorded exchange yet, the button still opens an editor tab whose body is the exact text `No transcript yet.`, so every click produces visible feedback. (Chosen over a silent no-op or a disabled button.) This differs deliberately from the existing metadata-row button and the monitor reporting-tab snapshot button, which both no-op on empty content and are left unchanged.
- **The button uses the clipboard icon and reads "Open transcript."** It reuses `viewCaptureIcon` (the regular `faClipboard` already exported from `web/src/icons.ts`) — the same glyph as the agent tab's metadata-row transcript button — with `title` and `aria-label` both `Open transcript`. (Chosen over the monitor bars icon or the wording "Open context snapshot.")
- **Each ACP row carries a discriminated `acpRef` identifier, and one new RPC routes on it.** `ConnectionView` gains an optional `acpRef` with a `scope` discriminant: `{ scope: 'tab', label }` for the tab's own agent, `{ scope: 'monitor', name }` for a monitor session, and `{ scope: 'editor', label, persona }` for an editor-persona session. A new `openAcpTranscript` message carries the `acpRef`, and the server routes on `scope`. (Chosen over sending a bare row index or parsing the row's display text.)
- **Editor-persona recording reuses the monitor's snapshot format in place — no new shared module.** Rather than creating a shared context module, factor the monitor's existing snapshot body-builder into an exported `formatContext` in `src/monitor/context.ts` (where `recordContext`, the entry type, and the `SENT TO MODEL` / `MODEL RESPONSE` headers already live), and have `EditorAcpManager` import it — the editor manager already imports from `../monitor/` (`acp-manager.ts:4` `spawnMonitorSession`). Editor-persona recording is a plain append to a per-session entry array; it does **not** reuse `recordContext` and deliberately does not track byte size, because (unlike monitors) editor-persona rows have no byte counter — add byte tracking only if such a counter is ever introduced. The tab's own agent (`scope: 'tab'`) records nothing new — its transcript is the tab conversation log via the existing `transcriptText(tab.log)` path. (Chosen over a new shared module and over per-manager format duplication.)

## Proposed changes

**Factor out `formatContext` in `src/monitor/context.ts`.** Extract the snapshot body-building loop currently inline in `snapshotMonitorContext` (`src/monitor/context.ts:33`) into an exported `formatContext(entries: MonitorContextEntry[]): string` that joins each entry between its `SENT TO MODEL` / `MODEL RESPONSE` begin/end markers, and have `snapshotMonitorContext` call it — identical output, so the reporting-tab snapshot button (including its current empty no-op) is unchanged. `recordContext`, the entry type, and `HEADERS` stay exactly as they are; no new module and no changes to the existing `recordContext` call sites.

**Editor-persona session recording.** `EditorAcpManager` gains a private `contexts: Map<string, MonitorContextEntry[]>` keyed by the same `${label}:${persona}` key its `sessions`/`personas` maps use, initialized alongside them in `session()` (`acp-manager.ts:23-28`) and deleted in both `close()` and `closeTab()` (`:40-57`) so a torn-down session leaves no stale store. It exposes `record(label, persona, text, role)` (a plain append of `{ role, text }` to that key's array) and `transcript(label, persona): string` (returning `formatContext` of that key's entries, or `''` when the key is unknown or empty), importing `formatContext`/`MonitorContextEntry` from `../monitor/context.js` (as it already imports `../monitor/acp.js` at `:4`). The driver in `src/editor-suggest/handler.ts` calls `managers.editorAcp.record(label, persona.name, <prompt>, 'input')` for the prompt it sends at `handler.ts:61`, and `record(label, persona.name, reply, 'response')` inside `onEnd` (`:63`) where the full streamed `reply` is already accumulated — mirroring the monitor's input-then-response recording split. No byte counting is tracked (see the design decision).

**`ConnectionView` gains `acpRef`.** Add an optional `acpRef` field to `ConnectionView` (`src/protocol.ts:10`), a discriminated union on `scope` with variants `{ scope: 'tab', label }`, `{ scope: 'monitor', name }`, and `{ scope: 'editor', label, persona }`. Populate it where each ACP row is built: `ConnectionManager.connectionsFor` sets `scope: 'tab'` (with the owning tab `label`) on the `acp:` row (`connection/manager.ts:16-17`); `monitorConnections` sets `scope: 'monitor'` with the monitor's `name` (`monitor/info.ts:36`); `EditorAcpManager.connectionsFor` sets `scope: 'editor'` with the row's `label` and `persona` (`editor/acp-manager.ts:35`). Non-ACP rows leave `acpRef` undefined; every `kind: 'acp'` row sets it.

**New `openAcpTranscript` RPC.** Add `{ method: 'openAcpTranscript'; params: { acpRef: AcpRef } }` to the `RpcCall` union (`src/protocol.ts:240`, next to `openTranscriptFor`). Add `openAcpTranscript(managers, acpRef)` to the existing `src/controller/transcript.ts` (12 lines, alongside `openTranscriptFor`) rather than a new file. It produces the source text by `scope`: `'tab'` → `transcriptText(managers.tab.tabs.find(...label).log)`; `'monitor'` → a new `MonitorManager.transcript(name): string` returning `formatContext` of that named monitor's `contextText` (so the handler never reaches into the manager's private `monitors` map — mirror how `snapshotContext` at `manager.ts:216` already delegates); `'editor'` → `managers.editorAcp.transcript(label, persona)`. In every case, when the produced text is empty it substitutes the literal placeholder `No transcript yet.`, then writes it with `writeCaptureFile` and opens it via `managers.openFile.edit(command, file, managers.tab.cur().label)` (opening relative to the active tab, as `snapshotMonitorContext` does at `context.ts:35`), with a command label naming the source (e.g. `transcript acp:opencode`, `transcript monitor:security`, `transcript reviewer (acp)`). Wire it in `src/message-handler.ts` by importing the handler directly and calling `openAcpTranscript(controller.managers, message.params.acpRef)` in a new `case`, exactly as the `openTranscriptFor` case at `message-handler.ts:100` does (avoiding a passthrough on the line-limited `controller.ts`).

**Panel button.** `StatusPanels` renders a clipboard-icon button on any connection row whose `acpRef` is present (gating on `acpRef`, not merely `kind === 'acp'`, so a row without a ref never renders a dead button), using `viewCaptureIcon` with `title`/`aria-label` `Open transcript`, calling a new `onOpenAcpTranscript?(ref: AcpRef) => void` prop with `row.acpRef`. On editor-persona rows the transcript button renders alongside the existing `panel-row-close` `×` button. The prop is wired where the panel is interactive and shows ACP rows: `AgentTabBody` (agent tab's own agent + owned monitors; sends `{ method: 'openAcpTranscript', params: { acpRef } }`) and `EditorConnectionsPanel` (editor-persona sessions + owned monitors). The non-interactive harness-tab panel (`HarnessTabLayer:57`) is not wired, so no button renders there — consistent with how `onCloseRow` is editor-only.

**Styling.** Add a rule for the new button in `web/src/theme.css` mirroring `.tab-open-transcript` (`:223-227`) / `.panel-row-close` (`:326-330`) — transparent background, `var(--muted)`, `var(--fg)` on hover — sized to sit inline in a panel row.

**Ordering & constraints.** Land the changes in three checkpoints, each leaving typecheck/tests green: (1) factor out `formatContext` in `src/monitor/context.ts` (monitor snapshot output unchanged, monitor tests still pass); (2) server side — `acpRef` on `ConnectionView`, the three row builders, editor-persona recording + `transcript`/`record` methods, the `MonitorManager.transcript` method, the handler in `src/controller/transcript.ts`, and the `RpcCall`/message-handler wiring (adding the RPC to the union forces the handler case to exist); (3) web side — the `StatusPanels` button, its wiring in `AgentTabBody`/`EditorConnectionsPanel`, and the CSS. No dependency on any other draft plan. Constraints: all new `src/` imports carry `.js` extensions (NodeNext); the empty-text check uses an explicit `length === 0` / `=== ''` test, not `||`/`??`, to stay clear of `prefer-nullish-coalescing`; and no touched file approaches the 200-line `max-lines` cap — `StatusPanels.tsx` (75), `connection/manager.ts` (51), `editor/acp-manager.ts` (58), `monitor/context.ts` (36), and `controller/transcript.ts` (12) all have ample room.

- Note: `AgentTabBody`/`EditorConnectionsPanel` have no colocated unit test today, so the end-to-end "click sends `openAcpTranscript`" wiring is covered by the manual check in Verification rather than a new unit test (don't invent a test file where the area has none).

## Out of scope

- Changing the existing agent-tab metadata-row "Open transcript" button or the monitor reporting-tab context-snapshot button — both keep their current empty-content no-op behavior; the new per-row button is a separate entry point with its own placeholder behavior.
- A transcript button on non-interactive harness-tab connection panels (`HarnessTabLayer`), where the panel intentionally ignores pointer events.
- Copy-to-clipboard export of a transcript, or a live-updating transcript view (the snapshot is point-in-time, as with the monitor snapshot).
- Any change to how ACP sessions themselves are spawned, reused, or torn down.
