# Editor Tab GitHub Syncing

**Complexity: 6/10** — a new git-sync engine (workspace provisioning + commit/rebase/push cycle) with real concurrency reasoning (concurrent provisioning, own-save-vs-other-tab-refresh ordering) and a new server/client field on `EditorView`, but it composes almost entirely out of existing mechanisms (`workspace.ts` provisioning, `EditorWatchManager` live-reload, the harness `provisioning`/`provisionError` placeholder-tab pattern) rather than needing a novel protocol, and fits in one module rather than a new subsystem of files.

A file listed in the project's `.janissary/config.json` can be kept automatically synced with its `origin/master` branch. Rather than a per-tab toggle, syncing is entirely config-driven: any file whose project-relative path appears in a new sync-paths setting is always opened and edited from inside a single shared, lazily-created workspace clone dedicated to syncing, instead of the main project checkout. Saving that file commits the change (with a fixed commit message), rebases against `origin/master` with the remote always winning any conflict, and pushes. Opening a synced tab (or another synced tab saving) also pulls `origin/master` into the shared workspace and refreshes every open, unmodified tab pointed at a changed file. The metadata row shows a read-only status icon reflecting whether that file is still being provisioned, syncing, synced, or has hit a sync error — there is no interactive enable/disable control in the UI.

## Design decisions

- **Sync is config-driven, not a UI toggle.** The backlog's original "github icon to enable or disable" is replaced entirely: a file syncs if and only if its project-relative path is listed in `.janissary/config.json`. There is no button or click target to turn syncing on or off from the editor.
- **Master only.** Syncing only applies to files whose content lives on `origin/master`. There is no per-branch sync support, and the earlier idea of deriving the branch from "the file navigator or agent that launched the editor" is dropped.
- **One shared sync workspace, not one per file or per tab.** All synced files share a single workspace clone, dedicated to syncing and separate from any `agent --workspace` clones. It is lazily created the first time any config-listed file is opened under a fixed name (e.g. `git-sync`, not a tab label — existing workspace clones are always named after their owning tab, see `WorkspaceManager.create` at `src/workspace-manager.ts:28`), and removed only at app shutdown via `WorkspaceManager.removeAll()` (`src/workspace-manager.ts:65`, called from `src/controller.ts:229`) — **not** via the per-tab-close `WorkspaceManager.remove(dir)` path (`src/workspace-manager.ts:59`) that ordinary workspace clones use, since the sync workspace must outlive any single tab that happens to close. If two config-listed files are opened before the first clone finishes, the "ensure workspace exists" step must return the same in-flight provisioning promise to both callers rather than calling `provisionWorkspace` twice against the same target directory (see `provisionWorkspace`, `src/workspace.ts:85`).
- **Synced files are always edited from inside the shared workspace.** A config-listed file's editor tab targets the copy of that file living in the shared sync workspace, not the file in the main project checkout. This retargeting must happen before `src/openers/editor.ts`'s `openInEditor` (lines 24-34) resolves the `EditorView`'s `path`/`url`/`size` — since workspace provisioning is asynchronous but `openInEditor` is not, a synced file's open cannot simply extend `openInEditor` in place. Instead it mirrors `HarnessManager.spawnTab`/`finishSpawn`'s immediate-placeholder-then-async-fill-in pattern (`src/harness/manager.ts:151-212`, and `HarnessView.status: 'provisioning'`/`provisionError` at `src/types.ts:66-69`): the tab opens immediately in a provisioning state, and is filled in with the real workspace-resolved path/size/url once the shared workspace's ready promise resolves.
- **Save behavior.** On save: `saveFile` (`src/editor/save.ts`) writes the buffer to the file's location inside the shared workspace and confirms the save exactly as it does today (unchanged synchronous write, size update, `markSaved` call, "Saved" flash) — this part is not slowed down by git. Only after that write succeeds does an asynchronous git-sync cycle kick off, not awaited by the save itself: commit with the fixed message `chore: planning`, then `git pull --rebase` against `origin/master`, then `git push`. A slow or failing network sync never delays or blocks the save confirmation the user already saw.
- **Remote always wins on conflict.** If the rebase hits a conflict, the remote version is taken; there is no merge-conflict UI or manual resolution step.
- **Pull-on-open, refreshed via the existing live-reload mechanism — no new notify mechanism needed.** Opening a synced editor tab, or another synced tab completing a save, pulls `origin/master` into the shared workspace. Because every synced tab's `EditorWatchManager` watcher (`src/editor/watch-manager.ts`) is already watching that tab's file at its resolved path inside the shared workspace (wired via `openEditorTab`'s `watch(label, path)` call at `src/tab/openers.ts:41`), a pull that changes a given file on disk is picked up automatically by the existing `fs.watch`-based check — no new "notify open tabs" mechanism is required. The existing dirty-vs-clean rule composes for free: a clean tab reloads automatically (`web/src/EditorTab.tsx`'s `mtimeMs` effect, around lines 95-112); a tab with unsaved changes is left alone, matching `editor-tab.md`'s "Live reload of external changes" section. The one thing to get right: only the tab that itself triggered a save calls `markSaved` (as `saveFile` already does at `src/editor/save.ts:45`) so its own write isn't misread as external; any other synced tab whose file changed as a side effect of that same sync cycle's rebase never had its baseline moved, so its watcher correctly treats the change as external, same as today.
- **First-open provisioning shows a loading state.** If the shared workspace doesn't exist yet when a synced file is opened, the editor tab shows a loading/provisioning indicator (the same `provisioning` idea as `HarnessView.status`, applied to a new optional state on `EditorView` — see Proposed changes) and only loads real content once the workspace clone and its initial pull have completed.
- **Sync failures are non-blocking.** A failed push or pull-rebase (network issue, auth failure, missing/expired token) never blocks editing or shows a dialog. It only changes the status icon to its error state. Because git-sync's commands run as direct server-side `execFile` calls (following `git-status.ts`'s pattern) rather than through a sandboxed spawn, they do not automatically inherit `GH_TOKEN` the way a sandboxed agent shell does (that injection happens in `sandboxSpawn`, `src/sandbox/index.ts:220`, which only applies to spawned agent/harness/shell processes, not to the server's own git calls) — git-sync's own pull/push calls must explicitly pass `GH_TOKEN: getGithubToken()` (`src/github-token.ts`) in their `execFile` environment for the existing `gh auth git-credential` credential helper (set up by `finishProvisioning`, `src/workspace.ts:114-125`, which does apply unchanged) to authenticate them. A repo whose default branch isn't literally named `master` (e.g. `main`) has no `origin/master` to sync against; this surfaces as a sync error via the status icon like any other failure, not a special case.
- **Status icon has four states, is read-only.** The metadata row's GitHub sync indicator shows `provisioning` (first-open only, before the shared workspace is ready), `syncing`, `synced`, or `error`, each with a fixed tooltip string naming the current state — the `error` tooltip is the static text "GitHub sync: error — see notifications", not per-failure detail, so no error-message field needs to travel with the state. It has no click behavior.
- **Fixed commit message.** Every sync commit uses the literal message `chore: planning`, with no per-save customization.

## Proposed changes

- A new `src/git-sync.ts` module, a single file — matching every existing git-related module in this codebase (`workspace.ts` at 153 lines, `git-status.ts` at 89, `github-token.ts` at 22, all single flat files despite comparable scope), not a preemptive multi-file directory. It owns the shared sync workspace end to end: ensuring it exists (provisioning it once, via `WorkspaceManager.create`/`src/workspace.ts`'s clone/HTTPS-rewrite/token machinery, under a fixed name rather than a tab label, and caching the in-flight readiness promise so concurrent callers never trigger a second clone — `WorkspaceManager.create` itself does not dedupe concurrent calls for the same name, so this cache is the module's own responsibility), and one internal pull-rebase step (`git pull --rebase` against `origin/master` with remote-wins conflict handling, with `GH_TOKEN` passed explicitly per the Design decisions) shared by both a pull-only cycle (used when a tab opens — nothing to commit or push) and a save-triggered sync cycle (commit with the fixed message, then the same pull-rebase step, then `git push`) — the two cycles differ only in whether they commit/push first, not in how they pull, so the pull-rebase logic is written once and called from both. If this module later grows past the 200-line limit, split it the way `src/sandbox/` is already split (by concern, e.g. workspace-lifecycle vs. sync-cycle) rather than compacting it — but that split isn't warranted up front.
- `Config` (`src/types.ts:598-613`) and `DEFAULT_CONFIG` (`src/config.ts:15-31`) gain a new setting holding the list of project-relative file paths eligible for syncing, loaded and persisted the same way as the existing flat settings.
- `EditorView` (`src/types.ts:97-113`) gains an optional field carrying the sync state for a synced file's tab — `'provisioning' | 'syncing' | 'synced' | 'error'` (absent entirely for a non-synced file's tab, so the new status icon simply doesn't render, the same "field present ⇒ feature active" convention `HarnessView.provisionError` already uses). No separate error-detail field is needed: the `error` state's tooltip is the fixed string described in Design decisions, not per-failure text, and the actual failure detail is already reported through the existing notifications tab.
- A new open path for a synced file (checked before `src/openers/editor.ts`'s `openInEditor`, since a config-listed project-relative path needs to resolve into the shared workspace instead of the main checkout) opens the tab immediately with the sync field set to `'provisioning'`, mirroring `HarnessManager.spawnTab`/`finishSpawn`'s immediate-placeholder-then-async-fill-in pattern, then fills in the real `path`/`url`/`size` (via the existing `registerFile` allow-list, same as `saveFile` already does for a renamed target at `src/editor/save.ts:33`) and clears the sync field to `'synced'` once the shared workspace's readiness promise resolves and an initial pull completes.
- `saveFile` (`src/editor/save.ts`) is extended so that, when the saved tab is a synced file, its existing synchronous write/size-update/`markSaved`/"Saved"-flash flow is unchanged, but afterward it also kicks off `git-sync`'s save cycle asynchronously (not awaited), updating that tab's sync field to `'syncing'` immediately and then to `'synced'`/`'error'` once the cycle resolves.
- Because every synced tab's `EditorWatchManager` watcher already points at its file inside the shared workspace, no new "notify open tabs" broadcast is needed: a pull that changes a file on disk is picked up by the existing `fs.watch`-based mechanism, refreshing clean tabs and leaving dirty ones alone exactly as `editor-tab.md`'s "Live reload of external changes" already documents.
- A new small, non-interactive status-icon component is added to `EditorMetaRow.tsx` (`web/src/editor/EditorMetaRow.tsx`), alongside the existing save and connections buttons, rendered only when the tab's `EditorView` sync field is present, showing `provisioning`/`syncing`/`synced`/`error` with a fixed, state-naming tooltip string per state and no click handler.
- The shared sync workspace is registered once via `WorkspaceManager.create` and torn down only via `WorkspaceManager.removeAll()` (`src/workspace-manager.ts:65`, called from `src/controller.ts:229` at shutdown) — it must never be passed to the per-tab-close `WorkspaceManager.remove(dir)` path, since it needs to outlive any individual tab.

Suggested landing order, to keep `check-diff` green at each step: (1) the `Config`/`EditorView` type additions, (2) the `src/git-sync.ts` module in isolation with its own tests, (3) the editor-open provisioning-placeholder path, (4) the `saveFile` hook, (5) the status-icon UI. Steps 3-4 depend on 1-2; step 5 depends on 1 and 3-4 having a real field to render.

## Tests

- `src/git-sync.test.ts` (colocated, per the existing `src/**/*.test.ts` convention): the shared workspace is provisioned lazily exactly once even when two synced files are ensured concurrently (asserting a single `provisionWorkspace`/clone call); a save cycle commits with the message `chore: planning`, pulls/rebases before pushing, and resolves conflicts by taking the remote content; the pull-only cycle (tab-open path) exercises the same pull-rebase behavior without committing or pushing; `GH_TOKEN` is present in the environment of the pull/push `execFile` calls.
- `src/config.test.ts`: the new sync-paths setting round-trips through `loadConfig`/`updateConfig` like the other existing settings.
- `src/editor/save.test.ts` (extend): saving a synced file still writes/updates size/flashes "Saved" synchronously even if the git-sync cycle is slow or rejects; the tab's sync field transitions `syncing` → `synced`/`error` once that cycle resolves.
- Web editor tests (mirroring existing `EditorMetaRow.test.tsx`/`EditorTab.test.tsx` conventions): the new status icon renders each of the four states (`provisioning`/`syncing`/`synced`/`error`) with the correct tooltip and does not render at all when the sync field is absent; a dirty tab is left untouched when its file changes externally (already-covered live-reload behavior, exercised again here for a workspace-backed path) while a clean tab reloads; a first open on a not-yet-provisioned sync workspace shows the `provisioning` state until the placeholder tab is filled in.

## Out of scope

- Any interactive enable/disable control for syncing in the editor UI — syncing is controlled only by the config file's path list.
- Syncing files on any branch other than one literally named `master` — a repo whose default branch is named something else (e.g. `main`) is not supported and is not detected specially; it simply has no `origin/master` to sync against and surfaces as a sync error.
- Separate workspace clones per file or per tab — always the one shared sync workspace.
- Conflict-resolution UI or merge tooling of any kind — the remote always wins automatically.
- A configurable or per-file commit message — always the literal `chore: planning`.
- Any retry control or blocking dialog on sync failure — surfaced only through the status icon's error state and the existing notifications tab.

## Open questions

- The exact key name/shape for the new sync-paths config setting (e.g. a flat array of project-relative paths vs. a nested object) is left to implementation; no product behavior depends on it.
- Whether the sync-paths list matches exact file paths only, or also supports directories/glob patterns, was not raised during questioning; default to exact project-relative file paths only unless revisited during implementation.
