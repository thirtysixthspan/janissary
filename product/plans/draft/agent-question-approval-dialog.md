# Agent question / approval dialog API

**Complexity: 8/10** — a new first-of-its-kind cross-process channel: two `janus` client subcommands, a blocking token-gated HTTP endpoint with a server-side per-tab question registry, a new client RPC + non-modal per-tab panel, a new notification event, and — the sharpest part — extending tab-process env injection (server URL + tab label) across three distinct spawn sites (shell, harness PTY, ACP) so the CLI works from a harness, an ACP agent, or a shell script alike.

## Summary

Give an agent or script running inside a Janissary tab — a harness, an ACP agent, or a plain shell script in a workspace — a way to ask the human a question and block on the answer, surfaced as a dialog in Janissary that shows the agent asking. The mechanism is a pair of `janus` client subcommands: `janus ask "<question>"` for a free-text answer, and `janus approve "<question>" <option> [<option> …]` for a choice among named options. The subcommand connects to the running server, identifies the tab it is running in (via the `JANUS_AGENT_NAME` env var already injected into tab processes), registers a pending question, and blocks. Janissary renders a non-modal panel anchored to that tab (reusing the existing dialog components), and posts a `Question from <tab>` notification so the user notices when the tab isn't focused. When the user answers, the chosen option label or typed text is returned on the subcommand's stdout with exit code 0. If the user cancels or dismisses (or an optional `--timeout` elapses), the subcommand prints an empty string — or a caller-supplied `--default <value>` — and still exits 0, so the calling agent can always proceed; only a genuinely broken call (bad usage, no reachable server, no tab context) exits non-zero. Each tab has at most one pending question at a time; a second `ask`/`approve` from the same tab blocks until the first resolves.

## Design decisions

1. **Two subcommands, not one.** `janus ask "<question>"` collects free text; `janus approve "<question>" <option>…` collects a choice among the supplied option labels. They are distinct subcommands rather than one command that infers the shape from whether options were passed.

2. **Both surface the same underlying "pending question," differing only in answer shape.** `ask` renders a text input; `approve` renders one button per option. The server-side registry, tab correlation, blocking, notification, and cancel/timeout handling are identical for both; only the input control and the returned value differ (typed string vs chosen option label).

3. **The subcommands are foreground *client* calls to a running server, like `janus stop`.** `bin/janus.mjs` already treats `stop`, `--help`, and `--version` as foreground (attached) invocations distinct from the detaching server launch. `ask` and `approve` join that foreground set: they do not launch a server — they connect to one already running, block on the answer, print it, and exit.

4. **The server is reached over its existing HTTP server, token-gated, addressed by an injected env var.** The server already runs an HTTP+WebSocket listener (`src/index.ts`); its URL is `http://127.0.0.1:<port>/?token=<token>` (`src/index.ts:144`) and is written to stdout as `__JANUS_URL__ <url>` (`src/main.ts:179`), which `bin/janus.mjs` redirects to `.janissary/log/server.log`. The new endpoint reuses the exact guards `/open/<id>` uses: `originAllowed` + `tokenMatches(token, tokenFromReq(request))` (`src/index.ts:64,68-69`, `src/security.ts`). A plain CLI POST passes them because `originAllowed` requires only a loopback `Host` and checks `Origin` *only when present* (`src/security.ts:16-27`) — a CLI/curl request sends a loopback `Host` and no `Origin` — provided the URL carries `?token=<token>`. Because a workspaced process is sandboxed and may not read the root project's `.janissary/log`, the primary discovery path is an injected `JANUS_URL` env var (Decision 5a); reading `.janissary/log/server.log` is only the unsandboxed fallback. The sandbox permits localhost network by default; under `--offline` it denies `network*` (`product/specs/sandbox.md`), so an offline workspace's call fails fast (a broken call — see Decision 9).

5. **The question is correlated to the invoking tab via `JANUS_AGENT_NAME`.** The subcommand sends the tab label from `JANUS_AGENT_NAME`; the server resolves it against `managers.tab`. A call with no tab context (env var unset) or an unresolvable label is a broken call, not a cancel.

5a. **Env injection must be extended to all three spawn sites — it is not there today.** `JANUS_AGENT_NAME` is currently injected only at the app-command shell (`src/shell-manager.ts:50`, via `spawnShell`'s `extraEnvironment`, `src/shell.ts:6-11`) — **not** into harness PTYs or ACP subprocesses. To make `janus ask/approve` work "from a harness, acp, shell script," inject both `JANUS_AGENT_NAME` and the new `JANUS_URL` (the full token-bearing URL) at each of the three spawn sites: `spawnShell` (`src/shell.ts`), `spawnPty` (`src/pty.ts:29`, whose `extraEnv` at `:37` is merged *after* the sandbox scrub at `:47` — the safe place), and `connectAcp` (`src/acp/index.ts:28`, via its `options.env`). Neither `JANUS_URL` nor `JANUS_AGENT_NAME` matches `ENV_SCRUB_PATTERNS` (`src/sandbox/paths.ts:151-156` — AWS/token/secret/credential names only), so they survive `scrubEnv` for the shell/ACP paths that route env through it. The server URL and the owning tab's label must be threaded from the controller to these spawn sites (the controller holds the `RunningServer.url`/`token`, `src/index.ts:38,144`). This mirrors how `JANISSARY_NODE` is added for sandboxed processes (`src/sandbox/index.ts:218`).

6. **The dialog is a non-modal panel anchored to that tab, reusing existing dialog components.** It renders through the existing dialog shell components (`ConfirmDialogShell` / `ModalDialog` and the dialog keyboard hooks in `web/src`) but non-modally, so other tabs stay usable and other agents' questions never block the whole app. The panel is marked as the agent asking (the tab's identity and the question text), showing option buttons (`approve`) or a text input (`ask`).

7. **A `Question from <tab>` notification fires when the owning tab isn't focused.** Reusing the notification plumbing (`notify` in `src/notifications.ts`), a new notification event surfaces the pending question in the notifications tab with a link to the owning tab, on the same footing as the `auto-approve` event (always eligible, bypassing focus suppression, since it is an explicit agent action awaiting the user).

8. **Cancel and timeout return the default and exit 0.** On the user dismissing the panel, on an optional `--timeout <seconds>` elapsing, and on the owning tab closing while a question is pending, the subcommand prints the caller-supplied `--default <value>` (empty string if none) to stdout and exits 0. This lets the calling agent always make progress. Timeout is opt-in per call; with no `--timeout` the question waits until answered, cancelled, or the tab closes.

9. **Only a broken call exits non-zero.** Malformed usage (e.g. `approve` with no options), no reachable server, and no/unknown tab context write an error to stderr and exit non-zero. These are distinguishable from a cancel (Decision 8), which is a normal outcome.

10. **One pending question per tab; a second blocks server-side.** The registry allows at most one pending question per tab. A second `ask`/`approve` from the same tab is held (its HTTP call blocks) until the first is resolved, then its panel appears. This keeps the per-tab panel unambiguous without the caller needing to coordinate.

11. **Nothing is persisted.** A pending question is in-memory server state; it is not saved and not restored on `--relaunch`. A question pending when the server stops is simply lost (its blocked call ends when the connection drops).

## Proposed changes

### CLI: `janus ask` and `janus approve` client subcommands

Add the two subcommands as foreground client calls, dispatched exactly like `stop`: extend `parseCliArgs` (`src/cli-args.ts`) to recognize `ask`/`approve` (with `--default <value>` and `--timeout <seconds>`), and branch in `boot()` (`src/main.ts:153`, right where `args.stop` returns before `startServer`) to a new client module (e.g. `src/ask-client/`) instead of launching a server. Also add `ask`/`approve` to `bin/janus.mjs`'s `isForeground` set (`:28`) so the launcher runs them attached (stdout/stderr inherited) rather than detaching a server. The client module resolves the server URL+token from `JANUS_URL` (primary; the unsandboxed fallback reads the `__JANUS_URL__` line from `.janissary/log/server.log`), reads the tab label from `JANUS_AGENT_NAME`, POSTs the question, blocks on the response, prints the answer (or the `--default` value / empty string on cancel/timeout) to stdout, and sets the exit code per Decisions 8–9. Update `src/cli-info.ts`'s help text to document both subcommands. Keep each new module under the 200-line cap and use `.js` import extensions (NodeNext) throughout `src/`.

### Server: a question registry and token-gated HTTP endpoint

Add a new module (e.g. `src/questions/`) holding an in-memory registry keyed by tab label, each entry a pending `{ id, tab, kind, question, options?, resolve }`. Add a POST route to `serveStatic` in `src/index.ts` (`:63`, alongside the `/open/<id>` branch at `:68`) reusing the same `originAllowed` + `tokenMatches(token, tokenFromReq(request))` guard (`:64,69`). The body carries `{ tab, kind: 'ask' | 'approve', question, options?, default?, timeoutMs? }`: the handler resolves the tab against the controller's tab manager (unknown/absent tab → a 4xx that the client maps to a non-zero exit, Decision 9), enforces one-pending-per-tab by holding the request when the tab already has one (Decision 10), registers the pending question, emits a state update so clients render the panel (via the controller's `emitState`, `src/index.ts:55`), arms the optional timeout, and holds the HTTP response open until the question resolves (answer, cancel, timeout, or tab close), then responds with the resolved value. Closing the owning tab resolves its pending question as a cancel (Decision 8). Note the server auto-exits ~100ms after the last *WebSocket* (browser) client disconnects (`src/index.ts:114-120`); a blocked `janus ask` HTTP call does not keep it alive, so if all windows close the pending question is lost (Decision 11) and the CLI's dropped socket is treated as cancel → default (Decision 8).

### Server: extend env injection to all three spawn sites

Per Decision 5a, thread the server's token-bearing URL and each tab's label into the process environment at all three spawn sites so `janus ask/approve` can reach the server from a shell, a harness PTY, or an ACP subprocess. Inject `JANUS_URL` and `JANUS_AGENT_NAME` via `spawnShell`'s `extraEnvironment` (`src/shell.ts:6-11`, already used at `src/shell-manager.ts:50`), `spawnPty`'s `extraEnv` (`src/pty.ts:37,47` — applied after the sandbox scrub, so no scrub concern), and `connectAcp`'s `options.env` (`src/acp/index.ts:28` — survives `scrubEnv` since `JANUS_*` isn't in `ENV_SCRUB_PATTERNS`). The server URL/token comes from `RunningServer` (`src/index.ts:38,144`) and must be plumbed to the controller/managers that own these spawn calls; the tab label is already known at each spawn site (it is what `JANUS_AGENT_NAME` carries today for the shell).

### Server: view field, answer RPC, and notification

Add a per-tab pending-question field to the tab view in `src/protocol.ts` (the question text, kind, and options; near the existing view fields at `:60-76`), populated when a question is registered and cleared when resolved. Add a `ClientMessage` RPC (e.g. `answerQuestion`) to the method union in `src/protocol.ts`, routed through the existing WS dispatch `handle(...)` (`src/index.ts:108-109`, `src/message-handler.ts`), carrying the tab/id and the answer (typed text, chosen option label, or a cancel marker); its handler resolves the registry entry, unblocking the held HTTP response. Add a new notification event (e.g. `question`) to `NotificationEventType` and `notificationText` in `src/notifications.ts` (`:9-16`, `:56-66`), always eligible like `auto-approve` (bypassing focus suppression), fired when a question is registered on a non-focused tab, formatted `Question from <tab>` and passing the owning tab via `notify`'s `openFile`/link mechanism (`src/notifications.ts:72`).

### Client: the non-modal question panel

Render the pending-question view field as a non-modal panel anchored to the owning tab, reusing `ConfirmDialogShell`/`ModalDialog` structure and the dialog keyboard hooks but without trapping global input. For `approve`, show one button per option; for `ask`, show a text input with a submit affordance; both offer a dismiss/cancel control. Answering or cancelling issues the `answerQuestion` RPC and clears the panel. The panel is marked to show the tab/agent is the one asking, and the question text is rendered verbatim.

### Skill and spec

Add `skills/ask-user/SKILL.md` following the existing skill format, documenting when to use `janus ask` / `janus approve`, the exit-code and default/timeout contract, and short examples. Add a new spec `product/specs/agent-questions.md` describing the subcommands, transport and env-var contract, the non-modal per-tab panel, one-pending-per-tab, the notification, and the cancel/timeout/tab-close/error semantics; cross-reference it from `harness.md` (near auto-approve) and `notifications.md`.

## Tests

- **Server, question registry** (colocated `.test.ts` beside `src/questions/`): registering a question exposes it on the tab view; answering resolves with the given value; cancelling resolves with the default; a timeout resolves with the default; closing the owning tab resolves pending questions as cancel; a second registration for the same tab is held until the first resolves (Decision 10).
- **Server, HTTP endpoint** (`src/*.test.ts` following the `src/index.ts` / `src/security.ts` test conventions): a bad/missing token is rejected; a loopback request with no `Origin` and a valid token is accepted (guards the CLI-transport assumption in Decision 4); an unknown/absent tab yields the broken-call error; a well-formed `ask` and `approve` round-trip through a fake client answering via the RPC.
- **Server, env injection**: `spawnShell`, `spawnPty`, and `connectAcp` each carry `JANUS_URL` and `JANUS_AGENT_NAME` into the child env (Decision 5a), and both survive `scrubEnv` (assert against `ENV_SCRUB_PATTERNS`). Extend the existing `src/sandbox` / spawn-site tests rather than adding a new suite where one already covers the site.
- **CLI client** (`src/ask-client/*.test.ts`): argument parsing for `ask`/`approve`, `--default`, `--timeout`; URL/token resolution from `JANUS_URL` and from the `server.log` fallback; missing `JANUS_AGENT_NAME` → non-zero; cancel/timeout prints the default and exits 0; a malformed `approve` (no options) → non-zero. Stub the HTTP call so no real server is needed.
- **Client, panel** (`web/src` colocated test, following the existing dialog component tests, e.g. `web/src/QuitDialog`): `approve` renders option buttons and emits the chosen label; `ask` renders a text input and emits the typed text; the cancel control emits the cancel marker; the panel is non-modal (does not trap unrelated input).

## Out of scope

- **MCP or any generic tool-protocol integration** — this is a purpose-built CLI + HTTP endpoint, not MCP server support (consistent with the backlog note that this should be tooling around API calls, not MCP).
- **Rich input beyond free text and a flat option list** — no multi-field forms, file pickers, or nested/multi-select choices in this version.
- **Persisting or restoring pending questions across `--relaunch` or server restart** (Decision 11).
- **Reaching a server on another host** — the transport targets the local server the tab belongs to; no remote addressing.
- **Answering from outside Janissary** (e.g. answering via the CLI or an external channel) — the human answers in the Janissary UI panel or its notification link only.
- **Changing the existing auto-approve behavior** — that flow (Janissary answering agent permission prompts automatically) is untouched; this is a separate, human-answered channel.
