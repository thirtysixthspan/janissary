# Electron packaging — high-level plan

## Why this is tractable

The app already has an architecture that maps cleanly onto Electron: `src/main.ts` boots a Node HTTP+WebSocket server (`src/index.ts`, `startServer()`) that serves the built `web/dist` bundle and a token-authenticated WS API, then spawns the user's system Chrome in `--app=` (kiosk-ish, chromeless) mode pointed at `http://127.0.0.1:<port>/?token=...` (`openApp()` in `src/main.ts:94-112`). Electron is just replacing that last step — instead of shelling out to system Chrome, an Electron `BrowserWindow` loads the same URL, and the Node server that used to run as a separate child process becomes Electron's own main process (or a child process it spawns).

Auth already works over `http://` origin with a query-string token (`web/src/ws.ts:18-19`, `src/security.ts`'s `originAllowed`/loopback check) — nothing here depends on being loaded from a real browser's chrome (tabs/omnibox); a `BrowserWindow` pointed at the same URL satisfies it unchanged.

## High-level phases

1. **Add Electron as a dependency and a minimal main process.**
   New `electron/main.ts` (or `electron/main.js` after build) that, on `app.whenReady()`, does what `boot()` in `src/main.ts` already does (acquire lock, init state dirs, `startServer(...)`), then opens a `BrowserWindow` at the returned `server.url` instead of calling `openApp()`. Reuse `startServer` directly — no need to duplicate server logic.

2. **Decide process topology: in-process vs. child process.**
   - *In-process*: Electron's main process directly calls `startServer()` in the same Node runtime Electron embeds. Simplest, avoids spawning anything, but means native addons (`node-pty`) must be built against Electron's Node ABI, not system Node's.
   - *Child process*: Electron spawns the existing CLI (`bin/janus.mjs --no-open`) as a subprocess and just points a `BrowserWindow` at the printed `__JANUS_URL__`. Keeps the server on plain Node (no Electron-ABI rebuild needed for `node-pty`), closer to today's proven code path, but means shipping a bundled Node runtime or requiring one on the host — largely defeats the point of an Electron build (self-contained app), unless Electron's own Node (via `utilityProcess` or `ELECTRON_RUN_AS_NODE`) is used to run it, which again hits the ABI question.
   - Recommendation: in-process, using `electron-rebuild` (or `@electron/rebuild`) to rebuild `node-pty` for Electron's ABI as part of the packaging step. This is the standard approach for Electron apps that need native pty support (VS Code, Hyper, etc. all do this).

3. **Replace the frame-enabler Chrome extension with Electron's `session.webRequest` API.**
   The bundled `chrome-extension/` (`declarativeNetRequest`, strips `X-Frame-Options`/CSP `frame-ancestors` for page-tab embedding) exists only because the app embeds arbitrary sites inside iframes in a real, unmodified Chrome. In Electron this whole mechanism (and the CDP-based unpacked-extension-loading workaround in `product/plans/ready/chrome-extension-load-flag-removed.md`) can be deleted and replaced with `session.defaultSession.webRequest.onHeadersReceived` stripping the same headers directly — no extension, no CDP fragility, no dependence on the user's installed Chrome version. This is a net simplification, not just a port.

4. **Build tooling and packaging config.**
   - Add `electron`, `electron-builder` (or `electron-forge`) as devDependencies.
   - New `electron-builder.yml` (or `forge.config.js`) describing app id, icons, `mac`/`win`/`linux` targets, and `files`/`extraResources` (needs `dist/`, `web/dist/`, `chrome-extension/` removed once step 3 lands, `agent-names.json`, `help.md` — same list as the existing npm `files` field).
   - New root scripts: `electron:dev` (run Electron pointed at `dev:web`+`dev:server` for hot reload), `electron:build` (build server+web, rebuild native deps for Electron ABI, then package via electron-builder), following the existing `./scripts/run.mjs` convention — likely a new `scripts/electron-build.mjs` wired into the runner rather than a raw npm script, per this repo's script conventions.
   - Code-signing/notarization (macOS) and Windows signing are separate follow-up concerns — flag but don't block on them for a first internal build.

5. **Multi-instance and single-instance semantics.**
   Today, multiple `janus` invocations in different project directories run as independent processes, each with its own lock (`src/instance-lock.ts`) and port. An Electron app is normally single-instance by convention (`app.requestSingleInstanceLock()`), with one main process spawning multiple windows. Decide: does opening a second project directory in the Electron app open a second `BrowserWindow` against a second in-process server (multiple `startServer()` calls, one per window/project), or does it stay one-process-per-project like today (each launch is its own Electron process)? This changes how much of `src/main.ts`'s per-launch setup (lock/state-dir init, `clearStateDirectory()` etc.) needs to become per-window instead of per-process.

6. **CLI coexistence.**
   `janus` (the terminal-launched CLI) should presumably keep working as-is — the Electron app is an additional distribution, not a replacement. Verify `bin/janus.mjs` and the new Electron entrypoint can both exist without the packaged Electron app also registering itself as a global `janus` shell command (unless that's desired), and that `npm run build` output isn't restructured in a way that breaks the existing CLI packaging (`files` in `package.json`).

## Major gaps / open questions to resolve before implementing

- **`node-pty` native rebuild.** Confirmed native addon (prebuilds + a `postinstall` chmod step) used for the shell/terminal feature. Needs `electron-rebuild` wired into the packaging script, and prebuilt binaries (or a rebuild step) per target OS/arch in CI. This is the single biggest technical risk — worth a spike before committing to the plan.
- **`playwright`'s role is dual and only one half is Electron's concern.** It's used two ways: (a) `src/browser/index.ts` launches a real headless/headed Chromium via `chromium.launch()` for an agent-facing browser-automation feature (page.tab? confirm exact feature) — this is unrelated to the app's own UI shell and keeps needing its own Chromium install (`playwright:install-chromium`) regardless of Electron; (b) `product/plans/ready/chrome-extension-load-flag-removed.md` uses `chromium.connectOverCDP` purely to load the frame-enabler extension into system Chrome — step 3 above deletes this second use case entirely. Packaging needs to keep bundling/installing Playwright's Chromium for (a) but that's an existing concern, not a new one from Electron.
- **Window-management feature parity.** `src/main.ts` currently manages exactly one external app window per process and kills it on exit (`killApp()`). Electron's `BrowserWindow` lifecycle, multi-window support, and OS-level window chrome (menus, dock icon, About panel, updater) are effectively unbuilt — this plan only covers getting one window to boot and load the existing UI, not full native-app polish (app menu, auto-update, deep links, tray icon), which should be scoped as follow-up plans if wanted.
- **Auto-update.** Not addressed here. If wanted, needs `electron-updater` + a release/publish pipeline decision (GitHub Releases, S3, etc.) — separate plan.
- **Existing chrome-extension code path becomes dead weight for the Electron build but still needed for the CLI+system-Chrome path** (unless that path is being retired). Decide whether the Electron app *replaces* the system-Chrome launch mode entirely or ships alongside it as an alternative distribution — this determines whether `chrome-extension/` and the CDP loader plan get deleted or kept.
- **Security posture change.** Today's loopback-origin + query-token model was designed for "a page loaded in the user's own real browser talking to a localhost server." Electron collapses attacker surface differently (no other browser tabs/extensions can reach the port from a hostile page, since there's no ambient browser) but introduces new ones (Electron's own security footguns: `nodeIntegration`, `contextIsolation`, preload script scope). Needs an explicit review of `webPreferences` (recommend `contextIsolation: true`, `nodeIntegration: false`, no preload access beyond what's needed) rather than assuming the existing token/origin checks are sufficient on their own.
- **CI/build machines for cross-platform packaging.** electron-builder needs macOS to build/sign macOS targets; confirm CI has (or plans to add) the right runners for whichever platforms are in scope.
- **Server auto-shutdown assumes exactly one client.** `startServer()` tears the server down ~100ms after the last WebSocket client disconnects. That's fine for today's one-process-per-window model but directly conflicts with the "one in-process server, multiple `BrowserWindow`s" option in phase 5 above — a second window's disconnect (e.g. a reload) could kill the server out from under the first. Needs an explicit decision, not just a topology choice.
- **Token is only checked on the WS upgrade and the `/open/<id>` route, not on `index.html`/static assets** — those are origin-checked only. This is a pre-existing, presumably-accepted trust boundary for the browser-window case; worth a conscious re-check under Electron's threat model rather than carrying it over silently (see security-posture point above).
- **`git`/`gh`/`ssh` CLI dependencies.** Several features (`workspace.ts`, `harness/manager.ts`, sandbox code, git-backed recognizers) shell out to `git`, `gh`, and `ssh` on `PATH`. An Electron app launched from Finder/Explorer/Dock (rather than a terminal) may not inherit the same `PATH` a shell-launched CLI does — a known Electron/macOS gotcha (GUI apps get a minimal launchd `PATH`, missing Homebrew/nvm paths). Needs a `PATH`-resolution strategy (e.g. shell out through the user's login shell to capture `PATH`, as apps like VS Code do) or these features silently break when launched as a GUI app.
