# Recognize claude code busy/ready status from captures

**Complexity: 4/10** — a small new detector module following an exact existing precedent, but correctness hinges on fixing a real single-callback-slot constraint in `HarnessScreenReader`'s wiring (see decision 5) that would otherwise silently limit busy/ready tracking to auto-approving sessions only.

## Summary

Today a harness tab's dot blinks (`tab.busy`) for the entire lifetime of the process — `addBusy` runs once at spawn and is never cleared until the harness exits, so the blinking dot doesn't actually mean "thinking right now." This feature makes the blinking dot accurate for the claude harness by recognizing, from the same rendered-screen captures auto-approve already reads (`HarnessScreenReader`/`ScreenCapture`), whether claude is actively generating ("busy") or sitting idle at its own prompt ("ready"), and calling `addBusy`/`deleteBusy` accordingly. The detection table is structured per-harness from the start (mirroring the existing permission-gate table's shape), so other harnesses can get their own detectors later without restructuring, even though only claude gets a real detector in this pass.

## Design decisions

1. **Detection signal.** The exact capture-text pattern(s) that distinguish claude "busy" (actively generating, e.g. a spinner/status line) from "ready" (sitting at its own empty prompt) are determined from real captures during implementation, the same way the existing permission-gate patterns in `auto-approve.ts` were originally captured and encoded — the plan does not hardcode literal text now.
2. **Harness scope.** The detector is a per-harness table from day one (same shape as `GATE_TABLE` in `auto-approve.ts`), so a harness with no entry falls back to today's coarse behavior (busy for the whole process lifetime). Only claude gets a real detector in this pass; opencode/codex are later work, consistent with how the gate table is scoped today.
3. **Interaction with the permission gate.** While claude is showing its permission-gate prompt, the tab's dot stops blinking (claude itself is idle, blocked on the user, not doing work) — but if the gate is not auto-approved, the tab's unread badge is set (via the existing `markUnread`) to call the user's attention to it, the same way an unread badge already flags other tab activity needing attention.
4. **Initial state.** A harness tab starts "busy" (blinking) immediately on spawn, exactly as today, until the first screen capture is read and classified — avoiding a flash of "ready" before the process has produced any output yet.
5. **`HarnessScreenReader` takes exactly one `onCapture` callback — busy/ready detection cannot simply be "a second consumer."** `HarnessScreenReader`'s constructor (`src/harness/screen.ts:29`) accepts a single optional `onCapture?: (capture: ScreenCapture) => void`. Today, `spawnTab` (`src/harness/manager.ts:112`, via `autoApproveHandler` at `:128-137`) passes `undefined` for this callback whenever `autoApprove` is false — meaning **no callback fires at all, for any purpose, on a non-auto-approving harness tab today**. Busy/ready tracking must work regardless of whether `-y`/auto-approve is enabled (per decision 3's gate interaction, which isn't conditioned on auto-approve). So `spawnTab` must build one **combined** callback instead of the current either/or: call the auto-approve handler's logic first (if `autoApprove` is true), then the busy/ready logic (if the harness has a real `BUSY_TABLE` entry) — constructing `undefined` only when *neither* applies, to avoid adding capture-handling overhead to a harness with no auto-approve and no busy detector (e.g. opencode today).
6. **Detecting "not auto-approved" for the unread badge requires a small, new public accessor.** `HarnessAutoApprover` tracks whether it's stuck on a repeated, un-clearable gate via a *private* `stuckNotified` field (`src/harness/auto-approve.ts:57`) — there is currently no way for anything outside the class to read this. The busy/ready handler (decision 3) needs it to decide whether to call `markUnread`: mark unread when either `autoApprove` is off entirely, or the approver for this harness reports it's stuck. `HarnessAutoApprover` gains a small public getter (e.g. `get isStuck(): boolean { return this.stuckNotified; }`) for this purpose — no other change to the class. Within the combined callback (decision 5), the auto-approve handler's `onCapture` (which updates `stuckNotified` synchronously) must run **before** the busy/ready handler reads `isStuck` for the same capture, so the just-updated stuck state is visible in the same cycle.

## Proposed changes

1. **New per-harness busy/ready table** (e.g. `src/harness/busy-status.ts`, mirroring `auto-approve.ts`'s shape): a `BUSY_TABLE: Record<string, { busy: (text: string) => boolean }>` (or equivalent), with a `detectBusy(text, harnessName): boolean` export that looks up the harness's entry and falls back to a value meaning "no real detector — keep existing coarse behavior" for any harness without one. The claude entry's `busy` function is authored against real captured screen text gathered during implementation (per decision 1).
2. **`src/harness/auto-approve.ts`**: add a public `get isStuck(): boolean { return this.stuckNotified; }` to `HarnessAutoApprover` (decision 6) — no other change to the class.
3. **New busy/ready capture handler** (in the new `src/harness/busy-status.ts`, or as a private method in `manager.ts`): a function taking `(name, label, managers, approver: HarnessAutoApprover | undefined)` and returning a `(capture: ScreenCapture) => void` that: looks up `BUSY_TABLE[name]`; if there's no entry, does nothing (preserving today's coarse behavior); otherwise calls `detectPermissionGate(capture.text, name)` — if a gate is showing, calls `deleteBusy(label)`, and additionally calls `markUnread(label)` when `!autoApprove || approver?.isStuck` (decision 6); if no gate is showing, calls `detectBusy(capture.text, name)` and sets `addBusy(label)`/`deleteBusy(label)` accordingly.
4. **`src/harness/manager.ts`'s `spawnTab`**: replace the current `this.autoApproveHandler(name, label, id, autoApprove)` argument to `HarnessScreenReader`'s constructor (`:112`) with a combined callback per decision 5 — build the auto-approve callback (if `autoApprove`) and the new busy/ready callback (if `BUSY_TABLE[name]` exists), and pass a function that calls whichever of the two apply, auto-approve's first (so `isStuck` reflects this capture, per decision 6), or `undefined` if neither applies. `autoApproveHandler`'s existing internals (constructing `HarnessAutoApprover`, registering it in `this.autoApprovers`) are otherwise unchanged; the busy/ready callback needs a reference to that same approver instance (or `undefined`) to read `isStuck`.
5. **`product/specs/harness.md`**: add a section describing claude-specific busy/ready recognition, its interaction with the permission gate and the unread badge, and that other harnesses keep the existing coarse (spawn-to-exit) busy behavior until they get their own detector.

## Tests

- `src/harness/busy-status.test.ts` (new): the claude busy/ready detector correctly classifies captured fixture text as busy vs ready (fixtures authored from real captures gathered during implementation); a harness with no table entry (e.g. opencode) always reports "no real detector" regardless of input text.
- `src/harness/auto-approve.test.ts`: add a case confirming `isStuck` is `false` initially and after a cleared gate, and `true` once the existing stuck-notification path fires (the test already at `:143`, asserting `notify` is called with `'Auto-approve could not clear the permission prompt; standing down'`). Existing tests continue to pass unchanged — the gate detector and approve logic aren't modified.
- `src/harness/manager.test.ts`: the shared `makeManagers()` fixture (`:26-52`) mocks `tab.addBusy` as a plain no-op function and has no `tab.deleteBusy` or `tab.markUnread` at all — upgrade `addBusy` to `vi.fn()` and add `deleteBusy: vi.fn()` and `markUnread: vi.fn()` to the mock's `tab` object before any of the assertions below are possible. Then: spawn `harness claude` **without** `-y` and emit a `pty` `data` event that renders as "busy" text — confirm `addBusy`/`deleteBusy` still fire (proving decision 5's combined-callback fix actually applies outside auto-approve, which is the bug this decision exists to close, not just a regression guard); a capture classified "ready" clears busy; a capture classified "busy" (re-)sets it; a detected, non-auto-approved gate clears busy and calls `markUnread`; a harness with no detector entry (opencode) keeps busy set across captures exactly as before this change (no new calls to `deleteBusy`); when both auto-approve and a busy entry apply, the auto-approve keystroke still fires exactly as today's existing auto-approve tests already assert (`:187-196`).

## Out of scope

- Real detectors for opencode/codex — the table exists for all harnesses structurally, but only claude gets an actual detector in this pass.
- Any change to `HarnessView.status` (`'running'`/`'exited'`) or the process-exit lifecycle.
- Any change to the permission-gate detector or auto-approve keystroke logic itself.
- A visually distinct "gate pending" indicator beyond the existing unread badge (e.g. a third dot color or icon) — the badge is the only new signal for this case.
