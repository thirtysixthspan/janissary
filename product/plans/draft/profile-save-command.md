# Save the current UI as a named profile

**Complexity: 6/10** — the entry/reserved-file writers are straightforward inversions of loaders that already exist, but two things push this up: a new client→server RPC plus a server-side holder to make live sidebar/tab-area sizes readable at capture time, and a new CDP get-bounds path mirroring the existing set-bounds resizer. Both span server and web, and the monitor capture needs a new `MonitorManager` accessor.

## Summary

Add a `profile save <name>` subcommand that snapshots the running session into `profiles/<name>/` — the inverse of `profile launch`. It captures everything a profile can express: one entry file per open agent and harness tab, plus the reserved profile-level files for layout (`_layout.json`), monitors (`_monitors.json`), and docked file/notifications/schedules tabs (`_files.json`, `_notifications.json`, `_schedules.json`). The result is a committable, reusable profile that reproduces the current arrangement of tabs and layout when launched later.

The feature request calls this a "cli command," but a `janus` subprocess subcommand cannot see the live, in-process UI state — that state exists only inside the running app, where `profile launch`/`profile list` already live. So `profile save` is a new subcommand of the existing in-app `profile` command, dispatched by `ProfileManager` alongside `launch` and `list`. Agent entries are captured as clean reusable templates (name, cwd, group, dot color, tab order) with no command history or transcript, so a saved profile is a fresh starting point rather than a frozen session. Live sidebar/tab-area sizes — which today live only in the web client — are captured by having the client report each manual resize to the server so the server always holds the latest values; the OS window size is read over CDP. Saving over an existing profile overwrites it. Tabs that have no profile-entry representation (image, page, markdown, editor, ssh, and non-docked file-tree tabs) are skipped and named in the command's output.

## Design decisions

- **Command surface and name.** `profile save <name>` is a new subcommand of the in-app `profile` command, parsed by `parseProfileCommand` (`src/profile-command.ts:8`) and dispatched by `ProfileManager.run` (`src/profile/manager.ts:13`), beside `launch` and `list`. It is not a `janus` CLI subcommand, because capture requires live in-process state.
- **`<name>` is used verbatim.** The directory is `profilePath(name)` (`src/profiles.ts:28`) with no dasherization — consistent with how `launch`/`profileExists` treat the name. Whatever the user types is the directory name.
- **Capture scope.** Capture everything a profile can express: agent tabs, harness tabs, layout sizes, running monitors, and docked file/notifications/schedules tabs. Each maps to the reserved-file or entry-file schema that `profile launch` already reads.
- **The issuing tab is captured too.** Unlike `profile launch`, which skips the tab it was issued from, `profile save` captures every tab in `managers.tab.tabs`, including the one the command was typed in — it is an ordinary agent/harness tab in the snapshot.
- **Agent-entry fidelity.** Agent entries are clean reusable templates written in the `AgentState` schema (`src/types.ts:231`) with exactly these fields: `name` (the tab label), `dotColor`, `active: false`, `number`, `group`, `groupColor`, and `cwd` (from `cwdOf`). The transcript/history fields (`log`, `cmdHistory`, `context`, `commandQueue`) are deliberately omitted — a saved profile is a fresh template, not a resumable snapshot. (`buildAgentStateFromTab` at `src/tab/agent-state.ts:5` is the shape reference, but the writer omits the history fields it copies.)
- **Live `group`/`dotColor` are captured as-is, but launch flattens groups.** Each captured entry keeps its live `group`/`dotColor`. Note that `profile launch` places all of a profile's entries into a single new group regardless of the per-entry `group` values (see `product/specs/profiles.md`'s "Relaunching"/grouping paragraph), so a multi-group session does not round-trip its group split. This is accepted, not a bug to solve here.
- **Harness-entry fidelity.** Harness entries are written in the `ProfileHarnessEntry` schema (`src/types.ts:255`): `harness` (the kind, from `HarnessView.name`), `model`/`effort` (from the tab's `HarnessView` — `src/harness/manager.ts:157-158`), `workspace` (true when `tab.workspaceDir` is set), `offline`/`autoApprove`/`dotColor` (from the `Tab`, not the view), `cwd` (from `cwdOf`), `number`, and `group`. Harness `run`/`schedule` one-shots are memory-only (they leave no persisted state) and are not captured.
- **Layout source.** The web client reports each completed manual sidebar/tab-area resize to the server via a new RPC; the server retains the latest values in a module-level holder so `profile save` reads them synchronously when writing `_layout.json`. This complements today's server→client `layout` event (`src/protocol.ts:117`), which has no reverse channel.
- **Window size.** The OS window size is read via a new CDP get-bounds call (mirroring `resizeAppWindow` in `src/cdp-window-resize.ts`) and written to `_layout.json`'s `window` field. Under `--no-open` no window exists — the bounds reader is never registered (main.ts only registers the resizer inside its open-app branch at `src/main.ts:129`), so the window portion is skipped and a note is added to the output; sidebar/tab-area sizes are still captured.
- **Docked-tab fidelity is `dock`-only.** For docked file-tree, notifications, and schedules tabs, capture the side from the tab's own `dock` field (`Tab.dock`, which the server tracks). The notifications `focus` flag and which docked tab is visible live only in client state (`web/src/useLayoutState.ts` `focusLeft`/`focusRight`) and are **not** captured — `_notifications.json`/`_schedules.json` record `dock` alone. A docked file-tree tab is captured as `_files.json` `{ dock, path }`, where `path` is the tree's live absolute root (`tab.files.absoluteRoot`); the `in <tab>` form is not reconstructed.
- **Name collision.** Saving over an existing profile name removes and rewrites the `profiles/<name>/` directory — an unconditional overwrite, no prompt and no `--force` guard.
- **Unrepresentable tabs.** Tabs with no profile-entry form (image, page, markdown, editor, ssh, monitor reporting tabs, and non-docked file-tree tabs) are skipped and listed by label in the command output so the user knows they were dropped. (Monitor reporting tabs are captured as `_monitors.json` entries via the `MonitorManager`, not as tab entry files — so they are not in the "dropped" list.)
- **Success output.** On success the command prints a one-line summary of what was captured — counts of agents, harnesses, layout, monitors, and docked view tabs — followed by the skipped-tab list when non-empty.

## Proposed changes

**Command parsing (`src/profile-command.ts:8`).** Extend `parseProfileCommand` to recognize `save` as a third action returning `{ action: 'save', name }`, with `profile save` (no name) returning a `Usage: profile save <name>` error. Add the `save` variant to the `ProfileParsed` type (`src/types.ts`) and update `PROFILE_USAGE` to list all three subcommands.

**Command dispatch (`src/profile/manager.ts:13`).** Add a `save` branch to `ProfileManager.run` that delegates to the new capture module and prints its summary string via the same `out` helper used by `list`/`launch`.

**Capture orchestration (new module `src/profile/save.ts`).** A `saveProfile(name, managers)` entry point that: resolves the target directory via `profilePath`; if it exists, removes it recursively (`rmSync(dir, { recursive: true, force: true })`) then recreates it (`mkdirSync(dir, { recursive: true })`) — the unconditional overwrite; walks `managers.tab.tabs`, routing each tab to the right writer by `view`; writes the reserved files from the monitor manager, tab docks, and layout holder; and returns a summary object (per-kind counts plus the ordered list of skipped tab labels) that the manager formats into the one-line summary. This module will exceed the 200-line `max-lines` limit if it holds every writer inline — split the entry writers and reserved-file writers into sibling modules (e.g. `src/profile/save-entries.ts`, `src/profile/save-reserved.ts`) and keep `save.ts` as the orchestrator, matching how `src/profile/` already splits `agent-opener.ts`/`layout.ts`/`schedules.ts`.

**Entry writers.** The filename (minus `.json`) is the authoritative label on read (`loadProfileEntries` in `src/profiles.ts:53`), so each entry file is named `<tab.label>.json`.
- *Agent entries* (tab `view` undefined or `'agent'`): write an `AgentState` object with only `name`, `dotColor`, `active: false`, `number`, `group`, `groupColor`, and `cwd` (from `cwdOf(label)`). Omit `log`, `cmdHistory`, `context`, `commandQueue`, `schedule` — the clean-template decision above.
- *Harness entries* (tab `view === 'harness'`): write a `ProfileHarnessEntry` with `harness: tab.harness.name`, `model: tab.harness.model`, `effort: tab.harness.effort`, `workspace: tab.workspaceDir !== undefined`, `offline: tab.offline`, `autoApprove: tab.autoApprove`, `dotColor: tab.dotColor`, `cwd: cwdOf(label)`, `number`, and `group`. Omit undefined optionals rather than writing `null`. No `run`/`schedule` (memory-only).

**Reserved-file writers (new; each emits the exact JSON its loader in `src/profile-reserved-files.ts` parses).**
- `_monitors.json` — a `ProfileMonitor[]` built from the monitor manager's new capture accessor (below), one `{ persona, targets }` per live monitor: `persona` from `reg.persona.name`; `targets` an array where each `MonitorTarget` maps to its authored word (`t.kind === 'tab' ? t.label : 'group:' + t.group`), and an inline monitor (`reg.inline`) emits `targets: []`. Written only when at least one monitor exists.
- `_files.json` — a `ProfileFilesEntry[]` from every docked file-tree tab (`view === 'files'` and `dock` set), each `{ dock: tab.dock, path: tab.files.absoluteRoot }`. No `in`.
- `_notifications.json` — `[{ dock }]` when the singleton notifications tab is docked (`view === 'notifications'`, `dock` set). No `focus`.
- `_schedules.json` — `[{ dock }]` when the singleton schedules tab is docked (`view === 'schedules'`, `dock` set).
- `_layout.json` — a single `{ layout: { window?, sidebarLeft, sidebarRight, tabAreaPct } }` object from the server-retained client layout (below) plus the CDP-read window bounds; `window` is omitted when no bounds reader is registered (`--no-open`), and a skip note is added to the summary.

**Monitor capture accessor (`src/monitor/manager.ts`).** Add a public method to `MonitorManager` (e.g. `snapshot(): { persona: string; targets: MonitorTarget[]; inline: boolean }[]`) returning one record per entry in its private `monitors` map — the capture path must not read the private map or the display-only `list()` strings. Reach it from the save module via `managers.monitor`.

**Layout reporting (client→server).** Add a `reportLayout` RPC — `{ method: 'reportLayout'; params: { sidebarLeft: number; sidebarRight: number; tabAreaPct: number } }` in `RpcCall` (`src/protocol.ts:129`), a `case 'reportLayout'` in `src/message-handler.ts` delegating to a new `controller` method, and a corresponding `client.reportLayout(...)` sender in `web/src/ws.ts` — modelled on the client-only, no-reply `editorSync`/`syncEditorBuffer` pair. The web side sends it whenever a manual sidebar or split drag completes; wire it into the resize setters exposed by `web/src/useLayoutState.ts:42-45` (`setSidebarLeftWidth`/`setSidebarRightWidth`/`setReportingHeightPct`), converting the reporting-height back to `tabAreaPct` (`100 - reportingHeightPct`, the inverse of the mapping at `useLayoutState.ts:37`). The controller method stores the values in a new module-level holder (`src/client-layout.ts` with `setClientLayout`/`getClientLayout`), mirroring `window-resizer.ts`; last report wins.

**Window bounds reader (`src/cdp-window-resize.ts` + `src/window-resizer.ts` + `src/main.ts`).** Add `getAppWindowBounds(writePipe, readPipe)` to `src/cdp-window-resize.ts` using CDP `Browser.getWindowBounds` (companion to `resizeAppWindow`), a `setWindowBoundsReader`/`getWindowBoundsReader` pair in `src/window-resizer.ts` (or a second registry module), and registration in `src/main.ts` next to the `setWindowResizer(...)` call at `:129` — inside the open-app branch, so no reader exists under `--no-open`. The save module calls the reader when present and skips `window` otherwise.

**Spec (`product/specs/profiles.md`).** Add a `profile save <name>` section under the existing `profile launch`/`profile list` sections describing what is captured (agents/harnesses as clean templates, the reserved files), the verbatim-name/overwrite behavior, the `dock`-only fidelity for docked tabs (no `focus`, no `in`), the skipped-tab report, the `--no-open` window-size skip, and the summary output; update the `profile` command summary paragraph to list `save` alongside `launch`/`list`.

**Ordering and independence.** No dependency on another plan. Within this plan the pieces are independently typecheck-clean and can land in any order, but a sensible sequence keeps each checkpoint green: (1) parser + `ProfileParsed` variant; (2) `client-layout` holder and window-bounds reader (pure additions, no callers yet); (3) the save module + writers + `MonitorManager.snapshot`; (4) `ProfileManager` `save` branch wiring it in; (5) the `reportLayout` RPC on both sides; (6) `main.ts` bounds-reader registration; (7) spec. Layout capture yields defaults until (5) and (6) are wired, so land those before relying on non-default `_layout.json` in a manual check.

## Tests

All tests colocated as `src/**/*.test.ts` (vitest project `server`); web tests as `web/src/**/*.test.tsx` (project `client`), per `CLAUDE.md`. The strongest oracle is round-tripping: assert the writers' output parses back through the real loaders.

- **`src/profile-command.test.ts`** (extend if present, else colocate) — `parseProfileCommand` returns `{ action: 'save', name }` for `profile save writing`, a `Usage: profile save <name>` error for bare `profile save`, and the updated `PROFILE_USAGE` for malformed input.
- **`src/profile/save.test.ts`** (new) — with a fabricated tab set on a test `Managers`: writes one agent entry per agent tab with only the clean-template fields (`name`/`dotColor`/`active:false`/`number`/`group`/`groupColor`/`cwd`, and none of `log`/`cmdHistory`/`context`/`commandQueue`); writes a harness entry with `harness`/`model`/`effort`/`workspace`/`offline`/`autoApprove`/`cwd` and no `run`/`schedule`; emits `_monitors.json`/`_files.json`/`_notifications.json`/`_schedules.json` only when the corresponding monitors/docked tabs are present; overwrites an existing non-empty profile directory (a stale file from a prior save is gone afterward); and returns a summary listing skipped image/editor/ssh/etc. tab labels. Assert the emitted JSON round-trips through the real loaders (`loadProfileEntries`, `loadProfileMonitors`, `loadProfileFiles`, `loadProfileNotifications`, `loadProfileSchedules`, `loadProfileLayout`) — the profile a save writes must be launchable.
- **Monitor accessor** (`src/monitor/manager.test.ts`, extend) — the new `snapshot()` returns one record per live monitor with `persona`, the authored `targets` array (`group:<n>`/label), and `inline`; an inline monitor reports `inline: true` so the save writer emits `targets: []`.
- **`_layout.json` capture** (in `save.test.ts`) — the layout writer emits `sidebarLeft`/`sidebarRight`/`tabAreaPct` from `getClientLayout()` and includes `window` only when a window-bounds reader is registered; with no reader (the `--no-open` case) it drops `window` and the summary records the skip note.
- **Layout report plumbing** — server: a `reportLayout` RPC updates the `client-layout` holder so `getClientLayout()` returns the reported values (`src/client-layout.test.ts` or the message-handler/controller test). Web: `web/src/useLayoutState.test.tsx` (or the resize-handler component test) asserts a completed sidebar/split drag calls `client.reportLayout` with the current sizes and the `100 - reportingHeightPct` conversion.

## Out of scope

- Any change to `profile launch`/`profile list` behavior or to the reserved-file loaders themselves.
- Capturing harness `run`/`schedule` one-shot commands (memory-only; not recoverable).
- Capturing command history or transcript into agent entries (agents are saved as clean templates).
- Representing image, page, markdown, editor, ssh, or non-docked file-tree tabs as profile entries — these are skipped and reported.
- A `janus` CLI subcommand form, an overwrite-confirmation prompt, or a `--force`-style guard (overwrite is unconditional).
- Capturing the notifications `focus` flag or the file-tree `in <tab>` form (docked tabs record `dock`, and file trees record a literal `path` from `absoluteRoot`).
- Round-tripping the live per-tab group split (launch flattens all entries into one group).
- Editing profile files after capture (renaming entries, reordering) beyond what the snapshot writes.
