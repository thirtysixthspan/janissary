# Editor agent query line

**Complexity: 4/10** â€” client-only, no new protocol/persistence/server work, and the query line reuses the existing editor model (`model.ts`/`motion.ts`), parsing (`suggest-request.ts`), and the whole downstream suggestion flow â€” so almost no new logic is written. What keeps it at a 4 rather than lower is that correctness depends on threading a small ephemeral state through several editor call sites (keydown routing, the hidden-textarea input flush, inline rendering, pill suppression, and the `[run]` click handler) and on keeping the query text out of the buffer's `EditorState` so save/dirty/sync exclude it for free.

## Summary

Today an in-editor persona request is typed as an ordinary buffer line whose first non-whitespace character is `>` (see `product/specs/editor-tab.md` â†’ "In-editor persona suggestions", implemented in `web/src/editor/suggest-request.ts` and `web/src/editor/useEditorSuggest.ts`). Because the request is real buffer text, it saves to disk with the file and is part of the text buffer, and the whole flow is built to keep a Markdown blockquote from being mistaken for a request.

This feature changes how a request is *entered*: typing `>` at the start of an otherwise-empty line opens an **ephemeral agent query line** rendered inline at that line's position, visually distinct from surrounding text and excluded from the text buffer and from every save. The user types the persona and prompt there (with the existing `>>` assistant shorthand and Tab persona-completion), and sends with Enter, Ctrl/Cmd+Enter, or the `[run]` pill. Everything downstream of sending â€” the per-persona ACP connection, the inline hunk preview in the buffer, the thumbs-up/down accept/decline, and the `[agent?]`/`[query?]`/`[run]`/`[runningâ€¦]`/`[no suggestion]` pill states â€” is unchanged; only the entry surface moves out of the buffer. Because the query text never enters `state.lines`, it is never saved, never synced to a monitor as buffer content, and never included in the buffer content sent to the persona.

## Design decisions

1. **The query line is an inline ephemeral row, not a dedicated bar.** When triggered, the query line renders inline at its anchor line's on-screen position as a visually distinct, non-buffer row â€” matching today's spatial model where the request sat at a buffer line â€” rather than a command bar pinned below the metadata row. Only one query line exists at a time (the backlog wording is "the agent query line", singular).

2. **The query text lives outside the *buffer's* text entirely.** The query line is held as its own separate single-line `EditorState` (its own `lines`/`cursor`, reusing the same model as the buffer, `web/src/editor/model.ts`) plus an anchor line index â€” a distinct state object from the buffer's `EditorState`, so the buffer's `lines` are never touched. Keeping it out of the *buffer's* `EditorState` is what makes every buffer-derived computation exclude it *for free*: `dirty` compares `toText(state) !== lastSaved` (`web/src/EditorTab.tsx:86`), `save` writes `toText(s)` (`web/src/EditorTab.tsx:60`), and the monitor draft sync sends `toText(state)` keyed on `state` (`web/src/editor/useEditorSync.ts:20-27`). Folding the query text into the buffer's `EditorState.lines` instead would break all three (Decision 4) â€” so it must not be a buffer line. No buffer line is consumed by it.

3. **`>` opens the query line only as the first character on an otherwise-empty line.** Pressing `>` when the caret is at column 0 of an empty line opens the query line anchored at that line instead of inserting a `>`. Typed anywhere else â€” mid-line, after other text, or when a selection would be replaced â€” `>` inserts literally, exactly as today. As a direct consequence, a Markdown blockquote whose `>` is the very first character of an empty line cannot be typed by pressing `>` first; the literal `>` must be entered when the line is not empty (e.g. type the blockquote body, then insert the leading `>`). This is the accepted trade-off of triggering on the empty-line case rather than adding a separate escape hatch.

4. **Opening or editing the query line never dirties the buffer.** Because the anchor buffer line stays empty and the query text is not buffer content, opening the query line, typing in it, and closing it leave the buffer clean, do not enable the save button, and do not trigger live-draft sync.

5. **Escape closes the query line and inserts nothing.** Pressing Escape while the query line is active â€” whether empty or partly typed â€” closes it and returns keyboard focus to the buffer at the anchor line, with no character inserted and the buffer unchanged. Escape is the only dismiss affordance.

6. **This replaces the in-buffer request entry; the downstream flow is untouched.** The old behavior where a `>` request is an ordinary buffer line removed only if a hunk is accepted (`useEditorSuggest.ts`'s `resolveHunk` splice of `requestLineText`) is removed â€” there is no buffer line to splice. Everything after *sending* stays as specified today: the persona ACP connection opened on first request and reused for the tab's life, the inline hunk preview at each hunk's own buffer position, per-hunk thumbs accept/decline, buffer-edit suppression while any hunk is pending, and the pill states. Only the entry surface changes.

7. **The query line mirrors today's request-line affordances.** It shows a `>` prompt marker with placeholder text `persona requestâ€¦` when empty. The `>>` shorthand still names the `assistant` persona; Tab still completes an available persona name while the caret is in the persona-name token; the status pill still shows `[agent?]` â†’ `[query?]` â†’ `[run]` and, while running/after a null reply, `[runningâ€¦]`/`[no suggestion]`; and Enter, Ctrl/Cmd+Enter, and clicking `[run]` all send when the query is a valid `> <persona> <prompt>`. Enter while the query is not yet runnable is a no-op â€” it never inserts a buffer newline, because keyboard focus is in the ephemeral query, not the buffer.

8. **After resolution, keep-or-close mirrors today.** Once every proposed hunk is resolved, the query line closes if at least one hunk was accepted; if all hunks were declined, or the persona proposed nothing (`[no suggestion]`), the query line stays open with its text intact so it can be edited and retried. Only one request may be in flight or awaiting resolution per tab, as today.

9. **The buffer caret is suppressed while the query line is active.** While the query line owns keyboard focus, the buffer's blinking caret is hidden and printable keys, backspace, Left/Right/Home/End, and Tab operate on the query text; Up/Down are no-ops within the single-line query. This matches the existing rule that the caret is only shown for the active editing surface.

## Proposed changes

- **Ephemeral query-line state â€” reuse the editor model, don't rebuild it.** The query line is a separate single-line `EditorState` plus an anchor line index. Do **not** write new insert/backspace/caret-motion transitions: reuse `web/src/editor/model.ts` (`insertText`, `deleteBackward`, `deleteForward`, `fromText`, `toText`) and `web/src/editor/motion.ts` (`moveCursor`, `moveLineEdge`) on that single-line state, and `suggest-request.ts` (`personaToken`, `completePersonaName`, `suggestPillLabel`, all of which take the line string `state.lines[0]`) for all `>`/persona parsing. The only genuinely new logic is the tiny bit with no home elsewhere: pairing the anchor line index with the query `EditorState`, opening it on an empty line, and closing it. Keep that with the query-line ownership in `useEditorSuggest.ts` (next bullet); only extract it into a small `web/src/editor/query-line.ts` if `useEditorSuggest.ts` would otherwise cross the 200-line cap.

- **Hold the query-line state alongside the suggest surface.** Extend `web/src/editor/useEditorSuggest.ts` (currently 117 lines â€” add a sibling hook/module if the additions would push it past the 200-line cap) to own the active query-line state and expose it and its mutators to `EditorTab`. Change `fireOnLine` â€” today `fireOnLine(state, lineIndex)` reading `state.lines[lineIndex]` (`web/src/editor/useEditorSuggest.ts:58-60`) â€” so it sends the query from the ephemeral query text (its parsed persona + prompt, priming `editorSuggest` with the live buffer via `toText(state)` â€” which no longer contains the request text) rather than from a buffer line. Its two callers must follow the new signature: `handleSuggestKeyDown.ts:23,29` and `handleSuggestPillClick.ts:17`. Remove the request-line splice from `resolveHunk` (`web/src/editor/useEditorSuggest.ts:83-97`, the `state.lines.indexOf(p.requestLineText)`/`splice` block): on finalize, close the query line when `acceptedAny`, otherwise leave it open for retry (Decision 8). The `PendingSuggest.requestLineText` field is dropped since there is no buffer line to locate; `acceptedAny` is retained to drive keep-or-close.

- **Route keys to the query line.** Rework `web/src/editor/handleSuggestKeyDown.ts` (or add a companion handler it delegates to), called first in `EditorTab.tsx`'s `onKeyDown` at `:168` â€” before `actionForKey` at `:169` â€” so that, ahead of the normal key pipeline: (a) the pending-hunk block stays first; (b) when no query line is active, `>` as the first character of an empty line opens the query line instead of inserting (note that `>` reaches `onKeyDown` as an `insert` action because `plainAction` returns `{kind:'insert', text}` for any length-1 key, `web/src/editor/keys.ts:83`, so returning `true` here suppresses the buffer insert); (c) when a query line is active, printable keys, Backspace, and Left/Right/Home/End edit the query text, Tab completes the persona name via `completePersonaName` or focuses the runnable pill, Enter / Ctrl/Cmd+Enter / pill-Enter send when runnable (else no-op), and Escape closes it. This keeps `EditorTab.tsx`'s `onKeyDown` thin, consistent with today's split.

- **Route the hidden-textarea input path too.** Typed characters normally reach `onKeyDown`, but paste (Cmd/Ctrl+V) and IME composition bypass it and flow through the hidden `<textarea>`'s `onInput` â†’ `flushTextarea` â†’ `api.insert` (`web/src/EditorTab.tsx:176-180`, textarea at `:199-207`). When the query line is active, `flushTextarea` must route the textarea value into the query text instead of `api.insert`-ing it into the buffer â€” otherwise a paste (or IME commit) while the query line is focused would corrupt the buffer. This is the one non-obvious input path; the keydown path (previous bullet) covers ordinary typing.

- **Render the inline query row.** Update `web/src/editor/EditorLines.tsx` / `web/src/editor/render.tsx` to render the active query line inline at its anchor position as a visually distinct row: the `>` prompt marker, the query text (with the `persona requestâ€¦` placeholder when empty), the query caret, and the trailing status pill from `suggestPillLabel`. Today the pill is computed per buffer line from `suggestPillLabel(text, â€¦)` (`web/src/editor/EditorLines.tsx:52`) and the caret from `caretCol={onCursorLine && active ? state.cursor.col : -1}` (`:69`); for the query row the pill is computed from the ephemeral query text and the caret rendered within the row, replacing the empty anchor buffer line's normal render. The pill-while-pending suppression currently keyed on `pending?.requestLineText` (`web/src/editor/EditorLines.tsx:56`, passed to `suggestPillLabel`'s `pendingLine` param) must instead hide the query row's pill whenever that query line's own request is pending. The buffer caret is suppressed there (Decision 9).

- **Wire it in `EditorTab.tsx`.** Connect the query-line state, key handler, and rendering; ensure save/dirty (`web/src/dirtyEditors.ts`, `web/src/EditorSaveButton.tsx`) and live-draft sync (`web/src/editor/useEditorSync.ts`) see only buffer content, never the query text (this should follow for free from Decision 2, but assert it in tests).

- **Spec update (for the implementer, not this task).** Rewrite the `product/specs/editor-tab.md` â†’ "In-editor persona suggestions" section so the request is entered in an ephemeral, non-buffer, inline query line triggered by `>` on an empty line â€” not saved, not part of the buffer, and not synced to a monitor â€” while documenting that the connection, preview, accept/decline, and pill semantics are otherwise unchanged, and noting the empty-line trigger's effect on a leading-`>` blockquote (Decision 3).

The single-line editing mechanics themselves (insert/backspace/caret motion) need no new tests â€” they are the already-covered `model.ts`/`motion.ts` transitions; only assert the query state is wired to them.
- **`web/src/editor/handleSuggestKeyDown.test.ts` (extend):** `>` on an empty line opens the query line rather than inserting; while active, printable/backspace/arrow keys route to the query text and not the buffer; Enter/Ctrl/Cmd+Enter send only when runnable and never insert a buffer newline; Escape closes; the pending-hunk block still short-circuits first.

- **`web/src/editor/render.test.tsx` (extend):** the active query row renders inline at its anchor with the `>` prompt, placeholder, caret, and pill; the buffer caret is suppressed while it is active.
- **`web/src/EditorTab.test.tsx` (extend):** opening/typing/closing the query line leaves the buffer clean (save button stays disabled) and does not trigger live-draft sync (`toText`/save output never contains the query text); a paste/IME commit through `flushTextarea` while the query line is active routes into the query text and not the buffer; clicking the `[run]` pill fires from the query text (`handleSuggestPillClick`).
- **`web/src/dirtyEditors.test.ts` (extend):** the dirty-tracking that drives the save button is unaffected by an open/edited query line.

## Out of scope

- **Coexistence with the old in-buffer `>` request** â€” the buffer-line entry is replaced entirely, not kept as an alternative.
- **Redesigning the downstream preview/approval flow** â€” the persona connection, inline hunk preview, thumbs accept/decline, and pill states are untouched.
- **A dedicated escape hatch for a literal leading `>` on an empty line** (the "Escape re-inserts a literal `>`" alternative was not chosen); the literal `>` is entered by typing it when the line is not empty.
- **A dedicated command-bar placement** below the metadata row â€” the query line is an inline row, not a pinned bar.
- **Multi-line queries** â€” the query line is single-line, like the previous single request line.
- **Persisting or restoring the query line** â€” nothing about an in-editor suggestion is saved or restored on `--relaunch` today, and the query line follows that.
