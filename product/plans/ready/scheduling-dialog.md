# Scheduling dialog

**Complexity: 4/10** — spans server and web with a new protocol view, close RPC, and bare-keyword trigger, but every piece copies the existing harness-launch-dialog wiring end to end and submits through the unchanged `command` path, so there is no new server-side parsing, persistence, or firing logic.

A web modal for composing a `schedule` command through a form, opened by typing bare `schedule` — exactly mirroring how bare `harness` opens the "New harness" launch dialog today. Writing a schedule by hand means remembering the grammar (`schedule NAME [in TAB] <at TIME | on DATE [at TIME] | every N(m|h|d|w) | every DAY at TIME> COMMAND`); the dialog turns that into pick-a-type-and-fill-in-fields, then assembles the equivalent command string and submits it through the normal `command` RPC so the server's existing parsing, validation, and firing run unchanged. The dialog covers creation only — the five schedule forms, a target-tab selector, and the command to run — while managing existing entries stays on the `schedule list/cancel/clear` commands and the existing schedule window.

## Design decisions

- **Trigger.** Bare `schedule` (the word alone, no arguments) opens the dialog, replacing today's behavior of returning the `Usage:` message. This mirrors the bare-`harness` trigger in `CommandManager.run` (`src/command-manager.ts:81`). Every other `schedule …` form — including `schedule list`, `schedule cancel`, `schedule clear`, and any full creation form — continues to run exactly as it does now.
- **Platform.** Web app only, like `HarnessLaunchDialog`. No terminal/TUI dialog is added; the TUI keeps using the typed `schedule` command.
- **Schedule forms.** The dialog exposes a schedule-type selector covering all five forms: `at <time>` (one-shot today/next day), `on <date> [at <time>]` (one-shot date), `every N(m|h|d|w)` (recurring interval), `every day at <time>` (recurring clock time), and `every <weekday> at <time>` (recurring weekday). The selected type determines which time/date/interval inputs are shown.
- **Target tab.** The dialog offers a target-tab selector defaulting to the current active tab. Its options are the eligible tabs only — agent tabs and harness tabs — matching the scheduling spec's rule that image/page/markdown views cannot run scheduled commands. Choosing a non-active tab assembles the `schedule NAME in TAB …` form; choosing the active tab omits the `in` clause.
- **Fields.** Timer name (required), target tab (default active), schedule type + its time/date/interval inputs, and the command to run (required).
- **Validation.** Mirror the harness dialog: the Schedule button is disabled until the required fields are present and structurally valid for the chosen type (e.g. a time is required for `at`/`every day at`, an interval for `every N`). Because submission goes through the normal `command` path, server-side rejections — duplicate name within the target tab, an unparseable value — surface as a transcript line in the target tab exactly as they do for a typed command; the dialog does not re-implement those checks.
- **Wording.** Title "New schedule"; submit button "Schedule"; cancel button "Cancel". Escape cancels, Enter submits (as in `HarnessLaunchDialog` via `useDialogKeyboard`).
- **Command assembly.** A pure helper assembles the `schedule …` string from the field values and is unit-tested in isolation, mirroring `buildHarnessLaunchCommand`. Values are inserted verbatim (not shell-quoted), matching the harness helper's contract, since the server re-parses by whitespace-splitting.

## Proposed changes

**Protocol (`src/protocol.ts`).** Add a `ScheduleLaunchView` type carrying what the dialog needs that the client cannot already derive: the list of eligible target-tab labels (agent + harness tabs) and the default (active) label. Add a nullable `scheduleLaunch` field to the state view alongside `harnessLaunch`, and a `closeScheduleLaunch` RPC to the `RpcCall` union, both documented in the same style as their harness counterparts. There is one protocol file; the web app imports it via `@shared/protocol`.

**Server — schedule dialog state.** Put the dialog-open flag and view getter on `ScheduleManager` (`src/schedule/manager.ts`), mirroring how `HarnessManager` owns the harness dialog state — the schedule domain manager is the parallel home, and it already holds `private managers: Managers` (`src/schedule/manager.ts:14`) so it can reach the tab manager to build the view. Add `openScheduleLaunch()`, `closeScheduleLaunch()`, and `scheduleLaunchView()` methods copying the shape of `HarnessManager.openLaunchDialog`/`closeLaunchDialog`/`harnessLaunchView` (`src/harness/manager.ts:71`/`:77`/`:85`), each emitting `messageBus.emit('state', { type: 'dirty' })` on open/close as the harness ones do. `scheduleLaunchView()` returns `null` when closed and otherwise builds the eligible-label list plus the active label. Eligibility must use the exact predicate `resolveTargetTab` uses (`src/commands/schedule.ts:12` — `tab.view !== undefined && tab.view !== 'agent' && tab.view !== 'harness'` is rejected), i.e. include only tabs whose `view` is undefined, `'agent'`, or `'harness'`, so the selector can never offer a tab the server would reject. Wire a `scheduleLaunchView()` getter and a `closeScheduleLaunch()` delegate through `src/controller.ts` (beside `harnessLaunchView()`/`closeHarnessLaunch()` at `src/controller.ts:88`/`:92`), add the `scheduleLaunch` field to the emitted state in `src/state-event.ts:14` (beside `harnessLaunch: controller.harnessLaunchView()`), and add a `case 'closeScheduleLaunch'` to `src/message-handler.ts` (beside `case 'closeHarnessLaunch'` at `src/message-handler.ts:31`). Keep `src/schedule/manager.ts` (currently ~102 lines) under the 200-line limit — the three small methods fit, but if the view-building logic is non-trivial, extract it into a helper module beside the manager rather than inlining.

**Server — trigger.** In `CommandManager.run` (`src/command-manager.ts:77`), add a branch at the top — parallel to the bare-`harness` branch at `src/command-manager.ts:81` (`if (input.trim().toLowerCase() === 'harness') { this.managers.harness.openLaunchDialog(); return; }`) — that calls `this.managers.schedule.openScheduleLaunch()` and returns when the input trimmed and lowercased is exactly `schedule`, recording no transcript line. It must sit before the `resolveCommand` dispatch (`src/command-manager.ts:93`) so bare `schedule` never reaches the `schedule` command. All other `schedule …` inputs fall through unchanged; the `schedule` command's matcher (`command.match('schedule')`, asserted at `src/commands/schedule.test.ts:14`) is untouched and still matches — the interceptor only prevents the *bare* form from dispatching.

**Web — dialog component.** Add `web/src/ScheduleDialog.tsx` (a new module, keeping the file under the 200-line limit), structured like `HarnessLaunchDialog`: a modal reading a `ScheduleLaunchView`, holding form fields in state (with module-level remembered values across reopen), Enter/Escape via `useDialogKeyboard`, and Schedule/Cancel actions. On submit it sends `command` with the assembled string then `closeScheduleLaunch`; Cancel/Escape sends `closeScheduleLaunch`. The schedule-type selector switches which inputs render (time, date, interval, weekday), and the Schedule button is disabled until the chosen type's required inputs plus the name and command are present.

**Web — command builder.** Add `web/src/schedule-command.ts` exporting the field type and a `buildScheduleCommand` function that assembles `schedule NAME [in TAB] <form> COMMAND` from the fields, inserting values verbatim (matching `buildHarnessLaunchCommand`'s no-quoting contract at `web/src/harness-launch-command.ts:20`) and omitting the `in TAB` clause when the target equals the active label. The `<form>` string per schedule type must exactly match the grammar `parseScheduleCommand` accepts (`src/schedule/index.ts`, `SCHEDULE_USAGE` at `src/schedule/index.ts:17`): `at <time>`; `on <date>` with an optional trailing `at <time>`; `every <N><m|h|d|w>` (interval); `every day at <time>`; and `every <weekday> at <time>` (weekday one of `monday`…`sunday`). Times accept `3:35pm`/`2pm`/`14:00` and dates `august 12th`/`aug 12`/`8/12`, per `product/specs/scheduling.md` — the dialog's inputs should emit those spellings.

**Web — plumbing.** Extend the state listener chain (`web/src/ws.ts`, `web/src/useServerState.ts`) to carry `scheduleLaunch` alongside `harnessLaunch`, and render `{scheduleLaunch && <ScheduleDialog view={scheduleLaunch} client={client} />}` in `web/src/App.tsx` next to the existing harness-dialog render.

**Spec.** Update `product/specs/scheduling.md` to document the dialog: that bare `schedule` opens a web "New schedule" dialog (noting this replaces the former bare-`schedule` Usage response), the fields and five forms it covers, the active-tab-default target selector, that it submits through the normal command path (so server rejections surface in the target tab), and that it is creation-only.

## Tests

- `web/src/schedule-command.test.ts` (new): the builder produces the correct string for each of the five forms, with and without an `in TAB` clause, mirroring `harness-launch-command`'s coverage.
- `web/src/ScheduleDialog.test.tsx` (new): the type selector shows the right inputs per form; Schedule is disabled until required fields are valid; submitting sends `command` with the assembled string then `closeScheduleLaunch`; Cancel/Escape sends `closeScheduleLaunch`; remembered fields persist across reopen — modeled on `web/src/HarnessLaunchDialog.test.tsx`.
- Server: a test that bare `schedule` opens the dialog (calls `openScheduleLaunch`) while `schedule list` and a full creation form still dispatch to the `schedule` command, mirroring the `CommandManager bare-harness launch dialog` suite at `src/command-manager.test.ts:116` (which stubs `managers.harness.openLaunchDialog` — do the same for `managers.schedule.openScheduleLaunch`); and a `ScheduleManager` test that `scheduleLaunchView()` lists only tabs whose `view` is undefined/`'agent'`/`'harness'` with the active tab as default and returns `null` when closed. No existing integration test asserts the old bare-`schedule` Usage output, so none needs updating.

## Out of scope

- Managing existing schedules from the dialog — no list, cancel, or clear, and no editing of an existing entry. Those stay on the `schedule list/cancel/clear` commands and the existing schedule window.
- A terminal/TUI version of the dialog; it is web-only.
- Any change to the schedule grammar, firing behavior, or the schedule window/StatusPanels rendering.
- Re-validating schedule input client-side beyond the structural enable/disable gate; duplicate-name and parse errors are left to the server's existing responses.
