# Background the server and log its output

**Complexity: 5/10** — no new protocol, persistence, or UI, but it reworks process lifecycle across the launcher and server with real edge-case reasoning: foreground vs. background dispatch, readiness/timeout/early-exit detection, exit-code propagation, a new `stop` subcommand, and preserving an existing direct `__JANUS_URL__` consumer.

Launching `janus` should detach the HTTP server into the background and send all of its output to a log file rather than the user's terminal, so the terminal returns to the shell prompt immediately after a successful start. Today the launcher (`bin/janus.mjs`) runs the server attached with `stdio: 'inherit'` and stays in the foreground until Ctrl+C, and `src/main.ts` prints a machine-readable `__JANUS_URL__` line plus a human "Janissary is running at …" banner to the terminal. After this change, on a successful start nothing is printed and the shell prompt returns; the running server and the app window carry the session. Because Ctrl+C on the launcher no longer stops a detached server, this feature also adds a `janus stop` command that signals the running instance to shut down gracefully. Failed-to-start errors remain visible: they are written to both the log and the terminal, and the launcher exits with the server's own exit code.

## Design decisions

- **Backgrounding lives in the launcher; the server stays as-is.** `bin/janus.mjs` opens the log file and spawns the server entry (`dist/main.js` or `src/main.ts` via tsx, exactly as it selects today at `bin/janus.mjs:15`–`:24`) as a **detached** child with its `stdout` and `stderr` pointed at the log file descriptor, then `unref()`s it and returns to the shell. Because the child's fds point at a real file (not a pipe to the launcher), the server keeps writing after the launcher exits. `src/main.ts` needs no change to its output calls: its existing `__JANUS_URL__` line (`src/main.ts:172`), human banner (`src/main.ts:173`), warnings, and fatal banner (`src/main.ts:193`) all land in the log automatically because the launcher redirected the child's fds.
- **Launcher dispatch: foreground vs. background.** The launcher inspects its own `argv` (a plain, flag-aware scan — no dependency on the TypeScript `parseCliArgs`) and runs **attached** (`stdio: 'inherit'`, synchronous `spawnSync`, exiting with the child's code — i.e. today's behavior) when the invocation is a print-and-exit form: `stop` as the first argument, or `--help`/`--version` anywhere in `argv`. Every other invocation is a real server boot and runs **backgrounded** with the log redirection above. This keeps help/version/stop output on the terminal where the user needs it.
- **Silent on success.** On a successful background start the launcher prints nothing to the terminal and exits 0, leaving the user at their shell prompt. The `__JANUS_URL__` line and banner still go to the log.
- **Readiness, failure, and timeout detection.** After spawning the detached child, the launcher watches for one of three outcomes, mirroring the proven pattern in `scripts/docs-screenshots/janus.mjs:24`–`:40` (marker scan + `exit` handler + timeout): (1) the log file gains a line beginning with the `__JANUS_URL__ ` marker → success; the launcher `unref()`s the child, prints nothing (or the URL under `--no-open`), and exits 0. Because the fds point at the log file rather than a pipe the launcher can read, the launcher detects the marker by tailing/polling the log file it just opened (re-reading its growing contents) until the marker appears. (2) The child process exits before the marker appears → failure; the launcher reads the tail of the log (which holds the failed-to-start banner or a usage error) and writes it to the terminal, then exits with the **child's own exit code** (preserving 1 for startup failures and 2 for CLI usage errors). (3) Neither happens within a bounded timeout (reuse a constant like `docs-screenshots`'s `URL_TIMEOUT_MS = 20_000`) → the launcher kills the detached child, prints a `failed to start: timed out waiting for the server` banner to the terminal, and exits 1.
- **Log location and lifecycle.** Server output is written to `.janissary/log/server.log` (a single file inside the existing log directory that already holds transcript logs, created at `src/transcript/logger.ts:25`–`:26`). The launcher creates `.janissary/log` (recursive) before opening the file. On a normal launch the file is opened truncating (fresh), mirroring how state and workspace are cleared on a normal launch; when `--relaunch` is present in `argv` the file is opened in append mode so a resumed session keeps its prior server log.
- **Project-directory resolution in the launcher.** To know where `.janissary/log/server.log` lives, the launcher resolves the target directory the same way `src/main.ts:147` does (`args.projectDir ?? process.cwd()`) using a minimal flag-aware `argv` scan: skip `--help`/`--version`/`--relaunch`/`--no-open` (bare booleans) and the `--port` value in both `--port=<n>` and `--port <n>` forms; the first remaining non-flag token is the project directory, otherwise the current working directory. `--port` is the only value-taking flag today (`src/cli-args.ts:27`); a comment must flag that this scan needs updating if another value-taking flag is added.
- **Readiness marker is a shared literal.** The launcher matches the exact `__JANUS_URL__ ` marker string the server prints. It cannot import a constant across the `.mjs`/`.ts` boundary, so the marker is duplicated as a literal with a cross-reference comment — exactly as `scripts/docs-screenshots/janus.mjs:9` (`const URL_MARKER = '__JANUS_URL__ '`) already does. Do not attempt a shared constant module.
- **Failed-to-start is loud.** When the detached server fails to start (port already in use, missing web bundle, instance lock held, or any other startup error), the launcher surfaces the complete failed-to-start banner — the `<name> <version> — failed to start: <reason>` text from `formatFatal` in `src/startup-errors.ts:16` — to the terminal (read from the log tail) as well as leaving it in the log, and exits with the child's code. `JANUS_DEBUG=1` stack traces continue to work because they are part of that same server output now captured in the log; the terminal relay includes them.
- **`--no-open` still prints the URL.** Under `--no-open` no app window opens, so the user would otherwise have neither a window nor a URL. In that case, after detecting the readiness marker, the launcher prints the server URL (parsed from the marker line, as `docs-screenshots` does at `:32`) to the terminal; without `--no-open` it stays silent.
- **Warnings go to the log only.** Non-fatal warnings that today print to stderr — invalid `.janissary/config.json` (`src/config.ts:47`), invalid `agent-names.json` (`src/agent-names.ts:16`), invalid `harness-models.json` (`src/harness/models.ts:16`), and the chrome frame-enabler extension load failure (`src/chrome-extension-loader.ts:75`, `src/main.ts:127`) — need no change: they already go through `process.stderr.write`, which the launcher has redirected to `.janissary/log/server.log`. They are not failed-to-start errors and do not reach the terminal.
- **`janus stop` command.** `janus stop [<project-dir>]` reads the `.janissary/lock` PID for the target directory (current directory by default) and, if that PID is alive, sends `SIGTERM` to trigger the existing graceful-shutdown path (`src/main.ts:176`–`:180`). When no live instance is running for that directory (no lock file, or the PID is dead), it prints `no running janus instance for <dir>` to the terminal and exits 0 (nothing to stop is not an error). `stop` runs in the foreground (see dispatch above) so its message reaches the terminal.

## Proposed changes

**Launcher backgrounding and dispatch (`bin/janus.mjs`).** Replace the single synchronous, attached `spawnSync(..., { stdio: 'inherit' })` (`bin/janus.mjs:26`) with a dispatch: (a) a foreground branch — for `stop` as the first arg or `--help`/`--version` present — that keeps today's `spawnSync` inherit behavior and `process.exit(result.status ?? 1)`; and (b) a background branch for a normal boot. The background branch resolves the project directory via the flag-aware scan above, ensures `.janissary/log` exists, opens `.janissary/log/server.log` (truncate normally, append when `--relaunch` is present), spawns the server entry `detached` with `stdio: ['ignore', logFd, logFd]`, then waits for the readiness marker / early exit / timeout as described, and finally `unref()`s the child and returns control to the shell. On success it prints nothing (or the URL under `--no-open`); on early exit it relays the log tail and exits with the child's code; on timeout it kills the child and exits 1. Keep the marker string and timeout as local literals with a comment cross-referencing `scripts/docs-screenshots/janus.mjs`.

**Server output (`src/main.ts`).** No change to the boot sequence or its `process.stdout`/`process.stderr` writes is required — redirection happens at the launcher's fd level. Confirm during implementation that the `__JANUS_URL__` line remains the first thing written on success (`src/main.ts:172`) so the launcher's marker scan fires promptly, and that the fatal banner path (`src/main.ts:191`–`:193`) still writes to `process.stderr` (now the log) before `process.exit`. If `boot()` needs to dispatch the `stop` action (see below), that is the only addition here.

**`janus stop` subcommand (`src/cli-args.ts` + a stop handler).** In `parseCliArgs` (`src/cli-args.ts:16`), recognize `stop` as the first positional before the existing project-dir handling (`src/cli-args.ts:48`): when `positionals[0] === 'stop'`, return a parsed shape carrying a `stop` action and an optional following `<project-dir>` positional, so `janus stop /some/dir` is not rejected by the "unexpected argument" branch (`src/cli-args.ts:56`). Add a `stop` action field to the `CliArgs` interface (`src/cli-args.ts:7`). In `boot()` (`src/main.ts:133`), dispatch on that action before acquiring the lock: resolve the target directory, read `.janissary/lock` via the new `readLockPid` helper, check the PID is alive with the same logic as `isPidAlive` (`src/instance-lock.ts:4`), and if alive send `SIGTERM`; otherwise write `no running janus instance for <dir>` and return (exit 0). Export `readLockPid(projectDir): number | undefined` from `src/instance-lock.ts` so `stop` does not duplicate `lockPath`/parse logic. Put the `stop` handler in its own small `src/stop-instance.ts` module that `boot()` calls rather than inlining it in `src/main.ts` — `main.ts` is already ~195 total lines and near-full of boot logic, so a dedicated module keeps it cohesive and comfortably under the `max-lines` (blanks/comments-excluded) limit.

**Usage text (`src/cli-args.ts:70`).** Extend `usageText()` to document the `stop` subcommand (e.g. a line `janus stop [<project-dir>]  Stop the running instance for a directory`).

**Spec updates.** Update `product/specs/cli.md`: the startup sequence (its step list at `product/specs/cli.md:44`–`:55`) now ends by detaching the server and returning the shell prompt silently on success (URL printed only under `--no-open`); server output — including the URL banner and non-fatal warnings — goes to `.janissary/log/server.log` (truncated on a normal launch, appended on `--relaunch`); failed-to-start errors go to both the log and the terminal, and the launcher exits with the server's code. Document the new `janus stop [<project-dir>]` command (add it to the flags/arguments table and the shutdown section at `product/specs/cli.md:58`) and its no-instance behavior. If `product/specs/state-directory.md` enumerates log contents, note `server.log` there.

## Tests

- `src/cli-args.test.ts` (extend the existing file): `stop` is recognized as a subcommand producing the `stop` action with an optional project-dir; `janus stop /dir` does not raise the "unexpected argument" usage error; `usageText()` mentions `stop`; a normal invocation still parses project-dir/flags unchanged.
- `src/instance-lock.test.ts` (extend the existing file, which already builds temp `.janissary/lock` files at `:33`–`:38`): `readLockPid` returns the PID for a live lock, and `undefined`/a dead PID when the lock is absent or stale — mirroring the existing live-vs-dead-PID fixtures.
- Server: a unit test for the `stop` handler module (`src/stop-instance.ts`) — with a lock file naming a live PID it signals shutdown (spy on `process.kill`); with no lock file or a dead PID it prints the `no running janus instance for <dir>` message and returns without signalling. Model process/PID stubbing on `src/instance-lock.test.ts`.
- Launcher (`bin/janus.mjs`) is a thin process-orchestration shim not currently unit-tested; if the log-target selection (truncate vs. append, project-dir scan) and the marker/early-exit/timeout logic are extracted into an importable helper, cover those with a focused test. Do not add a CI test that actually spawns and detaches a real server. `scripts/docs-screenshots/janus.mjs` remains the working reference for the runtime behavior.
- Confirm no existing test asserts the old always-printed success banner on stdout/stderr; `scripts/docs-screenshots/janus.mjs` spawns the server entry **directly** (not via the launcher) and reads `__JANUS_URL__` from the child's stdout — this keeps working unchanged and must not be broken, since the direct-spawn path still writes the marker to its inherited stdout.

## Out of scope

- Any log rotation, size capping, or pruning of `.janissary/log/server.log` — it is truncated on a normal launch and appended on `--relaunch`, nothing more.
- A structured/JSON server log format — `server.log` captures the server's existing human-readable stdout/stderr verbatim.
- Changing what the in-app `quit` command (`src/commands.ts:12`) or the close-last-window shutdown does; `janus stop` reuses the same graceful-shutdown path, it does not replace those triggers.
- A cross-directory "stop all instances" command — `janus stop` targets exactly one directory's instance.
- Changing `scripts/docs-screenshots/janus.mjs`, which deliberately bypasses the launcher and is unaffected.
- Windows-specific process-group/detach nuances beyond what the existing detached-spawn code (`openApp`/`openUrl` at `src/main.ts:52`,`:110`–`:121`) already handles.
