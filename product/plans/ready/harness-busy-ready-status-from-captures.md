# Recognize harness busy/ready status from captures

**Complexity: 5/10** — three small per-harness detectors following an exact existing precedent, but correctness hinges on three things beyond the detectors themselves: fixing a real single-callback-slot constraint in `HarnessScreenReader`'s wiring (decision 5), threading the terminal's OSC title through `ScreenCapture` so the title-based signals are available (decision 8), and a small debounce so a mid-generation pause doesn't flicker the dot to "ready" (decision 7).

## Summary

Today a harness tab's dot blinks (`tab.busy`) for the entire lifetime of the process — `addBusy` runs once at spawn and is never cleared until the harness exits, so the blinking dot doesn't actually mean "thinking right now." This feature makes the blinking dot accurate for **all three harnesses (claude, codex, opencode)** by recognizing, from the same captures auto-approve already reads (`HarnessScreenReader`/`ScreenCapture`), whether the harness is actively generating ("busy") or sitting idle at its own prompt ("ready"), and calling `addBusy`/`deleteBusy` accordingly.

The detection strategy follows the approach proven by [herdr](https://github.com/ogulcancelik/herdr) (`src/detect/manifests/*.toml`), which recognizes these states from terminal captures. The harnesses fall into **two signal families**, which is exactly why the detector is a per-harness table (mirroring the existing permission-gate table's shape) rather than one shared heuristic:

- **claude and codex — OSC-title-driven.** Both write an animated Braille spinner glyph into the terminal title while generating, and a non-spinner title when idle. A single rule covers both: **a leading Braille glyph (`U+2800`–`U+28FF`) in the title means "busy"; any other title means "ready."** (claude's idle title leads with `✳` and codex's with the working directory name — neither collides with the Braille busy range.) claude additionally has a rendered prompt-box fallback for when no title is present.
- **opencode — rendered-screen-text only.** No OSC-title state signal exists (its title is a static app/session name), so a progress bar (a run of `■`/`⬝` glyphs) or an interrupt-hint footer means "busy"; their absence means "ready."

A **busy→ready transition is briefly debounced** so a short pause in the middle of generation doesn't flash the dot to "ready." Every harness janissary launches now has a real detector; the table still degrades cleanly to today's coarse (spawn-to-exit) blinking for any future harness without an entry.

## Design decisions

1. **Detection signals fall into two families — claude/codex key off the OSC title, opencode off rendered text.** The `classify` function is per-harness (decision 2), because the harnesses expose different signals. All three were confirmed against the real CLIs via live PTY spot-checks (2026-07-15), and the final literals are authored from captures gathered during implementation the same way the existing permission-gate patterns in `auto-approve.ts` were — the plan fixes the *shape* of each signal, not brittle exact strings.
   - **claude and codex — OSC-title-driven (shared rule: leading Braille glyph = busy).** Both emit their activity into the terminal title (OSC 0, which sets both icon and window title). While generating, the title **leads with** an animated Braille spinner glyph in `U+2800`–`U+28FF`; when idle, the title leads with a non-Braille character, so the classifier keys on the **leading glyph only**: leading Braille → busy, anything else → ready.
     - *claude* (confirmed against 2.1.210): busy title cycled `⠂` (`U+2802`) / `⠐` (`U+2810`) followed by a prompt summary (`⠂ Write a haiku about the sea`); idle title led with `✳` (`U+2733`), e.g. `✳ Claude Code`.
     - *codex* (confirmed against codex-cli 0.144.4): busy title cycled the standard 10-frame spinner `⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏` (`U+280B 2819 2839 2838 283C 2834 2826 2827 2807 280F`) followed by the cwd basename (`⠹ scratchpad`); idle title was the bare cwd basename (`scratchpad`, leading with a letter). Codex has **no** positive idle marker like claude's `✳` — idle is simply "title present, not a spinner" — which the leading-Braille rule handles without special-casing.
     - Only **claude** also needs a rendered-screen fallback for when no title is present: a live input prompt box (a bare `❯` caret line) with **no** permission gate showing means "ready", otherwise "busy" (mirrors herdr's `live_prompt_box` idle rule; the spot-check also showed a discriminating footer — `esc to interrupt` while busy, `? for shortcuts` while idle). Codex reliably emitted a title in every observed state, so its fallback (if any) is a lower priority; author from captures during implementation.
   - **opencode — rendered screen text only (no OSC-title state signal).** Confirmed against opencode 1.17.18: opencode's OSC title is a static app/session name (`"OpenCode"`, later `"OC | <session>"`) and does **not** animate with activity, so the title approach is unavailable and detection must read the rendered frame. Busy is signalled by either (a) a **progress bar** — a run of block/dot glyphs, herdr's `(■|⬝){4,}` regex, observed live as `⬝⬝⬝⬝■■■■` — or (b) an **interrupt-hint footer**. The progress bar is the robust primary signal; the interrupt hint is secondary because its literal drifts across versions (herdr's manifest matches `"esc to interrupt"`, but opencode 1.17.18 renders `esc interrupt` with no "to" — so author this literal from a live capture, don't copy herdr's). "Ready" is the **absence** of both — opencode has no positive idle marker (herdr models idle as the known-agent fallback when no working/blocked rule fires); the idle frame observed was the input box with a `tab agents  ctrl+p commands` / token-usage footer.
2. **Harness scope.** The detector is a per-harness table from day one (same shape as `GATE_TABLE` in `auto-approve.ts`), so a harness with no entry falls back to today's coarse behavior (busy for the whole process lifetime). All three harnesses janissary launches — claude, codex, and opencode — get real detectors in this pass; the no-entry fallback remains for any harness added later.
3. **Interaction with the permission gate (claude-specific).** While claude is showing its permission-gate prompt, the tab's dot stops blinking (claude itself is idle, blocked on the user, not doing work) — but if the gate is not auto-approved, the tab's unread badge is set (via the existing `markUnread`) to call the user's attention to it, the same way an unread badge already flags other tab activity needing attention. The gate check uses the existing `detectPermissionGate` and takes precedence over the busy/ready signals (a visible gate always means "not busy"), matching herdr, where the `blocked` rules outrank the working/idle rules. This badge behavior is claude-only because `GATE_TABLE`/auto-approve are claude-only today: for codex and opencode, `detectPermissionGate` returns `false`, so a permission prompt in those harnesses simply classifies as "ready" (not busy — no spinner in the codex title, no progress bar in opencode) with no badge — the dot correctly stops blinking, and dedicated codex/opencode gate + badge signals are deferred (see out of scope).
4. **Initial state.** A harness tab starts "busy" (blinking) immediately on spawn, exactly as today, until the first screen capture is read and classified — avoiding a flash of "ready" before the process has produced any output yet.
5. **`HarnessScreenReader` takes exactly one `onCapture` callback — busy/ready detection cannot simply be "a second consumer."** `HarnessScreenReader`'s constructor (`src/harness/screen.ts:29`) accepts a single optional `onCapture?: (capture: ScreenCapture) => void`. Today, `spawnTab` (`src/harness/manager.ts:114`, via `autoApproveHandler` at `:128-137`) passes `undefined` for this callback whenever `autoApprove` is false — meaning **no callback fires at all, for any purpose, on a non-auto-approving harness tab today**. Busy/ready tracking must work regardless of whether `-y`/auto-approve is enabled (per decision 3's gate interaction, which isn't conditioned on auto-approve). So `spawnTab` must build one **combined** callback instead of the current either/or: call the auto-approve handler's logic first (if `autoApprove` is true), then the busy/ready logic (if the harness has a real `BUSY_TABLE` entry) — constructing `undefined` only when *neither* applies, to avoid adding capture-handling overhead to a harness with no auto-approve and no busy detector. With all three current harnesses now carrying a busy detector, that `undefined` branch only fires for a future harness with neither; opencode/codex build a busy/ready callback even without `-y`.
6. **Detecting "not auto-approved" for the unread badge requires a small, new public accessor.** `HarnessAutoApprover` tracks whether it's stuck on a repeated, un-clearable gate via a *private* `stuckNotified` field (`src/harness/auto-approve.ts:57`) — there is currently no way for anything outside the class to read this. The busy/ready handler (decision 3) needs it to decide whether to call `markUnread`: mark unread when either `autoApprove` is off entirely, or the approver for this harness reports it's stuck. `HarnessAutoApprover` gains a small public getter (e.g. `get isStuck(): boolean { return this.stuckNotified; }`) for this purpose — no other change to the class. Within the combined callback (decision 5), the auto-approve handler's `onCapture` (which updates `stuckNotified` synchronously) must run **before** the busy/ready handler reads `isStuck` for the same capture, so the just-updated stuck state is visible in the same cycle.
7. **A busy→ready transition is debounced; ready→busy is immediate.** A harness briefly stops producing output mid-turn (e.g. between a tool result and the next model tokens; for opencode, the progress bar can blank for a frame between steps). A naive "no busy signal right now → ready" would flicker the dot off and back on. Following herdr's `PendingIdleConfirmation` (`src/pane/agent_detection.rs:39`), a transition from busy to ready is only committed after the ready signal persists across a short confirmation window; a transition to busy (or a detected gate, or process exit) is applied immediately. Because captures are already debounced ~1s by `HarnessScreenReader` (`CAPTURE_DELAY_MS`), the confirmation is expressed in captures/elapsed time rather than a new fast timer — the simplest form that removes the flicker (e.g. require the ready classification to hold for two consecutive captures, or for a small elapsed interval, before clearing busy). Exact form is settled during implementation against real capture cadence; the requirement is "one transient ready frame does not clear the dot."
8. **The OSC title must be threaded through `ScreenCapture`.** `ScreenCapture` is `{ text, capturedAt }` today (`src/harness/screen.ts:12`) — rendered buffer text only; the OSC title is not captured. xterm headless surfaces the title via the `onTitleChange` event, not the buffer, so `HarnessScreenReader` subscribes to `this.term.onTitleChange`, retains the latest title, and includes it on each `ScreenCapture` (new optional `title` field). This is purely additive: `HarnessAutoApprover` ignores the new field and is unaffected. Keeping the title as data on the capture (rather than a side channel) means the busy/ready detector is still a pure `(capture) => state` function, testable from fixtures.

## Reference: how herdr detects these states

herdr recognizes `idle` / `working` / `blocked` states from terminal captures using per-agent rule manifests (`src/detect/manifests/*.toml`, `src/detect/manifest.rs`). Each manifest is a priority-ordered rule list; a harness with no matching rule falls back to `idle` (`DEFAULT_KNOWN_AGENT_IDLE_FALLBACK`). The signals this plan reuses:

**claude** (`claude.toml`) — the richer, OSC-title-driven case:

- **Working (busy)** — highest-priority rule `osc_title_working` matches a Braille spinner glyph (`^[\x{2800}-\x{28FF}] `) at the start of the **OSC title**. Verified against claude 2.1.210 (`⠂`/`⠐` observed).
- **Idle (ready)** — `osc_title_idle` matches `^\x{2733} ` (the `✳` marker) in the OSC title; `live_prompt_box` matches a `❯` prompt-caret line in the rendered input box that is **not** any permission prompt. Both verified against claude 2.1.210.
- **Blocked** — permission/selection forms ("do you want to proceed?", numbered `1. Yes` / `2. No`) outrank the working/idle rules — the same structure janissary's `detectClaudeGate` already recognizes.

**codex** (`codex.toml`) — also OSC-title-driven (newest manifest, engine v2), so janissary reuses the same title machinery as claude:

- **Working (busy)** — `osc_title_working` matches the standard 10-frame Braille spinner `[⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏]` anywhere in the **OSC title** (herdr uses word boundaries). Verified against codex-cli 0.144.4: all 10 frames observed leading the title, followed by the cwd basename (`⠹ scratchpad`).
- **Idle (ready)** — `osc_title_idle` matches any non-whitespace title that is **neither** the spinner **nor** `"Action Required"` — i.e. there is no positive idle glyph; idle is "title present, not busy/blocked." Verified: the idle title was the bare cwd basename (`scratchpad`). janissary's shared "leading Braille glyph → busy, else ready" rule captures this cleanly (the cwd basename leads with a non-Braille character).
- **Blocked** — `osc_title_blocked` matches `"Action Required"` in the title; rendered-screen backups (`live_strong_blocker`: "press enter to confirm or esc to cancel", "allow command?"; `weak_blocker`: `[y/n]`, "do you want to" + yes/❯). Not wired into janissary this pass (see decision 3 / out of scope).

**opencode** (`opencode.toml`) — a simpler, screen-text-only case (older manifest, engine v1, no OSC/`visible_idle` rules), which is why its janissary detector is shaped differently from the title-driven pair:

- **Working (busy)** — `progress_bar_working` matches `(■|⬝){4,}` (four-plus block/dot glyphs), and `interrupt_hint_working` matches an "esc … interrupt" footer. Verified against opencode 1.17.18: the progress bar rendered as `⬝⬝⬝⬝■■■■` (matches), but the footer rendered `esc interrupt` — herdr's `"esc to interrupt"` literal is **stale**, so the progress bar is the durable signal and any footer literal must come from a live capture.
- **Idle (ready)** — no positive rule; idle is the known-agent fallback when no working/blocked rule fires. The observed idle frame was the input box with a `tab agents  ctrl+p commands` / token-usage footer.
- **Blocked** — `"△ Permission required"` or a selection-menu structure. Not wired into janissary this pass (see decision 3 / out of scope).
- **No OSC-title signal** — confirmed live: opencode's title stayed a static `"OpenCode"` / session name throughout generation, never a spinner.

**Flicker control** — `PendingIdleConfirmation` (`src/pane/agent_detection.rs`) holds a working→idle transition until the idle signal is confirmed (up to a short cap), so a brief mid-turn pause doesn't publish "idle." Decision 7 applies the same idea to all three harnesses.

janissary deliberately keeps the simpler per-harness detector-function table (claude, codex, and opencode detectors — the first two sharing a title rule) rather than herdr's full data-driven, remotely-updatable manifest engine — that engine solves a fleet-of-20-agents problem janissary doesn't have yet.

## Proposed changes

1. **`src/harness/screen.ts`**: add an optional `title` to `ScreenCapture` and populate it. Subscribe to `this.term.onTitleChange` in the constructor, store the latest title, and set it on the `ScreenCapture` built in `captureNow` (decision 8). Existing `text`/`capturedAt` behavior is unchanged; the auto-approver ignores `title`.
2. **New per-harness busy/ready table** (e.g. `src/harness/busy-status.ts`, mirroring `auto-approve.ts`'s shape): a `BUSY_TABLE: Record<string, { classify: (capture: ScreenCapture) => 'busy' | 'ready' }>` (or equivalent), with a `classifyBusy(capture, harnessName): 'busy' | 'ready' | undefined` export that looks up the harness's entry and returns `undefined` for any harness without one (meaning "no real detector — keep existing coarse behavior"). Three entries this pass (decision 1):
   - **claude** — check the OSC title first (leading Braille glyph → busy, leading non-Braille such as `✳` → ready), then fall back to the rendered prompt box (live `❯` input box and no gate → ready, else busy).
   - **codex** — OSC title only: leading Braille spinner glyph → busy, any other non-empty title (e.g. the cwd basename) → ready. This is the **same leading-Braille title check as claude** — factor it into one shared helper the two entries call, rather than duplicating it. Codex needs no prompt-box fallback (it emitted a title in every observed state).
   - **opencode** — rendered `text` only: a progress-bar run `(■|⬝){4,}` **or** an interrupt-hint footer → busy; neither present → ready. (opencode's `classify` ignores `capture.title`, which is a static app/session name.)
   - Literals authored from real captures (decision 1) and kept obvious to update. Keeping each harness's signal in its own `classify` closure is why the table shape matters — even though claude and codex share the title helper, opencode is a genuinely different signal, so this is not one shared heuristic.
3. **`src/harness/auto-approve.ts`**: add a public `get isStuck(): boolean { return this.stuckNotified; }` to `HarnessAutoApprover` (decision 6) — no other change to the class.
4. **New busy/ready capture handler** (in the new `src/harness/busy-status.ts`): a factory taking `(name, label, managers, approver: HarnessAutoApprover | undefined)` and returning a `(capture: ScreenCapture) => void` that: looks up `BUSY_TABLE[name]`; if there's no entry, does nothing (preserving today's coarse behavior); otherwise checks `detectPermissionGate(capture.text, name)` first — if a gate is showing, calls `deleteBusy(label)` immediately and additionally calls `markUnread(label)` when `!autoApprove || approver?.isStuck` (decision 6); if no gate is showing, calls `classifyBusy(capture, name)` and applies it — `'busy'` calls `addBusy(label)` immediately, `'ready'` clears busy only after the debounce confirmation (decision 7) is satisfied. The small amount of transition state the debounce needs (e.g. pending-ready-since) is held in a closure created per tab by the factory.
5. **`src/harness/manager.ts`'s `spawnTab`**: replace the current `this.autoApproveHandler(name, label, id, autoApprove)` argument to `HarnessScreenReader`'s constructor (`:114`) with a combined callback per decision 5 — build the auto-approve callback (if `autoApprove`) and the new busy/ready callback (if `BUSY_TABLE[name]` exists), and pass a function that calls whichever of the two apply, auto-approve's first (so `isStuck` reflects this capture, per decision 6), or `undefined` if neither applies. `autoApproveHandler`'s existing internals (constructing `HarnessAutoApprover`, registering it in `this.autoApprovers`) are otherwise unchanged; the busy/ready callback needs a reference to that same approver instance (or `undefined`) to read `isStuck`.
6. **`product/specs/harness.md`**: add a section describing per-harness busy/ready recognition — the shared claude/codex OSC-title rule (leading Braille glyph = busy) with claude's prompt-box fallback, opencode's screen-text-only progress-bar / interrupt-hint signals, the busy→ready debounce, the claude-only interaction with the permission gate and the unread badge, and that any future harness with no detector keeps the existing coarse (spawn-to-exit) busy behavior until it gets its own.

## Out of scope

- Codex/opencode permission-gate + unread-badge signals — codex's blocked title (`"Action Required"`) and opencode's blocked frame (`△ Permission required`) are not wired into `GATE_TABLE`, so a permission prompt in those harnesses reads as "ready" (not busy) with no badge. Adding their gate detection + badge (and, separately, auto-approve for them) is deferred; the busy/ready dot for all three harnesses does not depend on it.
- A data-driven / remotely-updatable detection manifest like herdr's — janissary keeps hand-written per-harness detectors; a manifest engine is unwarranted at three harnesses.
- Any change to `HarnessView.status` (`'running'`/`'exited'`) or the process-exit lifecycle.
- Any change to the permission-gate detector or auto-approve keystroke logic itself.
- A visually distinct "gate pending" indicator beyond the existing unread badge (e.g. a third dot color or icon) — the badge is the only new signal for this case.
