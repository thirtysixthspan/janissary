# File navigator: color git-modified files yellow

**Complexity: 4/10** â€” one new small server module plus wiring into two existing call sites (`open`/`rebuild` in `file-tree-manager.ts`) and a client render tweak; no new protocol message and no persistence â€” but the git lookup is asynchronous and cached per tab (see decision 6), so correctness depends on catching both row-build call sites, threading a cached changed-set through the synchronous render path, and applying the async result without racing a reroot or tab close. The client render tweak is also constrained by a file at the 200-line ceiling.

## Summary

In the file tree tab (`files [path]`), any row that git considers changed â€” a working-tree modification, a staged change, or an untracked (new) file â€” renders its name in yellow, the same way a conventional editor's Explorer/SCM decoration highlights dirty files. A directory row also renders yellow whenever any file beneath it (at any depth, expanded or not) has a git change, so the signal is visible without expanding every folder. Rows in a directory that is not part of a git repository (or where git status cannot be determined) render with no special coloring and no error is shown.

## Design decisions

1. **Which git states count.** Modified (tracked, unstaged), staged, and untracked files are all treated identically â€” all render in the same yellow, with no visual distinction between the three states. This matches the literal ask ("indicate modified files as indicated by git") without introducing a broader status-coloring language than requested.
2. **Directory propagation.** A directory row is colored yellow if any descendant file anywhere under it has a git change, regardless of whether that descendant's parent directories are currently expanded. This lets a user spot changes in collapsed subtrees without walking the whole tree, mirroring how IDEs decorate folders.

   Note that `buildRows` (`src/file-tree.ts:35`) only descends into directories that are already in the `expanded` set â€” an unexpanded directory's row is present, but its contents are never read from disk, so this cannot be implemented by "walking into" collapsed directories. It works anyway because `git status` (run once per rebuild against the whole tree root, not per visible directory) already returns every changed path in the tree, regardless of expansion state â€” propagation is a prefix check (`changedPath.startsWith(rowPath + '/')`, or `changedPath === rowPath` for a file) against that one flat set, applied to whichever rows `buildRows` currently returns. No extra filesystem reads are needed to color a collapsed directory.
3. **Refresh trigger.** Git status is not polled on its own timer. It is recomputed every time the tree's rows are already being recomputed â€” on initial `open()`, and on every trigger of the existing `rebuild()` (the watcher's debounced refresh, `toggle`, `collapseAll`, `move`, `delete`, `undo`/`redo`, and `reroot`) â€” so there is exactly one signal that decides "the tree's displayed state is stale" and git status rides along with it rather than needing its own staleness tracking.
4. **Non-git roots.** If the tree's root is not inside a git repository, or the git status command fails for any reason, the tree renders exactly as it does today â€” no yellow anywhere, no error, no header notice. This keeps the feature purely additive for repos and inert everywhere else.

5. **Git invocation.** Use `node:child_process`'s **asynchronous** `execFile` (per decision 6, not the blocking `execFileSync`) with an argument array (`execFile('git', ['status', '--porcelain=v1', '--untracked-files=all', '-z', '--', '.'], { cwd: root })`), not a shell string â€” unlike the `execSync('git clone "${...}"', ...)` string-interpolation pattern already in `src/workspace.ts:74`, an argument array needs no shell-quoting and is safe against paths containing spaces or shell metacharacters. The trailing `-- .` pathspec scopes the scan to the tree's own root so that opening a tree deep inside a large monorepo does not pay a full-repository status cost (per decision 6). `--untracked-files=all` is required: without it, git collapses a wholly-untracked directory into a single `?? dir/` entry instead of listing its files individually, which would silently break decision 2's per-file propagation for new directories. `-z` NUL-terminates each entry as `XY path\0` and is required for filenames containing spaces or unusual characters to parse unambiguously; verified against a real repo, a rename/copy entry (status `R`/`C`) is two consecutive NUL-terminated fields â€” the new path first, then the old path â€” both of which count as one changed path each for this feature's purposes (no distinction between them is needed, since decision 1 already treats every git state identically).

6. **Performance: async, cached, and coalesced.** `git status` on a large or cold-cache repo can take from ~100 ms to several seconds, and `rebuild()` (`src/file-tree-manager.ts:247`) runs in the single Node process that also drives every harness PTY, every other tab, and all keystroke input. Running git **synchronously** on every rebuild would block that whole event loop, and `rebuild()` fires not only on the watcher's 100 ms-debounced refresh but on purely interactive, git-state-neutral actions (`toggle`, `collapseAll`, `move`, `delete`, `undo`/`redo`, `reroot`). Three mitigations, all required:
   - **Asynchronous.** The git lookup uses `execFile` (decision 5), never `execFileSync`. `rebuild()` stays synchronous and renders rows immediately from the tab's cached changed-set; the git result is applied later, when the async call resolves, via a second `messageBus.emit('state', { type: 'dirty' })`.
   - **Cached per tab.** Each tab's state holds its last-computed `changed: Set<string>` (added alongside `expanded`/`watchers` in the `FileTreeManagerTabState` shape). Every `rebuild()` applies this cached set synchronously so interactive redraws (expand/collapse) are instant and never spawn git. The set is *recomputed* only when the tree's underlying files may have changed â€” i.e. on `open()` (first paint), on `reroot()` (root changed, so the old set is invalid and must be discarded, not reused), and on **watcher-driven** rebuilds (`scheduleRebuild`). `move`/`delete`/`undo`/`redo` mutate the filesystem and therefore fire the watcher, so their git refresh arrives via that path ~one debounce later â€” the immediate redraw shows the slightly-stale cached coloring and self-corrects on the next watcher tick, which is acceptable for a decoration.
   - **Coalesced.** A git refresh is guarded by a per-tab in-flight flag: if a refresh is requested while one is already running, it is not started concurrently â€” instead a "stale" bit is set and exactly one more refresh runs when the current one resolves. This prevents a filesystem write-burst (a build, `npm install`, `git checkout`, a harness writing many files) from spawning a pile-up of overlapping git processes behind the 100 ms watcher debounce. On tab close (`untrack`) or a `reroot` mid-flight, a resolving refresh whose tab is gone or whose root has changed discards its result instead of writing stale rows.

## Proposed changes

1. **Server: git status lookup.** Add a small module `src/git-status.ts` exposing an **async** function (e.g. `changedPaths(root: string): Promise<Set<string>>`) that, given an absolute directory root, runs `git status --porcelain=v1 --untracked-files=all -z -- .` scoped to that root (per decisions 5 and 6) and resolves to the set of root-relative paths that are modified, staged, or untracked â€” or an empty set if the root isn't inside a git repository or the command fails. Use `execFile` (promisified via `node:util`'s `promisify`, or its callback wrapped in a `Promise`), and wrap it in a `try/catch` that resolves to an empty set on any failure (non-git directory, git not installed, non-zero exit) â€” never surfaced to the caller as an error, per decision 4. This function is pure I/O with no knowledge of tabs or caching; the caching, coalescing, and staleness handling from decision 6 live in the tree manager (item 2b), not here.
2. **Server: attaching the flag to rows.** Add a second, synchronous function in the same module (or in `src/file-tree.ts`, next to `buildRows`) â€” e.g. `markChanged(rows: FileTreeRow[], changed: Set<string>): FileTreeRow[]` â€” that takes the flat row list `buildRows` already produced and a changed-path set, and returns rows with a new field set: a file row is marked when its `path` is in `changed`; a directory row is marked when any path in `changed` starts with `${row.path}/` (per decision 2's prefix-check, not a recursive walk). This produces a `changed?: boolean` field on `FileTreeRow` meaning "this row itself, or something beneath it, has a git change." Both of `file-tree-manager.ts`'s `buildRows` call sites â€” `open()` at `:82` and the private `rebuild()` at `:257` â€” pass the tab's **cached** changed-set (item 2b) through `markChanged` so every render is colored synchronously and instantly, using the last known git state.
   2b. **Server: cached, async, coalesced refresh (decision 6).** Add to `FileTreeManagerTabState` (`src/file-tree-manager.ts:21-28`) a `changed: Set<string>` cache (initialized empty) plus a small pair of flags for coalescing (e.g. `gitRefreshing: boolean` and `gitRefreshStale: boolean`). Add a private method (e.g. `refreshGit(label)`) that: returns early if a refresh is already in flight for that tab, instead setting the stale bit; otherwise marks in-flight, captures the tab's current `root`, `await`s `changedPaths(root)`, then â€” only if the tab still exists and its root is unchanged (guarding against a mid-flight tab close or `reroot`) â€” stores the result in `state.changed` and calls `rebuild(label)` to re-render with the fresh coloring (the second `dirty` emit of decision 6); finally clears the in-flight flag and, if the stale bit was set, clears it and runs one more `refreshGit`. Trigger `refreshGit` from exactly three places: `open()` (first paint), `reroot()` (root changed â€” also reset `state.changed` to empty so the stale old-root coloring does not flash), and the watcher path `scheduleRebuild()` (`:238`). The interactive triggers (`toggle`, `collapseAll`, `move`, `delete`, `undo`/`redo`) do **not** call `refreshGit` directly â€” they reuse the cache and self-correct via the watcher-driven refresh their filesystem changes provoke, per decision 6.
3. **Shared type.** Extend `FileTreeRow` in `src/types.ts` (`:105`) with the new optional boolean field, documented the same way its existing fields are (a one-line comment on what "changed" means for a directory vs. a file row). No change needed in `src/protocol.ts` â€” it re-exports the type as-is (see the "what already exists" table).
4. **Client: rendering.** `web/src/FileTreeTab.tsx` is already 200 raw lines (`wc -l`), at the project's `max-lines` ceiling â€” adding the modifier-class logic inline risks tripping the lint rule. Extract the row's className computation (currently the inline template literal at `:164`, `` `files-row${row.path === selected ? ' selected' : ''}${drag.dropTarget?.path === row.path ? ' drop-target' : ''}` ``) into a small new helper module (e.g. `web/src/file-tree-row-class.ts`, exporting a function computing both the row wrapper's class and the `files-name` span's class from `row`, `selected`, and `drag.dropTarget`), and have `FileTreeTab.tsx` import it instead of inlining the logic â€” this keeps the component file's line count from growing while adding the `changed` modifier (e.g. `files-name--changed`) to the `files-name` span. Add the corresponding color rule to `web/src/theme.css`: reuse the existing `--running` custom property (already defined as a yellow/amber tone in every one of the file's theme blocks â€” `:root`/`dark` at `:15`, `light` at `:34`, `solarized-dark` at `:52`, `solarized-light` at `:70`, `nord` at `:88`, `dracula` at `:106` â€” currently used only for `.line.output.running` at `:273`) rather than inventing a new token that would need to be added to all six theme blocks.
5. **Spec update.** `product/specs/file-tree-tab.md`'s "Tree contents" section gains a short paragraph describing the git-modified coloring behavior (per the design decisions above), so the spec stays the source of truth for tree rendering rules.
6. **User documentation update.** `documentation/user-documentation/tab-types/file-navigator.md` gains a short paragraph (near the tree-contents description at `:12`, or as part of the "The tree stays current" section at `:37`) describing the git-modified coloring: files that git considers changed â€” modified, staged, or untracked â€” render yellow, an enclosing directory renders yellow when anything beneath it (even in a collapsed subtree) is changed, the coloring refreshes along with the tree, and a non-git directory shows no coloring and no error.

## Out of scope

- Distinguishing modified/staged/untracked with different colors or icons â€” all three render identically per decision 1.
- Any polling mechanism independent of the existing watcher-driven refresh.
- Showing an error or notice when a root isn't a git repository.
- Any change to the delete/undo/redo/drag-and-drop behaviors already documented in `file-tree-tab.md`.
