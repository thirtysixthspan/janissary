# Editor tab persona connections

**Complexity: 5/10** — spans server (a new persistent-session manager, a small new RPC, an extended close dispatch) and web (a new hook, wiring two existing but previously agent-tab-only components into a new surface), with real state-lifetime reasoning (get-or-create, error vs. explicit-close semantics, teardown on tab close) — more than routine wiring but every piece reuses an existing pattern (no new dependency, no new icon, no new UI mechanism), with no new subsystem or protocol shape invented from scratch.

Editor tabs gain their own connections list, just like agent tabs already have. Today, firing an in-editor persona suggestion (`> <persona> <prompt>`) spawns a fresh, single-shot ACP session per request via `spawnMonitorSession` and kills it once the reply lands — so a persona never remembers anything about a prior request in the same tab, and there is nothing for a connections window to show. This feature makes an editor tab's persona sessions persistent and multi-turn: the first suggestion request for a given persona in a tab opens an ACP connection and keeps it open; every later request to that same persona in that tab reuses it, and the persona's replies can build on what it said earlier in the tab. The tab's open persona connections appear in a connections window using the exact same button-driven, auto-fade hover window agent tabs already have (`useStatusWindows`/`AgentTabMeta`/`StatusPanels`), reached via a new plug-icon button in the editor tab's header. Each connection row also gets a close (×) control so a stuck or unwanted persona session can be closed individually, mirroring — and reusing the same underlying dispatch as — the existing `connection close <kind>:<id>` command.

## Design decisions

1. **Reuse `AgentTabMeta`/`StatusPanels` as-is; no new button/window components.** `AgentTabMeta`'s `connectionsButton`/`scheduleButton` props are already optional and independently gated (`web/src/AgentTabMeta.tsx:71-94`), and `StatusPanels` already renders purely off `tab.connections`/`tab.schedule` plus a `StatusWindowHandlers` pair (`web/src/StatusPanels.tsx`). `EditorTab.tsx` will render `<AgentTabMeta connectionsButton={...} />` (passing no `scheduleButton`, `cwd`, or agent-only callbacks, so only the plug button appears) alongside its existing bespoke `.image-meta` header, and `<StatusPanels tab={...} connections={...} schedule={...} interactive />` the same way `AgentTabBody` does. Because an editor tab's `TabView.schedule` is always empty, the schedule panel never shows without any extra `connectionsOnly` flag needing to exist on `StatusPanels`.

2. **`EditorTab` needs the owning `TabView`, not just `EditorView`.** `EditorTab` currently only receives `editor: EditorView` (`web/src/EditorTab.tsx:23`); `MountedViewLayers.tsx:43-52` already has the full `TabView` (`t`) in scope when it maps over editor tabs. Add a `tab: TabView` prop to `EditorTab` so it can reach `tab.label`/`tab.connections`. Wire the actual `useStatusWindows` call and close handling through a new hook rather than inline in `EditorTab.tsx` — see Decision 13's file-size constraint.

3. **A persona connection opens lazily on the first suggestion request, keyed by `(tab label, persona)`.** No connection exists until the user fires `> <persona> <prompt>` for that persona in that tab; from then on, every later request to the same persona in the same tab reuses the same open ACP session. Because editor tabs already dedupe one-tab-per-file — `openEditorTab` in `src/tab/openers.ts:29-43` (called from `src/tab/manager.ts:342-343`; not `src/tab-manager.ts`, which no longer exists — see `editor-tab-dedup.md`) finds and focuses an existing tab for the same `view.path` instead of creating a duplicate — a persona connection is unambiguously scoped to one tab.

4. **The reused session is multi-turn, not just process reuse.** Today's `editorSuggest` handler spawns via `spawnMonitorSession`, sends one primed prompt, and kills the session on reply (`src/editor-suggest/handler.ts`). The new behavior keeps the same `AcpSession` object alive across requests and calls `session.prompt(...)` again for each new request — the ACP agent process's own conversation state carries prior turns forward, exactly like the interactive `acp` command's per-tab session already does in `AcpManager.session()`/`.run()` (`src/acp/manager.ts:58-74,94-125`). Each request still re-sends the live buffer content (wrapped in a fresh random delimiter, as today) so the persona always sees the current file state, but it no longer needs the full persona body re-primed after the first turn.

5. **New server-side manager: `EditorAcpManager`, owning per-`(label, persona)` sessions.** Modeled directly on `AcpManager` (`src/acp/manager.ts`) and `MonitorManager`'s session map (`src/monitor/manager.ts:27-59`; class body starts at line 66), including its error-tolerance precedent (see Decision 6): a `Map<string, AcpSession>` keyed by `` `${label}:${persona}` ``, spawned via the existing `spawnMonitorSession` (`src/monitor/acp.ts`) the same way `editorSuggest` does today. Lives in a new module `src/editor/acp-manager.ts`, alongside the existing `src/editor/watch-manager.ts`, registered on `Managers` as `editorAcp: EditorAcpManager` (`src/managers.ts`) and constructed in `src/controller.ts` next to `editorWatch` (`src/controller.ts:47`). If the initial spawn itself fails (the subprocess-level `onError` in `connectAcp`, `src/acp/index.ts:42`, distinct from a per-prompt error — see Decision 6), the manager does **not** auto-evict the map entry; this matches `AcpManager`/`MonitorManager`, neither of which self-heals a failed connection either, and the existing manual-close capability (Decision 8) is the recovery path a user already has for a stuck connection of any kind — no extra eviction logic needed.

6. **`src/editor-suggest/handler.ts`'s `editorSuggest` function switches from spawn-and-kill to get-or-create-and-reuse.** Instead of calling `spawnMonitorSession` directly, it calls `managers.editorAcp.session(label, persona.name, cwd, { onError })` to get the tab's existing or newly-spawned session for that persona. Both of `session.prompt()`'s current handlers call `session.kill()` today (`src/editor-suggest/handler.ts:63-64`: `onEnd: () => { session.kill(); finish(...) }` and `onError: (message) => { session.kill(); finish(...) }`) — **both** `session.kill()` calls are removed, not just the one on the success path, so a single failed prompt (e.g. a transient ACP error) no longer tears down the connection. This matches how the interactive `acp` command's per-tab session already behaves on a prompt error: `AcpManager.run()`'s `onError` hook only appends the error to the tab transcript (`src/acp/manager.ts:99`) and never calls `.close()`, leaving the session live for the next prompt. After this change, only `EditorAcpManager.close()`/`.closeTab()` (explicit user close, Decision 8, or tab close, Decision 11) ever calls `session.kill()`. The priming text (persona body, hunk-reply format, delimited buffer content) is still sent with every request, per Decision 4.

7. **Connections rows: kind stays `'acp'`, text is `` `${persona} (acp)` ``.** `EditorAcpManager` contributes rows into `ConnectionManager.connectionsFor(label)` (`src/connection/manager.ts:11-29`) the same way `MonitorManager.connectionsFor` already does for monitor sessions — one row per open persona connection for that label, `{ text: `${persona} (acp)`, kind: 'acp' }`. This keeps the existing `acp`-kind row styling/icon instead of inventing a new `ConnectionKind`.

8. **Manual close, reusing the existing close dispatch.** `src/connection/close.ts`'s `'acp'` case (lines 22-26) currently only closes a tab's single interactive session (ignoring `id`) via `managers.acp.close(label)`, then emits `messageBus.emit('state', { type: 'dirty' })` and reports `out('Closed connection acp:opencode.')`. Extend it to first check `managers.editorAcp.close(label, id)` (`id` being the persona name); on success it likewise emits `messageBus.emit('state', { type: 'dirty' })` (so the client's next `state` broadcast drops the closed row from `tab.connections` without any extra client-side bookkeeping) and reports `out(`Closed connection acp:${id}.`)`; only when `editorAcp.close` returns `false` does it fall back to today's interactive-session close. This reuses the same `kind:id` grammar (`connection close acp:<persona>`) already used everywhere else, with no new `ConnectionKind` and no parser change needed (`src/connection/parsing.ts` already threads an arbitrary `id` through). The web connections window gets a new capability to trigger this: each row in `StatusPanels` optionally renders a close (×) control when the caller supplies a handler; `EditorTab`'s call site supplies one that issues a new RPC (Decision 9), while `AgentTabBody`'s call site leaves it unset, so agent-tab rows are unchanged (read-only, as today). The close handler recovers the persona name by stripping the row's known `" (acp)"` suffix from its `text` (Decision 7's format) — this is a deliberate, accepted coupling to the exact display format `EditorAcpManager.connectionsFor` produces (no other code populates an editor tab's rows), and is safe because persona names are simple identifiers (see `loadPersona`/`listPersonas` in `src/personas.ts`) that never contain the literal `" (acp)"` substring.

9. **New RPC `closeEditorConnection`, fire-and-forget.** `{ method: 'closeEditorConnection'; params: { url: string; persona: string } }` in `src/protocol.ts`'s `RpcCall` union, dispatched in `src/message-handler.ts` next to `editorSuggest`/`editorPersonas` (around line 115): resolves the owning tab label from `url` the same way `editorSuggest`'s `ownerLabel` helper does, then calls the extended `closeConnection('acp', persona, managers, label, out)` from Decision 8 with a no-op `out` (an editor tab has no transcript to append the confirmation text to — the row disappearing from the next `state` broadcast is the only user-visible feedback needed for a button click). The client fires this with `client.send(...)` (fire-and-forget, matching `editorSync`/`renameTab` in `web/src/ws.ts:93,99`), not `client.request(...)` — there is no reply payload to await, unlike `editorSuggest`/`editorPersonas` which carry data back. This is the mechanism the connections window's close button calls — the CLI grammar and the window button both funnel through the one `closeConnection` dispatch function, satisfying "manual close" without a parallel one-off code path. A dedicated RPC is deliberately smaller and safer here than routing a synthesized `connection close acp:<persona>` string through the generic `command` RPC (`src/command/manager.ts`'s `dispatch`/`run`): that pipeline also records command history, records global history, and passes every input through the agent-busy queue gate (`dispatchOrRunOp`) — machinery built for a transcript-and-command-bar tab, none of which an editor tab has or needs for a single button click.

10. **Closing a connection aborts any in-flight request on it.** If a suggestion request is currently awaiting a reply on the persona connection being closed, closing kills the session immediately (`session.kill()`), and the pending client-side request resolves the same way any other ACP session error does today (`onError` → `finish([], message)` in `editor-suggest/handler.ts`), posting the existing "unknown persona"-style failure notification rather than hanging.

11. **Tab close tears down every persona connection for that tab.** `src/tab/cleanup.ts`'s `closeTabResources` gains `managers.editorAcp.closeTab(tab.label)` alongside the existing `managers.acp.close(tab.label)` (`src/tab/cleanup.ts:26`), killing every open persona session for that editor tab, matching how agent-tab connections are already torn down on close.

12. **Button icon and tooltip match the existing connections button exactly.** Reusing `AgentTabMeta`'s `connectionsButton` prop means the plug icon (`connectionsWindowIcon`), `"connections"` active tooltip, and `"no active connections"` empty tooltip (`web/src/AgentTabMeta.tsx:71-82`) apply unchanged — no new icon or wording to introduce.

13. **`EditorTab.tsx` is already close to the 200-line limit — extract the new wiring into a hook instead of inlining it.** `web/src/EditorTab.tsx` is 233 raw lines (197 by the `max-lines` rule's blank/comment-skipping count — `eslint.config.mjs:60`), against the 200-line cap; inlining a `useStatusWindows` call, an `AgentTabMeta`/`StatusPanels` pair of JSX blocks, and the close-row handler (persona parsing plus the RPC call) would push it over. Follow the file's own established precedent for this exact problem: `web/src/editor/useEditorSuggest.ts:1-5` documents that it was split out of `EditorTab.tsx` for the same 200-line reason. Add a new hook, `web/src/editor/useEditorConnections.ts`, that takes `(client: JanusClient, tab: TabView)` and returns the `useStatusWindows` result (for the `connectionsButton` prop) plus a `closeRow: (row: ConnectionView) => void` function implementing Decision 8/9's persona-parsing-and-RPC-send. `EditorTab.tsx` then only adds a few lines: one hook call, one `<AgentTabMeta connectionsButton={...} />` element (no `scheduleButton`, `cwd`, or agent-only callbacks), and one `<StatusPanels tab={tab} connections={...} schedule={...} interactive />` element passing `closeRow` through to the new `onCloseRow` prop from Decision 8's `StatusPanels` change.

**Implementation ordering.** Land the server-side pieces first and in this order, keeping `check-diff` green at each step: `EditorAcpManager` (Decision 5) → wire it into `src/managers.ts`/`src/controller.ts` → `editor-suggest/handler.ts`'s get-or-create switch (Decision 6) → `connection/manager.ts` and `connection/close.ts` (Decisions 7-8) → the new `closeEditorConnection` RPC in `src/protocol.ts`/`src/message-handler.ts` (Decision 9) → `src/tab/cleanup.ts` (Decision 11). Only after `protocol.ts` carries the new `RpcCall` variant does the web side compile against it: `web/src/icons.ts` → `web/src/StatusPanels.tsx`'s `onCloseRow` prop → the new `web/src/editor/useEditorConnections.ts` hook (Decision 13) → `web/src/EditorTab.tsx` → `web/src/MountedViewLayers.tsx`.

## Proposed changes

**`src/editor/acp-manager.ts` (new).** `EditorAcpManager` class: `session(label, persona, cwd, hooks): AcpSession` (get-or-create, spawns via `spawnMonitorSession`), `connectionsFor(label): ConnectionView[]` (one `{ text: '<persona> (acp)', kind: 'acp' }` row per open session for that label), `close(label, persona): boolean` (kills and forgets one session, returns whether one was open), `closeTab(label): void` (closes every session for that label, used on tab close).

**`src/managers.ts`.** Add `editorAcp: EditorAcpManager`.

**`src/controller.ts`.** Construct `this.managers.editorAcp = new EditorAcpManager(this.managers)` alongside the existing `editorWatch` construction.

**`src/editor-suggest/handler.ts`.** Replace the direct `spawnMonitorSession` call with `managers.editorAcp.session(label, persona.name, cwd, { onError })`, and remove `session.kill()` from both the `onEnd` and `onError` handlers passed to `session.prompt()` (`src/editor-suggest/handler.ts:63-64`), so the connection persists across requests and across a failed prompt alike (Decisions 4 and 6).

**`src/connection/manager.ts`.** `connectionsFor` gains a line appending `this.managers.editorAcp.connectionsFor(label)` rows, mirroring the existing `monitor.connectionsFor(label)` line (`src/connection/manager.ts:18`).

**`src/connection/close.ts`.** The `'acp'` case (`src/connection/close.ts:22-26`) tries `managers.editorAcp.close(label, id)` first, emitting the same `messageBus.emit('state', { type: 'dirty' })` and an `out(`Closed connection acp:${id}.`)` message on success; only when that returns `false` does it fall back to today's `managers.acp.close(label)` interactive-session behavior.

**`src/tab/cleanup.ts`.** Add `managers.editorAcp.closeTab(tab.label)` to `closeTabResources`, next to the existing `managers.acp.close(tab.label)` at line 26.

**`src/protocol.ts`.** Add `{ method: 'closeEditorConnection'; params: { url: string; persona: string } }` to the `RpcCall` union.

**`src/message-handler.ts`.** Add a `case 'closeEditorConnection'` (next to `editorPersonas`/`editorSuggest`, around line 115) that resolves the owning label the same way `editorSuggest`'s `ownerLabel` helper does, calls the extended `closeConnection` from `src/connection/close.ts` with a no-op `out`, and replies `{ result: 'ok' }` (no payload needed — see Decision 9).

**`web/src/editor/useEditorConnections.ts` (new).** Per Decision 13: takes `(client: JanusClient, tab: TabView)`, instantiates `useStatusWindows(tab.label, tab.connections.length > 0, false)`, and returns its result plus `closeRow(row: ConnectionView)`, which strips the `" (acp)"` suffix from `row.text` to recover the persona name and calls `client.send({ method: 'closeEditorConnection', params: { url: tab.editor!.url, persona } })`.

**`web/src/EditorTab.tsx`.** Accept the owning `tab: TabView` as a new prop (passed alongside the existing `editor`/`client`/`active`). Call the new `useEditorConnections(client, tab)` hook. Render `<AgentTabMeta connectionsButton={{ hasContent: tab.connections.length > 0, onEnter: ..., onLeave: ..., onClick: ... }} />` alongside the existing `.image-meta` header (no `scheduleButton`/`cwd`/agent-only callbacks), and `<StatusPanels tab={tab} connections={...} schedule={...} interactive onCloseRow={closeRow} />` alongside the existing `<PendingSuggestPanel>`.

**`web/src/MountedViewLayers.tsx`.** Pass `tab={t}` into `<EditorTab>` at `web/src/MountedViewLayers.tsx:49`.

**`web/src/StatusPanels.tsx`.** Add an optional per-row close handler prop `onCloseRow?: (row: ConnectionView, index: number) => void`; when present, each connection row renders a small close button calling it with that row, reusing the existing plain-`×`-glyph pattern (`web/src/TabItem.tsx:80-89`'s `.tab-close` button — a bare `×` text button with `aria-label`/`title`, no icon import) rather than adding a FontAwesome icon. `AgentTabBody`'s call site does not pass this prop, so agent-tab rows stay exactly as they render today.

## Tests

- **`src/editor/acp-manager.test.ts`** (new): `session()` spawns once and returns the same session on a second call with the same `(label, persona)`; a different persona or a different label gets its own session; `connectionsFor` reflects only currently-open sessions with the `<persona> (acp)` text; `close(label, persona)` kills and removes just that one session and returns `true`, returns `false` when none was open; `closeTab(label)` closes every session for that label only.
- **`src/editor-suggest/handler.test.ts`** (extend): a second request for the same persona/tab reuses the same injected session (spawn called once, not twice); a request for a different persona in the same tab spawns a second session; the session is not killed after a successful reply nor after a failed one (only `editorAcp.close`/tab-close ever calls `kill()`).
- **`src/connection/close.test.ts`** (extend): `connection close acp:<persona>` closes an editor tab's matching persona connection (and emits the dirty state event) without touching an unrelated agent tab's interactive session; closing while a request is in flight surfaces the same failure-notification path as any other session error (Decision 10); closing a persona with no open connection falls through to the unchanged interactive-session branch and reports its existing message.
- **`web/src/StatusPanels.test.tsx`** (extend): a row renders a close control when `onCloseRow` is supplied and calls it with that row on click; no close control renders when the prop is omitted (agent tabs unaffected).
- **`web/src/editor/useEditorConnections.test.ts`** (new, mirroring `web/src/useStatusWindows.test.ts`'s conventions): `closeRow` strips the `" (acp)"` suffix and sends `closeEditorConnection` with the tab's `editor.url` and the recovered persona name; the returned status-window state reflects `tab.connections.length > 0`.
- **`web/src/EditorTab.test.tsx`** (extend): the connections button is dark/disabled with no open persona connections; once `tab.connections` is non-empty the button is active and the connections window shows the persona row; clicking a row's close control fires `closeEditorConnection` with the tab's `url` and that row's persona.

## Out of scope

- Any change to agent-tab or harness-tab connection behavior, the interactive `acp` command, or monitor connections — this plan only touches editor tabs' persona sessions.
- A schedule window/button on editor tabs (editor tabs have no schedules; only the connections button is added).
- Manual close controls on any connection kind other than editor-tab persona (`acp`) rows — shell/browser/sqlite/ssh rows stay read-only in the web window, as today.
- Sharing or reusing a persona connection across two different editor tabs (moot today since editor tabs already dedupe one-per-file), or persisting a persona connection across a tab close/reopen.
- Any UI to inspect a persona connection's accumulated conversation history (no equivalent of the monitor's context-snapshot feature for editor persona connections).
