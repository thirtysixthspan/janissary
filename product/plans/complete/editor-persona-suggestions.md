# In-editor persona suggestions

**Complexity: 8/10** — a first-of-its-kind interactive surface: an editor tab (which today has no agent session) gains a client-side pending-suggestion mode and persona completion, a new server single-shot ACP flow with an anchor-hunk reply format and parser, and new protocol messages in both directions — novel interaction logic spanning server and web, on top of the existing ACP/persona and editor-hook machinery.

## Summary

Give an editor tab a way to ask a persona for a change to the text it is editing and apply the answer inline, without leaving the buffer and without going through a monitor reporting tab. The user writes a request line in the buffer led by `>` and the persona's name — e.g. `> summarizer rewrite this paragraph in one sentence` — and completes the persona name by tab-completion after the `>`. Pressing Ctrl/Cmd+Enter on that line fires a single-shot query to the named persona, priming it with the editor's current content (including any unsaved edits) plus the request. The persona replies with one or more proposed edit hunks that may target anywhere in the file. Each hunk is rendered inline as a pending change titled `(A)ccept or (D)ecline this change?`; the user presses `a` to apply the focused hunk or `d` to dismiss it, one hunk at a time. The `>` request line is removed from the buffer only if at least one hunk is accepted; if every hunk is declined (or the persona had nothing to suggest), the request line stays so the user can edit and retry. Failures and empty replies surface as notifications in the notifications tab.

This plan covers only the **in-editor suggested-edit surface**. Delivering editor suggestions into a monitor reporting tab already ships today (see `monitoring.md`'s editor-view target) and is out of scope.

## Design decisions

1. **Request syntax is a `>`-led line naming the persona.** A request is a single buffer line whose first non-whitespace character is `>`, immediately followed by a known persona name, then the prompt text: `> <persona> <prompt>`. The persona name is matched case-insensitively against the personas available to `monitor` (the `.md` files in `ai/personas/`, via `listPersonas` in `src/personas.ts`). Everything after the persona name on that line is the prompt.

2. **Persona name completes by tab-completion after `>`.** While the caret is on a line that begins with `>` and the user is typing the first word after it, pressing Tab completes against the available persona names. This is the one place the editor gains completion; it does not add general word-completion to the buffer. **The client obtains the persona list from a new dedicated server request that returns `listPersonas()` (`src/personas.ts:50`), and filters it client-side.** It does *not* reuse the command-bar `complete` RPC (`src/protocol.ts:152`, `{ text, cursor }`): that request is command-line-shaped and only surfaces personas as `monitor`/`unmonitor` arguments (`src/completion/handlers.ts:15-26`), so pointing the editor at it would mean fabricating a fake command line. A small list request keeps the concern in the editor.

3. **The query fires on Ctrl/Cmd+Enter, never on a plain Enter.** Plain Enter in the editor keeps its normal meaning (insert a newline — `keys.ts:77` maps `Enter` to `insert '\n'`), so a genuine Markdown blockquote line that happens to start with `>` is never consumed by accident. Ctrl+Enter / Cmd+Enter on a valid `> <persona> <prompt>` line sends the query. Ctrl/Cmd+Enter on a line that is not a valid request (no `>`, or a first word that is not a known persona) does nothing. **This is handled in `EditorTab.onKeyDown` (`web/src/EditorTab.tsx:158`), not in the pure `actionForKey` (`web/src/editor/keys.ts:89`):** `actionForKey` returns buffer `KeyAction`s only, whereas firing a query is a side-effect. Cmd/Ctrl+Enter currently yields `null` from `actionForKey` (`keys.ts:91-92` route meta/ctrl to `metaAction`/`ctrlAction`, neither of which handles `Enter`), so `onKeyDown` already no-ops on it today and the combo is free to repurpose.

4. **The query is a single-shot ACP prompt, not the monitor batch cycle.** Each request spawns a fresh, one-prompt ACP session for the named persona using the existing ACP connection modules — the same harness-directive-to-subprocess mapping `spawnMonitorSession` (`src/monitor/acp.ts`) already performs, and the same tool-denial defaults. The session is primed with the persona body, the editor's current buffer content, and the request prompt; it is prompted exactly once; its single reply is parsed for hunks; then it is torn down. There is no 30-second flush, no accumulated context, and no persisted session.

5. **The persona sees the live buffer, including unsaved edits.** The content sent as context is the buffer as it currently stands in the editor client at the moment Ctrl/Cmd+Enter is pressed — the same live text the user is looking at, unsaved changes included — not the on-disk file. This mirrors how a monitor is fed the editor's live draft rather than disk (see `editor-tab.md` "Live draft sync" and `monitoring.md`).

6. **The persona may propose one or more hunks anywhere in the file.** A reply is not limited to editing the request line's location. The persona returns a list of edit hunks, each identifying the text to replace and its replacement, so a single request can rewrite a distant paragraph, insert in several places, or make no change at all. Each hunk is applied or dismissed independently. **Ceiling (deliberate cut for v1):** an anchor is matched by exact string, at its first occurrence in the current buffer; because the query is single-shot, one-pending-at-a-time, and buffer edits are suppressed while a suggestion is pending (Decisions 4, 8, 7), the buffer barely shifts between request and reply, so exact-first-match is sufficient. A hunk whose anchor no longer matches (e.g. the user edited that text before firing) is simply dropped from the pending set rather than fuzzy-matched. Fuzzy/positional anchoring is a later upgrade if it proves necessary.

7. **Hunks are resolved one at a time with `a` / `d`, per focused hunk.** When a reply yields multiple hunks, one hunk is focused at a time; pressing `a` accepts (applies) the focused hunk and `d` declines (dismisses) it, after which focus advances to the next pending hunk until none remain. Each pending hunk is titled `(A)ccept or (D)ecline this change?`. While any hunk is pending, `a` and `d` resolve the focused hunk rather than being typed into the buffer; all other keys are suppressed from editing the buffer until the pending set is resolved. **Because printable keys reach the buffer through the hidden textarea's `onInput`/`flushTextarea` (`web/src/EditorTab.tsx:169`), not through `actionForKey`, the pending-mode interception of `a`/`d` (and the suppression of other edits) must happen in `onKeyDown` with `e.preventDefault()` so the character never reaches the textarea** — the same `onKeyDown` that already calls `e.stopPropagation()` and `e.preventDefault()` for handled actions (`EditorTab.tsx:158-166`).

8. **Only one request is pending at a time per editor tab.** Firing a new request while a suggestion is still pending is ignored until the current one is fully resolved (every hunk accepted or declined). A pending suggestion is preserved when the user switches away from the tab and back.

9. **The `>` request line is removed only if a hunk is accepted.** If at least one hunk from the reply is accepted, the originating `> <persona> <prompt>` line is deleted from the buffer once the pending set is resolved. If every hunk is declined, the request line is left in place so the user can adjust the prompt and re-fire.

10. **Failures and empty replies notify, and keep the request line.** An unknown/unavailable persona, an ACP or query error, and a successful reply that proposes no edit each post a notification to the notifications tab, naming the persona (e.g. `<persona>: no suggestion` for the empty case). In all of these the request line is left untouched, and nothing is inserted into the buffer. **This adds a new notification event to the enum and text in `src/notifications.ts` (`NotificationEventType` at `:9-16`, `notificationText` at `:56-66`), carrying an arbitrary detail string and always eligible (bypassing focus suppression), modeled on how `auto-approve` was added there** — rather than overloading the `manual` event, which is reserved for the `notify` command. The `notify` call is made server-side by the suggestion handler on the owning editor tab's label.

11. **Nothing about the suggestion is persisted.** Like the rest of an editor tab's state, a pending suggestion is in-memory only and is not saved or restored on `--relaunch`.

12. **New client logic goes in a dedicated editor hook, not inside `EditorTab.tsx`.** `web/src/EditorTab.tsx` is already 229 lines and near the 200-line `max-lines` cap (which skips blanks/comments). The request-detection, query-firing, pending-hunk state, and `a`/`d` interception are added as a new hook (e.g. `web/src/editor/useEditorSuggest.ts`) alongside the existing `useEditor`/`useEditorMouse`/`useEditorSync`/`useSyntaxHighlight` hooks, with `EditorTab` wiring it in and rendering the pending panel from its returned state. The pure request-line/persona parsing lives in its own small module (e.g. `web/src/editor/suggest-request.ts`) so it is unit-testable without React, mirroring how `web/src/ghost-suggestion.ts` isolates the command-bar ghost logic.

## Proposed changes

### Server: a single-shot editor-suggestion query

Add a client→server request (e.g. `editorSuggest`) to `src/protocol.ts` carrying the editor's `url` (the identifier `editorSync` already uses, `src/protocol.ts:173`), the persona name, the current buffer content, and the prompt text. Its handler validates the persona against `listPersonas` (`src/personas.ts:50`); on an unknown name it posts a notification (Decision 10) and returns a no-hunks result. On a known persona it loads it with `loadPersona` (`src/personas.ts:30`), spawns a one-prompt ACP session via the existing `spawnMonitorSession` path (`src/monitor/acp.ts:16`), primes the session with the persona body plus a new output-format instruction (below) and the buffer content wrapped as data (following how `monitoring.md` wraps monitored-target content in a per-session random delimiter), sends the prompt once, awaits the single reply, tears the session down, and returns the parsed hunks to the client. Any spawn or prompt error is reported via `notify` and returns no hunks. The owning tab is resolved from the `url` so the notification (Decision 10) is attributed to that editor tab's label.

Keep this handler in a new small module under `src/` (mirroring the `src/monitor/` layout, e.g. a `src/editor-suggest/` folder) rather than growing `src/monitor/manager.ts`, to respect the 200-line file cap and keep the single-shot flow separate from the monitor lifecycle. Note that `.js` import extensions are required in `src/` (NodeNext) — new server imports must carry `.js`.

### Server: a hunk reply format and parser

Add an editor-suggestion reply format alongside `src/monitor/reply-format.ts`'s `SUGGESTION_FORMAT`, instructing the persona to return zero or more edit hunks, each delimited by marker lines that give an **anchor** (the exact existing text to replace) and its **replacement** — anchor-based rather than line-numbered, so a hunk still resolves if the buffer shifts. The only new hand-written code here is the marker parser: reuse the marker-capture approach of `captureMarker` (`src/monitor/reply-format.ts:14`) to return an ordered list of `{ anchor, replacement }` hunks (empty list when the reply proposes nothing). Define the contract for an insertion (empty anchor at a stated position) and a pure deletion (empty replacement) in the format string's prose. Applying an accepted hunk is a plain string splice at the located anchor (no patch library needed); rendering a pending hunk *as a diff* for the panel should reuse the installed `diff` dependency (`createPatch` etc., already used at `src/monitor/feed-diff.ts:1`) rather than hand-rolling a diff view.

### Client: request detection and persona completion in the editor

Add a pure, React-free helper module (e.g. `web/src/editor/suggest-request.ts`, modeled on `web/src/ghost-suggestion.ts`) that, given the current line text and the known persona list, decides whether the line is a valid `> <persona> <prompt>` request and extracts the persona and prompt. Add a new dedicated server request (Decision 2) that returns `listPersonas()`; the client fetches it (e.g. once per editor tab) and filters it for both request validation and Tab-completion of the persona name after `>`. Render the completion as inline ghost/dropdown text — reuse the visual treatment of the command bar's completion (`web/src/command-completion.ts`, `web/src/CommandInput.tsx`) rather than inventing a new style.

### Client: firing the query and rendering pending hunks

Add a `useEditorSuggest` hook (Decision 12) that owns the pending-suggestion state and is wired into `EditorTab`. On Ctrl/Cmd+Enter in `EditorTab.onKeyDown` (`EditorTab.tsx:158`, Decision 3), when the current line is a valid request, it issues the `editorSuggest` request with the current buffer text (`toText`, `web/src/editor/model.ts:23`), the persona, and the prompt, and marks the tab as having a pending suggestion (blocking a second request per Decision 8). When the reply's hunks arrive, resolve each anchor against the current buffer to a concrete range and render the hunks as inline pending changes — each showing the proposed replacement and the title `(A)ccept or (D)ecline this change?`, with one focused at a time. While hunks are pending, `onKeyDown` routes `a` to accept-focused and `d` to decline-focused (advancing focus) with `e.preventDefault()` so the hidden textarea never inserts them (`flushTextarea`, `EditorTab.tsx:169`, Decision 7), and suppresses other buffer edits until the set is resolved. Accepting applies the hunk's replacement to the buffer through the editor's existing mutation path (`api.apply`/`api.insert` in `EditorTab.tsx`); declining drops it. After the set resolves, delete the `>` request line iff at least one hunk was accepted (Decision 9). Keep the pending state in the persistently-mounted `EditorTab` so it survives tab switches (Decision 8, per the "Mounted persistently by App" note at `EditorTab.tsx:18-20`), and never persist it (Decision 11).

### Specs

Add an "In-editor persona suggestions" section to `product/specs/editor-tab.md` describing the `>`-led request syntax, persona Tab-completion, the Ctrl/Cmd+Enter trigger, single-shot querying with the live buffer as context, per-hunk `a`/`d` resolution with the `(A)ccept or (D)ecline this change?` title, the request-line removal rule, the one-pending-at-a-time rule, and the notification behavior for failures and empty replies. Add a one-line cross-reference from `product/specs/monitoring.md` distinguishing this inline single-shot flow from a monitor's batched reporting-tab suggestions, and note in `monitoring.md`/`editor-tab.md` that both consume the live buffer.

### Implementation order

So typecheck and tests stay green at each checkpoint: (1) the server reply-format + parser module and its unit tests (pure, no protocol); (2) the `editorSuggest` and persona-list protocol messages plus the server handler and notification event; (3) the client pure helper (`suggest-request.ts`) and its tests; (4) the `useEditorSuggest` hook, its wiring into `EditorTab`, and the pending panel/completion UI; (5) the spec updates. Run `./scripts/run.mjs check-diff` after each. There is no dependency on another plan; all pieces are self-contained to this feature.

## Tests

- **Server, reply-format/hunk parser** (`src/editor-suggest/*.test.ts`, mirroring `src/monitor/reply-format.ts`'s test conventions): a reply with one hunk, multiple hunks, an insertion (empty anchor), a deletion (empty replacement), and a reply proposing no edit → empty list. Cover a malformed/partial marker block degrading to no hunk.
- **Server, request handler** (`src/editor-suggest/*.test.ts`): unknown persona → no hunks and a notification recorded; known persona happy path returns parsed hunks; a spawn/prompt error → no hunks and a notification. Use the existing monitor-session test seams (a fake `AcpSession`, as `src/monitor/*.test.ts` do) so no real subprocess is spawned.
- **Client, request-line helper** (`web/src/editor/suggest-request.test.ts`, colocated like `web/src/ghost-suggestion.test.ts`): valid `> <persona> <prompt>` parsed to persona+prompt; a `>` line whose first word is not a known persona → not a request; a plain blockquote line → not a request; leading whitespace before `>` handled.
- **Client, editor interaction** (`web/src/editor/useEditorSuggest.test.ts` and/or `web/src/EditorTab.test.tsx`, following the existing `useEditor.test.ts`/`EditorTab.test.tsx` setup): Ctrl/Cmd+Enter on a valid line issues the request; plain Enter does not; a second request while pending is ignored; `a` accepts the focused hunk and advances, `d` declines; `a`/`d` while pending do not insert into the buffer; the request line is removed only when a hunk was accepted and left when all declined; pending state survives a simulated tab switch.

## Out of scope

- **Monitor reporting-tab delivery of editor suggestions** — already shipped via `monitoring.md`'s editor-view target; unchanged here.
- **More than one pending request at a time** — deferred by Decision 8.
- **General word/identifier completion in the editor** — only persona-name completion after `>` is added (Decision 2).
- **Persisting or restoring a pending suggestion across `--relaunch`** — it stays transient (Decision 11).
- **Streaming/partial application of a hunk** — hunks arrive from a single reply and are applied whole.
- **Enabling filesystem/terminal or other tools for the query persona** — it inherits the monitor default of no tools (web tools only if the persona opts in, per `monitoring.md`), and this plan does not extend that.
