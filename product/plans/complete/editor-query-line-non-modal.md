# Editor query line should not be modal — buffer and query editing become interchangeable

**Complexity: 5/10** — one new virtual focus-state flag threaded through several already-small, already-modular files, plus one genuinely new chunk of logic (mouse clicks routing to the query row). No change to the shared hidden-textarea/IME model, no restructuring of the caret rendering primitives (`render.tsx`), no protocol change.

## Goal

Once the ephemeral agent query line (`product/specs/editor-tab.md` "In-editor persona suggestions") is open, every keystroke is currently routed exclusively to it (`web/src/editor/handleSuggestKeyDown.ts:90-95`: `if (suggest.queryLine) return handleQueryLineKeyDown(...)`), so the buffer's own reducer never sees a key at all — the user cannot edit the document while the query line is open. Mouse clicks, meanwhile, already move the buffer's cursor regardless of the query line's state, with no awareness of the query row at all (`web/src/editor/useEditorMouse.ts`). Per the backlog request, the user should be able to edit the document and the query line interchangeably — clicking into the buffer types into the buffer, clicking back into the query row types into the query — without the query line auto-closing either way.

## Approach

Introduce a new virtual `focusTarget: 'buffer' | 'query'` flag on `EditorSuggestApi` (`web/src/editor/useEditorSuggest.ts`), a sibling of the hook's existing `pillFocused` sub-focus flag. `openQueryLine` sets it to `'query'`; `closeQueryLine` resets it to `'buffer'`. Keydown and paste/IME routing (`handleSuggestKeyDown.ts`, `EditorTab.tsx`'s `flushTextarea`) switch from testing `suggest.queryLine !== null` to testing `suggest.queryLine !== null && suggest.focusTarget === 'query'` — the query line can stay open while `focusTarget` is `'buffer'`, at which point keys fall through to the buffer's own reducer exactly as if no query line were open. Caret visibility (`EditorLines.tsx`) follows the same flag, so exactly one caret renders at a time (the design `render.tsx`'s per-row caret model already supports without any changes there) — the row whose target currently isn't focused simply doesn't render a caret, matching the pattern's existing `active` gating.

The one genuinely new piece of logic is teaching `useEditorMouse.ts` to recognize a click landing on the query row (`hit.line === queryLine.anchorLine`, from `hitFromEvent`'s existing DOM-based hit-testing — the query row already renders with `data-editor-line={anchorLine}`, so this needs no new markup) and route it into the query's own single-line `EditorState` via the same `clampPos`/`setSelection` primitives `model.ts` already exposes generically, setting `focusTarget` to `'query'`; any other hit sets `focusTarget` to `'buffer'` and proceeds through the existing buffer-selection logic unchanged.

## Implementation steps

1. In `web/src/editor/useEditorSuggest.ts`: add `const [focusTarget, setFocusTarget] = useState<'buffer' | 'query'>('buffer');`. Update `openQueryLine` to also call `setFocusTarget('query')`, and `closeQueryLine` to also call `setFocusTarget('buffer')`. Add `focusTarget` and `setFocusTarget` to `EditorSuggestApi` and the hook's returned object.
2. In `web/src/editor/handleSuggestKeyDown.ts`, change the routing branch in `handleSuggestKeyDown` (currently `if (suggest.queryLine) return handleQueryLineKeyDown(e, suggest, api.stateRef.current);`) to `if (suggest.queryLine && suggest.focusTarget === 'query') return handleQueryLineKeyDown(e, suggest, api.stateRef.current);`. The `>`-opens-the-query-line branch below keeps its existing `!suggest.queryLine` guard unchanged (still exactly one query line at a time) — no `focusTarget` check needed there since it's unreachable while a query line already exists.
3. In `web/src/EditorTab.tsx`'s `flushTextarea`, change `if (suggest.queryLine) suggest.setQueryLineState(...)` to `if (suggest.queryLine && suggest.focusTarget === 'query') suggest.setQueryLineState(...)` so paste/IME input while focus is on the buffer lands in the buffer even though the query line is still open.
4. In `web/src/editor/EditorLines.tsx`, gate each row's caret on `focusTarget` instead of implicitly relying on cursor position: in `renderLine`, change `caretCol={onCursorLine && active ? state.cursor.col : -1}` to `caretCol={onCursorLine && active && suggest.focusTarget === 'buffer' ? state.cursor.col : -1}`; in `renderQueryRow`, change `caretCol={active ? qs.cursor.col : -1}` to `caretCol={active && suggest.focusTarget === 'query' ? qs.cursor.col : -1}`.
5. In `web/src/editor/useEditorMouse.ts`, accept an optional fourth parameter `suggest?: Pick<EditorSuggestApi, 'queryLine' | 'setQueryLineState' | 'setFocusTarget'>` (import the type from `./useEditorSuggest`). At the top of `onMouseDown`, after computing `hit` (reorder so `hit` is resolved before the `state`/gutter logic, since a query-row hit doesn't need `api.stateRef.current`): if `suggest?.queryLine` exists, `!hit.inGutter`, and `hit.line === suggest.queryLine.anchorLine`, call `suggest.setFocusTarget('query')`, compute `clampPos(qs.lines, { line: 0, col: hit.col })` against the query's own single-line state (`qs = suggest.queryLine.state`), call `suggest.setQueryLineState(setSelection(qs, pos, pos))`, `e.preventDefault()`, and return — no drag-selection support inside the query row (out of scope, see below). Otherwise call `suggest?.setFocusTarget('buffer')` and fall through to the existing buffer-selection logic unchanged.
6. In `web/src/EditorTab.tsx`, reorder the two hook calls so `suggest` exists before `mouse` is constructed (`useEditorSuggest` moves above `useEditorMouse`), and pass `suggest` as the fourth argument to `useEditorMouse`.

## Tests

- `web/src/editor/useEditorSuggest.test.ts` — extend the existing open/close test ("opens the query line seeded with a leading > and closes it") to also assert `result.current.focusTarget` is `'query'` after `openQueryLine` and `'buffer'` after `closeQueryLine`.
- `web/src/editor/handleSuggestKeyDown.test.ts` — `makeSuggest`'s defaults need `focusTarget: 'query'` and `setFocusTarget: vi.fn()` added so every existing "while the query line is active" test (which only overrides `queryLine`) keeps its current, still-correct expectation that keys route to the query text. Add two new tests under a new `describe` block: (a) with `queryLine` set but `focusTarget: 'buffer'`, a printable key is *not* handled (`handled` is `false`, `setQueryLineState` not called) so it falls through to the buffer; (b) same setup, Escape is *not* intercepted either (the query line only closes via an explicit close, not via a stray Escape meant for the buffer).
- `web/src/editor/useEditorMouse.test.ts` — add tests: clicking on a row matching a `queryLine.anchorLine` (pass a `suggest` mock with `queryLine`, `setQueryLineState`, `setFocusTarget`) calls `setFocusTarget('query')` and `setQueryLineState` (not `api.setState`); clicking on an ordinary row while a `queryLine` is set calls `setFocusTarget('buffer')` and proceeds through `api.setState` as before. Existing tests (which don't pass a fourth argument) must keep passing unmodified — the parameter is optional and gated with `suggest?.`.
- `web/src/EditorTab.test.tsx` — in the `'in-editor agent query line'` describe block, add: (a) a test that types into the buffer (via a keydown on a non-empty line) while the query line is open — assert the buffer content changes and `.editor-row-query` is still present (not closed); (b) a test that opens the query line, clicks (via `fireEvent.mouseDown`) on an ordinary buffer row to move focus there, types a character, then clicks back on `.editor-row-query`'s content and types into the query — assert both texts end up correct and the query line never closed. Mirror the existing `openAndType`/`textarea()` helper patterns already in this describe block.
- Run `./scripts/run.mjs check-diff` after each step; all suites must pass.

## Out of scope

- The query line's bracket-free pill text, placeholder span, gutter number, and Escape-cancels-in-flight-request fixes (already done, separate plans).
- Drag-to-select *within* the query row (click-to-place-caret only; a drag started on the query row does not extend a selection there). The existing buffer drag-selection machinery (`beginDrag`/`extendTo`/window listeners) is unchanged and continues to apply only to buffer hits.
- Showing two carets simultaneously (one dim, one solid) — only one caret renders at a time, following whichever target last received focus, matching the existing single-`caretRef` wiring in `EditorTab.tsx`.
- Any change to the pending hunk-review panel's modality (still click-only resolution; unrelated to this fix per the existing spec language, which this plan does not touch).
