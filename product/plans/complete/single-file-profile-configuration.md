# Single-file profile configuration

**Complexity: 6/10** — no new protocol, RPC, or concurrency, but a broad refactor across the entire profile read/write path (loader, `profiles.ts`, five section openers, five save modules), two field renames whose call sites are scattered (`label`→`name`, `harness`→`type`), a re-keying of the monitor subsystem that touches ~a dozen `reg.persona.name` sites in an already-259-line file, plus converting three profiles and reworking their tests.

Today a profile is a *directory* under `profiles/` holding many files — one `<label>.json` per agent/harness entry (the filename is the tab label), plus reserved `_monitors.json`, `_files.json`, `_notifications.json`, `_schedules.json`, and `_layout.json` for profile-level config. This feature collapses each profile into a single JSON file, `profiles/<name>.json`: a root object with an `agents` array and a `harnesses` array of entries (each carrying its own `name`), plus plain `monitors`, `files`, `notifications`, `schedules`, and `layout` keys for profile-level config. The per-profile directory goes away entirely. The three committed profiles (`debugging`, `features`, `planning`) are converted to the new format, the launch and save code are rewritten to read/write one file, and the spec and user docs are updated to describe it.

A second, related deliverable ships alongside the format change: a `profile validate` command that checks a profile file against the schema and reports the specific problems it finds, so an author of a hand-written single file can find a mistake without launching it and reading a terse error.

## Design decisions

1. **One flat file per profile: `profiles/<name>.json`.** The per-profile directory is removed. `profiles/writing-code.json` is the entire profile. `profile list` enumerates `profiles/*.json` (stripping the extension) instead of subdirectories; `profileExists` and the path helper resolve to the file.
2. **Entries in two typed arrays: `agents` and `harnesses`.** The root object carries an `agents` array (agent-state entries) and a `harnesses` array (harness entries). Array membership is the discriminator — there is no `_`-key or filename-based separation. A harness entry names *which* binary to launch with a **`type`** field (`claude`/`opencode`/`codex`), renamed from the old `harness` field so it no longer collides with the array name or reads as "is this a harness." Because entries are array elements rather than object keys, each entry carries its own `name` field supplying the tab label: agents already use `name`; harness entries use `name` in place of the old filename-derived `label`.
3. **Profile-level config under plain top-level keys — no `_` prefix.** With the root now a fixed, known set of keys (`agents`, `harnesses`, and the config sections), the leading-underscore convention that kept reserved files out of the entry set is unnecessary. The sections are plain keys: `monitors`, `files`, `notifications`, `schedules` (each the same JSON array their `_*.json` file held), and `layout`. An unrecognized top-level key is ignored (a reserved namespace for future config).
4. **`layout` holds the layout object directly, with a nested `sidebar` object.** The old `_layout.json` wrapped its object under an inner `layout` key only to distinguish an object-shaped reserved file from the array-shaped ones; that disambiguation is gone, so `layout`'s value is the layout object itself, not `{ "layout": { … } }`. Within it, the left/right sidebar widths are grouped into a `sidebar: { left, right }` object parallel to `window: { width, height }`, replacing the old flat `sidebarLeft`/`sidebarRight` fields; `tabAreaPct` stays a top-level number. Example: `{ "sidebar": { "left": 300, "right": 280 }, "tabAreaPct": 75, "window": { "width": 1280, "height": 800 } }`. This `sidebar` grouping is an **on-disk** shape only — see Decision 12.
5. **Hard cutover — no directory fallback.** Launch reads `profiles/<name>.json` only; an old-style `profiles/<name>/` directory is no longer recognized. `profile save` only ever writes the single-file format. This repo's three profiles are converted as part of this change. No auto-migration path is built.
6. **A malformed file is all-or-nothing at launch.** If the file is not valid JSON, its root is not an object, any element of `agents`/`harnesses` is not a structurally valid entry, or any config section has the wrong shape, `profile launch` reports `Profile "<name>" is malformed.` and opens nothing — no partial launch. This is validated once, up front, before any tab is opened. (Contrast the old per-file loaders, which each silently dropped a bad element; that best-effort policy is replaced by fail-loud for the whole file.)
7. **Structural malformation is distinct from the existing semantic launch-time checks.** "Malformed" (Decision 6) means the file's *shape* is wrong. The launcher's existing *semantic* per-entry checks are unchanged: an unknown harness, an unknown model (against `harness-models.json`), `autoApprove` on a non-claude harness, and unparseable `schedule`/`run` lines are still each reported and that one entry skipped while the rest launch (see `openHarnessEntry` in `src/profile/entry-openers.ts`). A structurally valid file naming an unknown model is not "malformed" — it launches, skipping that entry, exactly as today.
8. **The file is parsed once and its sections threaded down.** A single loader parses and validates the file and returns a structured result carrying the entries and every reserved section; the entry-opening flow consumes those in-memory sections rather than each opener re-reading the file (as the five `loadProfile*` accessors do today). This is what makes the up-front all-or-nothing check in Decision 6 the single source of validity.
9. **`profile validate [<name>]` command.** A new subverb alongside `list`/`save`/`launch`. `profile validate <name>` reports `Profile "<name>" is valid.` or a list of the specific problems (which key or field is wrong). Bare `profile validate` validates every profile and reports each. It shares the schema predicates the loader uses, but collects *all* problems with location context instead of failing on the first, so the terse launch error and the detailed command stay consistent.
10. **The validator checks structure only.** It verifies the file's shape — root object, each entry a valid agent or harness entry, each reserved section the right type/field types — and names the offending key/field. It does **not** perform the semantic launch-time checks from Decision 7 (unknown model/harness, etc.); those depend on runtime catalogs and remain the launcher's job. This keeps `validate` a pure, catalog-free schema check.
11. **No schema library — reuse hand-written predicates.** Validation uses the existing pure type-guard predicates (`isProfileMonitor`, `isProfileFilesEntry`, `isDockEntry`, `isProfileWindow` in `src/profile-reserved-files.ts`), not zod or any other validator library. zod is not a current dependency, and the codebase's established idiom (architecture principle 4) is pure hand-written `parseX`/`isX` functions returning structured data or `{ error }`; adding a schema library would introduce a competing validation style beside predicates the plan already reuses, for a small and stable schema. The loader and the `validate` command share these predicates — the loader treats a failed predicate as all-or-nothing malformation, the command builds a located message from the same check. Adopting zod project-wide is a larger, separate decision than this feature and is out of scope here.
12. **The `sidebar` grouping is mapped at the file boundary; the runtime layout type stays flat.** `sidebarLeft`/`sidebarRight` are threaded through the whole layout wire path — `ClientLayout` (`src/client-layout.ts:1`), the `reportLayout` RPC (`src/protocol.ts:176`) and the `layout` bus channel (`LayoutEvent` in `src/bus.ts:90`, emitted from `src/profile/layout.ts:38` as `{ type: 'update', sidebarLeft, sidebarRight, tabAreaPct }` and consumed in `src/controller/events.ts:21-22`), `applyProfileLayout` (`src/profile/layout.ts`), and the web client (`web/src/ws.ts`, `web/src/useLayoutState.ts`). None of that changes. The nested `sidebar: { left, right }` shape exists only in the profile file: the loader reads `layout.sidebar.left`/`.right` into the flat internal fields, and the save writer emits them back nested. A dedicated on-disk layout type (`ProfileLayoutFile`) describes the file shape; the existing internal `ProfileLayout` stays flat.
13. **A profile monitor carries a `name` that is its runtime identity, distinct from `persona`.** Each element of `monitors` is `{ name, persona, targets }`. The `name` — not the persona — becomes the monitor's identity for launch idempotency and for capture: relaunching a profile refreshes (stops-then-restarts) the monitor whose owner+`name` matches, so two monitors may share a persona yet coexist and each refresh independently. This reaches into the monitor subsystem: the monitor manager stores a name per running monitor and keys the "already monitoring" / relaunch-refresh check on owner+name instead of owner+persona. A monitor started from the interactive `monitor <persona> <targets>` command (which takes no name) defaults its name to its persona, so today's persona-only monitors keep their current single-per-persona behavior and round-trip through `profile save` unchanged.
14. **Tab presentation and placement grouped under a `tab` object.** Each entry (agent or harness) nests its dot color, tab number, group, and group color into a `tab` object — `{ color, number, group, groupColor }`, with `color` renamed from the flat `dotColor`. This applies to both arrays. The entry file shape therefore diverges from the raw runtime `AgentState`/`ProfileHarnessEntry` types (which keep their flat `dotColor`/`number`/`group`/`groupColor`), so the loader maps `tab.*` down to the flat runtime fields (including reading `tab.number` for the entry ordering and `tab.group` for the authored-group check) and the saver maps them back up — the same map-at-the-file-boundary approach as `layout.sidebar` (Decision 12). Downstream launch/open code is untouched; only the loader and saver know about `tab`.

Add a `name` field to `ProfileMonitor` (Decision 13). Add an on-disk `ProfileLayoutFile` type with `sidebar?: { left?: number; right?: number }`, `window?`, and `tabAreaPct?` (Decision 12); leave the internal flat `ProfileLayout` as-is. Rename `ProfileHarnessEntry.harness` to `type` (Decision 2), and change its `label` to an authored `name` field (matching agents) rather than a loader-populated one. Add on-disk entry file types (`ProfileAgentFile`, `ProfileHarnessFile`) that carry `name`, a `tab?: { color?: string; number?: number; group?: number; groupColor?: string }` object (Decision 14), and the remaining fields; the loader maps these to the flat runtime `AgentState`/`ProfileHarnessEntry`, so those runtime types keep their flat `dotColor`/`number`/`group`/`groupColor`. Update the doc comments on `AgentState.name`, `ProfileHarnessEntry`, `ProfileMonitor`, `ProfileFilesEntry`, `ProfileNotificationsEntry`, `ProfileSchedulesEntry`, and the layout types that currently say "populated by the loader from the filename" / "authored in a profile's reserved `_x.json` file" so they describe the `agents`/`harnesses` array elements' `name` and the plain config keys of the single file. Extend `ProfileParsed` with a `validate` variant carrying an optional `name`.

**Read path — new module `src/profile-file.ts` (or `src/profile/load.ts`).** A `loadProfile(name): LoadedProfile | { error: string }` that reads `profiles/<name>.json` once, `JSON.parse`s it, and validates the whole structure per Decisions 2–6: root must be an object; `agents`, if present, must be an array whose every element is a structurally valid agent entry (an object with a string `name` and, if present, a `tab` object of correctly typed fields); `harnesses`, if present, must be an array whose every element is a structurally valid harness entry (string `name`, string `type`, an optional well-typed `tab` object, other known optional fields correctly typed); `monitors` must be an array of `{ name, persona, targets }` elements; `files`/`notifications`/`schedules` must be arrays whose every element passes the corresponding reused predicate; `layout`, if present, must be an object of correctly typed fields with the nested `sidebar` shape, mapped down to the flat internal `ProfileLayout`; unrecognized top-level keys are ignored. On any failure it returns `{ error }` (the launcher maps this to the terse malformed message); on success it returns the `LoadedProfile`, its combined `agents`+`harnesses` entries mapped to the flat runtime types (`tab.*` → `dotColor`/`number`/`group`/`groupColor`, `tab.color` → `dotColor`) and sorted by `number` (the existing `loadProfileEntries` ordering). Remove `src/profile-reserved-files.ts`'s file-reading loaders, keeping its predicates (move them into the new module or a shared `profile-schema.ts` imported by both loader and validator), and extend the layout predicate to read the nested `sidebar` object into the flat fields.

## Out of scope

- **Reading the old multi-file directory format.** Hard cutover (Decision 5); old `profiles/<name>/` directories are not recognized and there is no fallback.
- **Auto-migration.** No code converts an old directory into the new file at launch or save; the repo's three profiles are converted by hand as part of this change, and any external directory-format profile must be reformatted manually.
- **Semantic validation in the `validate` tool.** Unknown harness/model, `autoApprove`-on-non-claude, and schedule/run parseability are not checked by `profile validate` (Decision 10); they remain per-entry launch-time reports.
- **A standalone CI/pre-commit validation script.** Validation ships only as the in-session `profile validate` command; a `./scripts/run.mjs` variant is not built here.
- **A published JSON Schema document** (e.g. for editor integration) is not part of this change.
