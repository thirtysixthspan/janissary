# Harness screen capture

**Complexity: 3/10** — a new server-side headless terminal per harness PTY with throttled capture timing and a new `resize` variant on the `pty` bus channel, plus a small `harness capture <name>` subcommand that writes a file and opens it via the existing editor path; no new view type, no protocol/RPC change visible to the client, no status classification, no dot-indicator changes.

## Summary

Janissary has no server-side terminal emulator today — the only `xterm.js` instance lives in the browser (`web/src/useXterm.ts`), so there is no way to inspect a harness's screen contents except by having a browser tab open and looking at it. This plan adds a minimal server-side capture mechanism: one headless `@xterm/headless` terminal per harness PTY, fed from the existing raw-byte `pty` bus channel (and kept at the real PTY's dimensions via a small new `resize` event on that channel), that schedules a re-read of its screen buffer 1 second after a `pty` byte arrives and keeps only the single latest capture in memory (no history, no classification of what it means). Capturing is tied to actual PTY activity rather than a periodic timer specifically so a harness sitting idle for a long stretch — no output at all — produces no repeated, redundant captures of an unchanged screen.

A new `harness capture <name>` subcommand writes that latest terminal capture to a plain-text file on disk and opens it in a normal editor tab — reusing the existing `EditorView`/`openInEditor` file-preview machinery instead of inventing a new view type. The result is a one-time, point-in-time capture of what the harness's screen looked like as of the last capture tick, not a live view; the file is just a regular file the editor tab already knows how to display (and, incidentally, edit/save, though editing a capture has no effect on anything).

This plan is deliberately narrow: it builds only the capture mechanism and a way to look at its output. It does not attempt to classify screen state (idle/working/blocked), does not touch the tab dot/busy indicator, and does not add any pattern-matching rules.

## Decisions (to be confirmed with user)

1. **Headless terminal engine: `@xterm/headless`**, added as a new server-side dependency alongside the existing `@xterm/xterm` (`^5.5.0`) already used client-side — same vendor/version family. One headless `Terminal` instance is created per harness PTY, fed the same raw bytes already flowing over `messageBus`'s `pty` channel (`PtyEvent` union at `bus.ts:86`; the `data` events are emitted from `pseudoterminal-manager.ts:22`, `onData: (id, data) => messageBus.emit('pty', { type: 'data', id, data })`), and never rendered — it exists purely to produce `buffer.active` line text. Construct it with `scrollback: 0` so `buffer.active` holds exactly the on-screen rows and nothing else — the capture is "the screen", not the scrollback history.
2. **Capture cadence: throttled on PTY activity, not a periodic timer and not a debounce.** On a `pty` `data` event for a harness, if no capture is currently scheduled, schedule one for 1000ms later (`setTimeout`) and mark one as pending; further `data` events while a capture is already pending are ignored (the timer is *not* reset/extended). When the pending timer fires, capture, then clear the pending flag so the next `data` event can schedule a new one. Net effect: a harness producing output gets captured roughly once per second of activity; a harness that goes idle simply stops producing `data` events, so nothing gets scheduled and no captures happen at all — this is the specific goal, avoiding repeated captures of an unchanged screen during long idle stretches. Confirm the 1000ms delay.
3. **Only the latest capture is kept.** No history/log of past captures, no diffing, no persistence — each scheduled capture simply overwrites the previous one in memory per harness PTY. Confirmed by the "latest capture" framing of the command.
4. **A capture is materialized to a real file, then opened via the existing editor opener — no new view type.** `EditorView` (`types.ts:88-100`) and its opener (`openInEditor`, `openers/editor.ts:23-33`) already do exactly "stat a file, register it under `/open/<id>`, open an editor tab" — the file just needs to exist on disk first. Write each capture to `.janissary/captures/<label>-<capturedAt-iso>.txt` in the project directory (same `.janissary/` root `agent-state.ts` uses for state, already gitignored) and then call `this.managers.openFile.edit(input, absolutePath, invokingLabel)` (`OpenFileManager.edit`, `open-file-manager.ts:47-52`) — the existing public "hand any file straight to the editor" entry point, which builds the `OpenContext` itself and passes absolute paths through unchanged. This means zero new `types.ts` fields, zero new web components, no hand-built context, and the resulting tab gets ordinary editor behavior (syntax-free plain text, save-to-disk if edited, external-change detection) for free.
5. **Capture files are written on-demand, not on every scheduled capture.** The throttled timer (Decision 2) only updates the in-memory `HarnessScreenReader` capture; a file is written to `.janissary/captures/` only when `harness capture <name>` actually runs, using whatever the reader's latest in-memory capture is at that moment. This avoids disk I/O for harnesses nobody is looking at.
6. **Capture files are cleared at launch, mirroring `agent-state.ts`'s `clearStateDirectory`.** Each `harness capture <name>` invocation adds one more `.txt` file to `.janissary/captures/`; rather than accumulating indefinitely across runs, the whole directory is wiped at startup — same as `.janissary/state/` — so captures only ever accumulate within a single run, not across restarts. Follow the exact same conditional as `clearStateDirectory` (`main.ts:138`: only when `!args.relaunch`, so a live handoff/relaunch doesn't discard captures from the run it's continuing).
7. **Command shape: `harness capture <name>` — a subcommand of `harness`, not a standalone verb.** `harness` is already special-cased ahead of the normal command registry (`command-manager.ts:78-83`: `/^harness\b/i` dispatches straight to `managers.harness.run(input)`, bypassing the `{ name, match, run }` `Command` module pattern entirely). `capture` is added as a second form recognized inside that same path — `parseHarnessCommand` (`harness.ts:19-35`) or a sibling parser distinguishes `harness capture <name>` from `harness <name> [as <label>] [-w] [--offline]` on the literal `capture` keyword, which can never collide with a real harness name since `capture` isn't a key in `HARNESS_COMMANDS`. `<name>` targets an existing harness tab by its tab label (the same tab-label targeting `send <tab> <text>` and `queue <agent> <command>` use), matched exactly and case-sensitively, not a harness type — so `harness capture claude` captures the tab labeled `claude`, not "launch-and-capture a new claude instance."
8. **Each invocation opens a new tab.** Consistent with how `open`/`edit` behave — no attempt to find-and-refresh a previously opened capture tab for the same harness. Confirm this rather than a "reuse existing capture tab" behavior. Two invocations with no intervening PTY activity produce the same `capturedAt`, hence the same filename — the second write overwrites the file with identical content and opens a second editor tab on it; both are harmless.
9. **The headless terminal tracks the real PTY's dimensions via a new `resize` event on the `pty` bus channel.** Harness PTYs do not stay at their spawn size: the client fits each terminal to the browser viewport on mount and on every container resize (`web/src/useXterm.ts:35-41`, `syncSize` sends the `ptyResize` RPC → `controller.ts:127` `ptyResize` → `PseudoterminalManager.resizeOne`, `pseudoterminal-manager.ts:33`). A headless terminal left at 80×24 while the real PTY runs at, say, 220×60 would re-wrap every line and scramble the reconstruction. Fix: add a third variant `{ type: 'resize'; id: string; cols: number; rows: number }` to the `PtyEvent` union (`bus.ts:86`) and emit it from `resizeOne` after `session.resize(...)`; `HarnessScreenReader` subscribes to it (same `id` filter) and calls `term.resize(cols, rows)`. The only existing `pty` subscriber (`controller.ts:62`, subscribed to `['data', 'exit']` only) never receives the new type, so nothing else changes. For the initial size, expose the dimensions `spawn` currently uses privately (`this.cols`/`this.rows`, `pseudoterminal-manager.ts:12-13`) through a small public getter on `PseudoterminalManager` and pass them to the reader's constructor.

## Verified codebase facts that shape the design

- **No server-side terminal emulator exists today.** Raw PTY bytes flow `pty.ts:49` (`proc.onData`) → `messageBus.emit('pty', { type: 'data', id, data })` (`pseudoterminal-manager.ts:22`; the `PtyEvent` union itself is `bus.ts:86`) → websocket → browser, where `web/src/useXterm.ts` is the *only* place bytes are ever parsed into a screen (via `@xterm/xterm`). This plan must add a headless emulator; there is nothing to reuse server-side.
- **Harness PTYs run at the client's fitted size, not the 80×24 spawn default.** `useXterm`'s `syncSize` (`web/src/useXterm.ts:35-41`) fires on mount and on every `ResizeObserver` event, sending `ptyResize` → `resizeOne` (`pseudoterminal-manager.ts:33`). No resize information reaches the `pty` bus channel today — it carries only `data` and `exit` (`bus.ts:86`) — which is why Decision 9 adds a `resize` event.
- **`HarnessView` is minimal today**: `{ name, program, ptyId, status: 'running'|'exited', exitCode?, destination? }` (`types.ts:58-61`), attached to a `Tab` via `harness?: HarnessView` (`types.ts:158`), and explicitly **never persisted** to `AgentState` (`harness-manager.ts:46-47`, "Never persisted — harness tabs have no agent state") — consistent with keeping captured screen text in-memory-only too.
- **ssh tabs also carry a `harness` payload** (`name: 'ssh'`, `ssh-manager.ts:32`) but spawn their PTY in `SshManager`, not through `HarnessManager.spawnTab` — so they will never have a screen reader. `HarnessManager.capture` must therefore not assume "has `tab.harness`" implies "has a reader" (see §4's error handling).
- **Tab close funnels into the same PTY exit event, so one disposal path covers everything.** Closing a harness tab calls `managers.pty.closeTab(tab.label)` (`tab-cleanup.ts:17`), which kills the PTY; node-pty's `onExit` then fires and `handleExit` emits `messageBus.emit('pty', { type: 'exit', id, exitCode })` unconditionally — even after the session was already dropped from the registry (`pseudoterminal-manager.ts:74-96`). And a harness process exiting on its own closes its tab (`controller.ts:62-71`). So a reader that disposes itself on its PTY's `exit` bus event needs no separate hook in `tab-cleanup.ts`.
- **`EditorView` and its opener already do exactly what a capture tab needs, given a real file**: `openInEditor(file, context, line?)` (`openers/editor.ts:23-33`) stats the file, computes a human-readable size, calls `context.registerFile(file)` (→ `managers.tab.registerFile`, `tab-manager.ts:334`, which allocates the `/open/<id>` mapping) and `context.openEditorTab({ name, path, size, url, line })` (→ `managers.tab.openEditorTab`, `tab-manager.ts:394`). Better still, `OpenFileManager.edit(command, target, label, line?)` (`open-file-manager.ts:47-52`) is a public entry point that builds the `OpenContext` itself (`buildContext`, `open-file-manager.ts:54-64`) and hands the file straight to `openInEditor` — absolute paths pass through its resolution untouched (`path.isAbsolute` check at `open-file-manager.ts:50`). There is no in-memory-text equivalent today — but there doesn't need to be, since a capture can just be a real file.
- **`.janissary/` is the existing convention for gitignored, project-local runtime files**: `agent-state.ts` (`initAgentStateDirectory`, `ensureStateDirectory`) uses `<projectDirectory>/.janissary/state/`; `.gitignore:6` ignores `.janissary/` wholesale, so a sibling `.janissary/captures/` directory needs no new ignore rule.
- **Launch-time clearing already has a working precedent to copy exactly**: `main.ts:129,138` — `initAgentStateDirectory(cwd)` runs unconditionally at startup, then `if (!args.relaunch) { clearStateDirectory(); TranscriptStore.clear(); clearWorkspaceDir(); }` wipes prior runtime state, gated on *not* being a `--relaunch` (live handoff) invocation. A capture-directory clear should sit in that same conditional block.
- **`harness` is not a `Command`-module verb — it's special-cased ahead of the normal registry.** `command-manager.ts:77-83`: before falling through to `resolveCommand`/the `src/commands/*.ts` registry, `run()` checks `/^harness\b/i` and dispatches straight to `this.managers.harness.run(input)` (`harness-manager.ts:17-21`), which calls `parseHarnessCommand` (`harness.ts:19-35`) and then `this.open(...)`. A `capture` subcommand belongs in this same path, not as a new `src/commands/*.ts` file.
- **`node-pty` (`^1.1.0`) and `@xterm/xterm` (`^5.5.0`) are already dependencies**; `@xterm/headless` is not yet a dependency and must be added (same `@xterm` scope/version line).
- **Plan doc convention**: `plans/ready/application-themes.md` establishes the section structure this document follows (Summary → Decisions → Verified facts → Proposed changes with a "what already exists" reuse table → Implementation order → Out of scope → Verification).
- **Docs live in two tiers, and this feature only touches one**: `specs/*.md` are per-feature functional specs updated whenever behavior changes (`CLAUDE.md`: "specs — functional specs, one file per feature; when a change alters behavior, update the matching spec"), and `help.md` is the in-app command reference (a flat `### Commands` table; the `harness` row is `help.md:23`). The public docs site is `public-documentation/` (VitePress, per `CLAUDE.md`'s project-structure section) — a separate, deliberately-opted-into tier, out of scope for this plan.

## Proposed changes

### What already exists (reuse, don't rebuild)

### 1. Headless capture layer — `src/harness-screen.ts` (new), plus the `resize` bus event

- Exports `class HarnessScreenReader` that, given a harness PTY `id` and initial `cols`/`rows` (from the `PseudoterminalManager` getter of Decision 9), creates one `@xterm/headless` `Terminal` with those dimensions and `scrollback: 0` (Decision 1), subscribes to `messageBus.on('pty', ...)` filtered to that `id`, and on each `data` event: calls `term.write(data)` to keep the emulator's buffer current, then — only if no capture is currently pending — starts a `setTimeout` for 1000ms and marks a capture as pending (Decision 2). Subsequent `data` events that arrive while a capture is already pending are written to the terminal as usual but do not touch the timer.
- On a `resize` event for its `id` (the new `PtyEvent` variant of Decision 9, emitted from `PseudoterminalManager.resizeOne` after `session.resize(...)` at `pseudoterminal-manager.ts:33`), calls `term.resize(cols, rows)` so the emulator wraps lines exactly as the real PTY does.
- When the pending timer fires, reads `term.buffer.active` line-by-line (`buffer.active.length` rows — exactly the visible screen, given `scrollback: 0` — via `buffer.active.getLine(i)?.translateToString(true)`), joins with `\n`, drops trailing blank lines (the unused rows below the last content), stores it as `{ text, capturedAt: Date.now() }` — overwriting whatever was captured previously — and clears the pending flag so the next `data` event can schedule again.
- Exposes `latestCapture(): { text: string; capturedAt: number } | undefined` — returns the most recent capture, or `undefined` if the harness has never produced output for a full 1000ms yet.
- Disposes itself (clears any pending timeout, disposes the `Terminal`, unsubscribes from `pty`) on its PTY's `exit` bus event. That single path also covers tab close and app shutdown — both kill the PTY, whose `onExit` still emits the bus `exit` event (see the tab-close verified fact above) — so no change to `tab-cleanup.ts` is needed.

### 2. Wiring into `HarnessManager`

- In `harness-manager.ts`'s `spawnTab`, after `managers.pty.spawn(...)` (line 71), construct a `HarnessScreenReader` for the new PTY id and keep it in a private instance field `Map<string, HarnessScreenReader>` keyed by `ptyId` (matching `PseudoterminalManager`'s `private ptys = new Map(...)` idiom, `pseudoterminal-manager.ts:11` — not module-level state, and not on `HarnessView`, which is serializable view data). Because both `open` and `openFromProfile` funnel through `spawnTab`, profile-launched harnesses get readers with no extra wiring.
- Remove the entry from the map on the PTY's `exit` bus event — a single `messageBus.on('pty', 'exit', ...)` subscription in the `HarnessManager` constructor that also calls the reader's dispose. No `tab-cleanup.ts` change (see §1).
- No changes to `HarnessView`, `busy` state, or any status field — this plan doesn't touch status/dot behavior at all.

### 3. Capture file writer — `src/harness-capture-file.ts` (new)

Mirrors `agent-state.ts`'s directory-management style:

- `initHarnessCaptureDirectory(projectDirectory: string): void` — sets the module-level directory to `<projectDirectory>/.janissary/captures`, called once at startup alongside `initAgentStateDirectory`.
- `ensureCaptureDirectory(): void` — `mkdirSync(..., { recursive: true })`.
- `writeCaptureFile(label: string, capturedAt: number, text: string): string` — sanitizes `label` for use in a filename by replacing every character outside `agent-state.ts`'s `VALID_NAME` class (`/^[\w-]+$/`, `agent-state.ts:15`) with `-` (never rejects — the tab exists, so its label is legitimate; `as <label>` labels are single whitespace-free tokens but may contain `/`, `.`, etc.), builds the filename `<sanitized-label>-<timestamp>.txt` where `<timestamp>` is `new Date(capturedAt).toISOString()` with `:` and `.` replaced by `-` (e.g. `claude-2026-07-10T18-30-05-123Z.txt`), ensures the directory exists, writes `text` with `writeFileSync`, and returns the absolute file path.
- `clearCaptureDirectory(): void` — mirrors `agent-state.ts`'s `clearStateDirectory()` (`rmSync(directory, { recursive: true, force: true })`, swallowing errors) — wired into `main.ts` in the same `if (!args.relaunch) { ... }` block as `clearStateDirectory()` (Decision 6).

### 4. New subcommand — `harness capture <name>` (extends `harness.ts` + `harness-manager.ts`, no new command file)

- `harness.ts`: extend `parseHarnessCommand` (or add a sibling `parseHarnessCaptureCommand`) to recognize `harness capture <name>` before falling into the launch-parsing path — i.e. check whether the first token after `harness` is exactly `capture` (case-insensitive) and, if so, parse `<name>` as a target tab label rather than a harness type, returning something like `{ capture: true, label: string } | { error: string }` (`Usage: harness capture <name>.` when the label is missing).
- `harness-manager.ts`: add a `capture(input: string, label: string): string | undefined` method (same error-return convention as `run`/`open`) called from `run()` when `parseHarnessCommand` reports a capture request. Behavior, in order: look up the tab by exact label via `this.managers.tab` — no tab: error `No tab labeled "<label>".`; tab without a `harness` payload: error `"<label>" is not a harness tab.`; then look up its `HarnessScreenReader` by `harness.ptyId` from the map in §2 and read `latestCapture()` — a missing reader (an ssh tab, which carries a `harness` payload but spawns outside `HarnessManager` — see the verified fact above) and a reader with no capture yet both return `No capture available for "<label>" yet.` and skip opening a tab. Otherwise: call `writeCaptureFile(label, capturedAt, text)` (§3) to get an absolute path, then call `this.managers.openFile.edit(input, path, invokingLabel)` (`open-file-manager.ts:47-52`) where `invokingLabel` is `this.managers.tab.cur().label` — the same "creator tab" convention `open()` uses at `harness-manager.ts:27`. That stats the freshly written file, registers it under a new `/open/<id>`, and opens a normal editor tab, exactly like `edit <file>` would for any other text file, with no hand-built `OpenContext` and no extra `messageBus.emit('state', { type: 'dirty' })` beyond what the `open`/`edit` path already does.
- `command-manager.ts` needs no changes — the existing `/^harness\b/i` branch (`command-manager.ts:78-83`) already forwards the whole input string to `managers.harness.run(input)` and appends whatever error string comes back to the tab's transcript, so `harness capture <name>` flows through unchanged plumbing.

### 5. Tests

- `src/harness-screen.test.ts` (new): feed synthetic byte chunks through a `HarnessScreenReader` (via the real `messageBus`) using `vi.useFakeTimers()` (precedents: `schedule-manager.test.ts`, `editor-watch-manager.test.ts`), and cover: a single chunk followed by advancing 1000ms captures it; a second chunk arriving at, say, 500ms into an already-pending 1000ms window does *not* push the capture later (it still fires at the original 1000ms mark, reflecting whatever `term.buffer.active` looks like at that point); after a capture fires, a new chunk schedules a fresh 1000ms timer; no `data` events at all means `latestCapture()` stays `undefined` indefinitely (nothing ever gets scheduled); a `resize` event resizes the terminal (a subsequent capture reflects the new width); the `exit` event disposes (no capture fires from a timer still pending at exit). Caution for the implementer: xterm parses `term.write()` input asynchronously — tests must flush pending writes (the `write(data, callback)` completion callback, or advancing fake timers, which also drive xterm's internal write queue) before asserting on buffer contents.
- `src/harness-capture-file.test.ts` (new): `vi.mock('node:fs')` style like `agent-state.test.ts` — assert the directory is created, the filename is built as expected from `label`/`capturedAt` (including sanitizing a label containing a `/` or `.`), the file content matches `text` exactly, and `clearCaptureDirectory()` calls `rmSync` with the capture directory and swallows a thrown error the same way `agent-state.test.ts` exercises `clearStateDirectory()`.
- `src/harness.test.ts` (extend — it exists): cover `parseHarnessCommand`'s new `harness capture <name>` recognition — the malformed-usage error (missing `<name>`), and that it's correctly distinguished from `harness <name> [as <label>] [-w] [--offline]` (i.e. `capture` never gets treated as an unknown harness name).
- `src/harness-manager.test.ts` (new — no test file exists for `harness-manager.ts` today): cover `HarnessManager.capture(input, label)` — the no-such-tab error, the not-a-harness-tab error, the no-capture-yet message (including a harness-payload tab with no reader, the ssh case), and the success path — asserting a file is written and `managers.openFile.edit` is invoked with the written file's path — following this repo's existing manager test style (stub the `Managers` collaborators).

### 6. Docs — `specs/harness.md` and `help.md` (no public documentation)

- **`specs/harness.md`**: add a new `## Screen capture` section (after `## Lifecycle`, alongside the other behavioral sections like `## Connections panel`) documenting: the `harness capture <name>` subcommand and its usage error, that `<name>` targets an existing harness tab by label (not a harness type), the activity-throttled capture cadence (captured ~1s after output resumes, nothing captured during idle stretches) and that only the latest capture is kept in memory, where the file is written (`.janissary/captures/<label>-<capturedAt-iso>.txt`, cleared at launch) and that the result opens as a normal editor tab via the existing file-open path — not a new view type. This is this repo's spec convention (per `CLAUDE.md`: "when a change alters behavior, update the matching spec").

## Out of scope

- Any status classification (idle/working/blocked), pattern-matching rules, or per-harness rule modules — this plan only captures screen text, it doesn't interpret it.
- Any change to the tab dot / busy-set behavior — harness tabs keep their current (imperfect) busy behavior untouched.
- A live-updating capture tab (auto-refresh) — each `harness capture <name>` invocation is a one-time capture written to a new file; a live view is a possible fast-follow, not part of this plan.
- Pruning capture files *within* a run (e.g. capping the directory's size, deleting per-harness on tab close) — only whole-directory clearing at launch (Decision 6) is in scope; anything finer-grained is a fast-follow.
- Capture history or diffing between in-memory captures — only the single latest in-memory capture is ever kept per harness PTY.
- Title/progress (OSC) capture — this plan only reads the screen buffer text.
- ssh tabs. They share the harness view shape (`ssh-manager.ts:32`) but spawn outside `HarnessManager`, so they get no reader; `harness capture <ssh-tab>` reports `No capture available for "<label>" yet.` (§4). Extending capture to ssh tabs is a possible fast-follow.
