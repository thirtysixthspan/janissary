# Developer documentation

**Complexity: 3/10** — no new code surface (`src/`/`web/src/` untouched), but correctness depends on catching every one of several small call sites across a directory move: `config.mts`'s nav/sidebar/description, `index.md`'s hero links, three npm scripts, three `.gitignore` entries, two `ai/guidelines/` files, and `CLAUDE.md`'s project-structure line, plus authoring ten-plus new content pages from README material. Comparable in shape to `tab-name-alias` (a 4) — small in mechanism, wide in the number of places that must move together.

Add a developer-facing counterpart to the existing user documentation site, pulling the contributor-workflow content that currently lives in `README.md` (testing, checking changes, coverage, code quality, duplication, CSS linting, dead code, security checks, linting, commit conventions, and the workspace-sandbox security writeup) out into its own documented section, and trim `README.md` down to a short pointer plus the minimum needed to get someone running. Both audiences — end users of the app and contributors to the codebase — get their own dedicated section of the docs site, reachable from a shared top nav, instead of everything living in one long README.

## Design decisions

- **One shared VitePress instance, not two independent projects.** A single `.vitepress` config serves both audiences, rather than a second standalone VitePress project with its own config/build. This avoids duplicating VitePress theme/build boilerplate for what is otherwise the same site.
- **New root folder: `documentation/`.** The existing `public-documentation/` directory is retired (not renamed in place) in favor of a neutral top-level root, `documentation/`, containing `documentation/.vitepress/` (shared config/theme), `documentation/user-documentation/` (today's content, moved as-is), and `documentation/developer-documentation/` (new content sourced from `README.md`).
- **Two persistent top-nav items, each with its own path-scoped sidebar.** `themeConfig.nav` gets two entries — "User Docs" and "Developer Docs" — and `themeConfig.sidebar` becomes path-keyed (`/user-documentation/...` vs `/developer-documentation/...`) so each section shows only its own sidebar, using VitePress's existing multi-sidebar support rather than one flat merged sidebar.
- **The root landing page (`/`) stays the User Docs home.** Today's hero (the archer gif, tagline, "Getting Started" links) is unchanged and continues to live at `/`, effectively doubling as the User Docs section's home page. Developer Docs gets its own plain index page (no hero) at `/developer-documentation/`, reached via the nav link — there is no separate neutral chooser page.
- **Content scope pulled from README: the whole `## Development` section plus `## Security` → `### Workspace sandbox`.** `README.md`'s `## Development` section (`README.md:64`) currently has ten `### ` subsections in this order: Documentation, Testing, Checking changes, Code Coverage, Code Quality, Code Duplication, CSS Linting, Dead Code, Security Checks, Linting, Commit conventions — all ten move, including **Documentation** (`README.md:76-94`, the `docs:dev`/`docs:build`/`docs:preview` commands and the screenshot-generation pipeline), which the initial draft of this plan omitted from its enumerated list even though it's squarely "developer workflow" content. Plus the `## Security` → `### Workspace sandbox` writeup (`README.md:34-58`: Seatbelt read/write carve-outs, GitHub token setup, known limitations) — implementation/audit detail for someone building or reviewing the sandbox, not end-user-facing product behavior.
- **README keeps only a short pointer, not a duplicate copy.** After the move, `README.md`'s `## Development` section shrinks to the bare `npm start` / `npm test` commands needed to get running, plus one line pointing to the Developer Docs site for everything else. The `## Security` section keeps just its `### Dev-tooling security checks` cross-reference line, pointing to Developer Docs instead of an in-repo anchor. No content is kept in both places.
- **README's existing top-of-file documentation link is updated to mention both sections**, rather than adding a second, separate link further down the file: the current "See the documentation site..." sentence near the top is reworded to name both the user-facing docs and the developer documentation, each linking to its respective section of the (single) hosted site.
- **Navbar label: "Developer Docs", replacing the existing "Guide" entry.** `config.mts`'s current `themeConfig.nav` (`public-documentation/.vitepress/config.mts:48`) has exactly one entry: `{ text: "Guide", link: "/getting-started/application" }`. That entry is renamed to `{ text: "User Docs", link: "/user-documentation/getting-started/application" }` (matching the plain, short style it already had) and a second entry, `{ text: "Developer Docs", link: "/developer-documentation/" }`, is added alongside it — there is no separate "Guide" item left over.
- **Site metadata: keep the title, broaden the description.** `config.mts`'s `title: "Janissary"` (`config.mts:44`) is unchanged, since the site is still one Janissary docs site. `description: "User documentation for Janissary, a tab-based terminal shell for working with AI agents"` (`config.mts:45`) is reworded to cover both audiences (e.g. "Documentation for Janissary, a tab-based terminal shell for working with AI agents — user guide and developer/contributor docs") now that it serves more than user docs.
- **Broken external deep links are out of scope, not redirected.** Pages moving from `public-documentation/<path>` to `documentation/user-documentation/<path>` (and thus a new hosted URL path) will break any existing bookmarks/search results pointing at the old hosted paths. There is no in-repo CI/deploy workflow (`.github/workflows/` doesn't exist in this tree) to wire up redirects, and the actual publish step is external to this repo — same call the earlier `public-documentation` build-out plan made for hosting generally. This plan only restructures doc source content; it doesn't add a redirect mechanism.

## Proposed changes

Do the directory move and config rewiring first, verify `npm run docs:dev` still serves the moved User Docs content correctly under its new paths, then author the new Developer Docs content, then edit `README.md` last (so the README's new links point at pages that already exist) — landing these out of order risks a build that 404s internally partway through.

- Move `public-documentation/` to `documentation/user-documentation/`, and move `public-documentation/.vitepress/` to `documentation/.vitepress/` (config, theme, sprite-trim script unchanged in content, only path references updated where they assume the old root). Verified: `copyAgentImages()` in `config.mts` (`public-documentation/.vitepress/config.mts:17-18`) computes `docsRoot` from its own file location one level up (`.vitepress/..`) and then reads `agent-images/` one level above *that* (`docsRoot/../agent-images`) — since `documentation/` sits at the same depth directly under the repo root that `public-documentation/` did, this path math keeps resolving correctly with no code change needed; only the directory names on disk move.
- `index.md`'s hero (`public-documentation/index.md`, `layout: home` frontmatter) stays at the site root as the User Docs home per the design decision above — but its two `actions` links (`/getting-started/why-janissary`, `/getting-started/startup`) must be updated to `/user-documentation/getting-started/why-janissary` and `/user-documentation/getting-started/startup`, since the pages they point to move into the `user-documentation/` subfolder while `index.md` itself does not.
- Update `package.json`'s `docs:dev`, `docs:build`, `docs:preview` scripts (`package.json:17-19`) to target `documentation` instead of `public-documentation`.
- Update `.gitignore`'s three `public-documentation/...` entries (`.gitignore:9-11`: `.vitepress/dist`, `.vitepress/cache`, `public/agents/`) to their `documentation/...` equivalents — these are generated/copied paths that must stay ignored under the new root.
- Update `documentation/.vitepress/config.mts`: rename the existing `"Guide"` nav entry to `"User Docs"` and add the `"Developer Docs"` entry (see Design decisions); update the `description` string; convert `themeConfig.sidebar` from a flat array to a path-keyed object with two entries, one for `/user-documentation/` (today's existing group list, paths prefixed accordingly) and one for `/developer-documentation/`.
- Author new pages under `documentation/developer-documentation/`, each a single-topic file (mirroring the file-size/single-responsibility convention `[[developer-documentation]]` and `[[code-guidelines]]` both call for), covering all ten moved subsections: Documentation (the doc-site build/screenshot commands — update its own `docs:dev`/`docs:build`/`docs:preview` examples to reflect the new `documentation` root rather than copying the stale `public-documentation` invocation), Testing, Checking changes (the diff-scoped vs. full-gate distinction), Code Coverage, Code Quality (FTA + sonarjs), Code Duplication, CSS Linting, Dead Code (Knip), Security Checks (lint rules, secrets scanning, dependency audit, threat model, inline suppression), Linting, Commit Conventions — plus a Workspace Sandbox page (the Seatbelt security writeup, GitHub push/PR token setup, known limitations). Content is adapted from `README.md`'s current text, not templated fresh — it's a relocation with light rewriting for the new format, not a rewrite of the underlying facts.
- Add a `documentation/developer-documentation/index.md` landing page (no hero) that briefly orients a contributor and links into the topic pages above.
- Edit `README.md`: reword the top-of-file documentation-site sentence (`README.md:7`) to mention both sections; shrink `## Development` to just `npm start`/`npm test` plus a pointer sentence to Developer Docs; shrink `## Security`'s `### Dev-tooling security checks` line (`README.md:60-62`) to a pointer instead of an in-repo anchor; remove the now-relocated `### Workspace sandbox` prose (keep the section heading only if something user-relevant needs to stay — see Open questions).
- Update the two `ai/guidelines/` files that name `public-documentation/` by path, so the guidelines don't describe a directory that no longer exists: `ai/guidelines/documentation.md` (two references, lines 3 and 7 — the decorative-sprite guideline's description of where the docs live and where `config.mts` copies sprites to) and `ai/guidelines/user-documentation.md` (four references, lines 3, 86, 162, 177 — every place it names `public-documentation/` as the location of user-facing docs) both become `documentation/user-documentation/`.
- Update `CLAUDE.md`'s Project Structure entry (`CLAUDE.md:161`: `` `public-documentation/` — user-facing docs site (VitePress; `npm run docs:dev` / `docs:build`) ``) to reflect the new `documentation/` root housing both `user-documentation/` and `developer-documentation/`.

## Tests

This is a documentation-only change with no `src/` or `web/src/` behavior touched, so there are no vitest additions. Verification is manual: confirm the VitePress build succeeds and both sections render, sidebar-navigate correctly, and don't 404 internally.

## Out of scope

- Deploying the rebuilt site or updating whatever external process publishes it to `https://thirtysixthspan.github.io/janissary/` — no in-repo CI/deploy workflow exists to change.
- Redirects or link-preservation for the old `public-documentation/`-rooted URLs.
- Any new screenshot/visual assets for the Developer Docs pages (the existing screenshot-automation pipeline is a User Docs concern per `[[user-documentation]]`; Developer Docs content here is prose/tables/code samples only, matching the source README material).
- Rewriting or restructuring existing User Docs content beyond the mechanical path move.
- Editing historical plans under `plans/complete/` that happen to mention `public-documentation/` by path (e.g. `plans/complete/public-documentation.md`, `plans/complete/docs-screenshot-file-tree-sidebar.md`) — those describe work as it was at the time and are left as a historical record, not updated to match the new layout.

## Open questions

- Whether `README.md`'s `### Workspace sandbox` heading should keep a one-paragraph summary (e.g. "workspaced tabs are sandboxed by default on macOS; see Developer Docs for the full model") or be removed entirely down to just the pointer line, since some of that section's content (the `--offline` flag, `sandboxWorkspaces: false` config option) is arguably user-observable behavior rather than pure contributor detail.
